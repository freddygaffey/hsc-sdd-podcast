NARRATOR: Now the heart of the dot-point, and the most examinable thing: choosing test data with path and boundary testing. Boundary testing — boundary value analysis — tests the values at the edges of a valid range, because that's where off-by-one errors live. The rule: for a range, test just below the minimum, the minimum, just above the minimum, a typical middle value, just below the maximum, the maximum, and just above the maximum. For a grade validated zero to one hundred, your boundary data is minus one, zero, one, fifty, ninety-nine, one hundred, and one hundred and one. The invalid ones — minus one and one hundred and one — must be rejected; the valid ones accepted. Pair that with equivalence partitioning — group inputs into classes that behave the same, then test one from each class plus the boundaries — so you test the representatives and the edges, not a thousand values.

NARRATOR: Path testing — path coverage — tests the execution routes through the code. Every branch, every decision, should be exercised by at least one test: the if-true path and the if-false path; the loop running zero times, one time, and many times. The hook, reused from Year eleven: "test every road through, and test the edges." Path testing is "every road through"; boundary testing is "the edges." Together they catch the two big bug families — wrong logic on an untested branch, and off-by-one errors at a limit. NARRATOR: Now the methodologies by visibility — black, white, grey box — the Year-eleven test types, which often appear alongside path and boundary. Black box testing — you test only inputs and outputs, with no view of the code; functional and boundary testing are usually black box, because you check behaviour against the spec without caring how it's implemented. White box testing — you can see the internal code, so you test its structure; path testing is white box, because you need to see the branches to cover them. Grey box is a mix — some knowledge of internals guides black-box testing. The hook: "Black sees nothing, White sees the code, Grey sees a bit." Boundary equals black box, path equals white box, is a connection markers love.

NARRATOR: Now a real story about skipping testing — the load-testing scandal I promised back in twenty-five oh-one. Healthcare-dot-gov, twenty-thirteen. The site was never properly load-tested before launch. It was sized for a few thousand concurrent users; on launch day, millions arrived, and it collapsed within hours. Load testing — a non-functional performance test, throwing realistic peak traffic at the system — would have caught this in advance. The lesson in today's terms: testing only the happy path, the functional "does it work for one user," while skipping the non-functional load test, is how a technically-correct system fails catastrophically in production. Non-functional testing isn't optional. The Knight Capital deploy from twenty-five oh-four was the other failure mode — an untested deployment of changed code. Both are testing failures.

NARRATOR: Now the second half: a language-dependent code optimisation technique — the second dot-point. Optimisation means making code run faster or use less memory; "language-dependent" means using a feature of your specific language to do it. The single most important rule, a hard interleave with Programming for the Web thirteen oh-three, performance and page-load: never optimise without measuring. Profile first to find the actual bottleneck, optimise that, measure again to confirm it helped. Optimising a part that isn't the bottleneck is wasted effort, and worse, can make code less readable for no gain. Measure, optimise, measure.

NARRATOR: The classic examinable technique is memoisation — said "mem-oh-eye-zay-shun" — caching the results of expensive function calls so you don't recompute them. The textbook example is a recursive function like Fibonacci: naively, computing the tenth Fibonacci number recomputes the same sub-results thousands of times, an exponential blow-up. Memoise it — store each result in a cache the first time, return the cached value after — and it becomes linear, dramatically faster. In Python that's language-dependent because the language gives you a built-in tool, the functools cache decorator, that does it in one line. Other examples: a list comprehension instead of an explicit append loop, or a set for fast membership testing instead of scanning a list. Listing three is a before-and-after — naive Fibonacci versus memoised. The point is to state the before and after performance: "I made it faster" without numbers is band three; "I reduced it from exponential to linear time, confirmed by timing, from two seconds to two milliseconds" is band six.

QUESTION: Question one, recall. Distinguish boundary testing from path testing. Pause, then play.

NARRATOR: Model answer: boundary testing, or boundary value analysis, selects test data at the edges of a valid input range — just below, on, and just above each limit — because errors such as off-by-one mistakes most often occur there; it is typically black-box testing, based on the specification. Path testing selects test data so that every execution route through the code — each branch and decision, and loops at zero, one, and many iterations — is exercised at least once; it is white-box testing, because it requires visibility of the code's structure. In short, boundary tests the edges of the data; path tests every road through the code. Naming the edges-versus-routes contrast and the black-versus-white-box link is the mark.

QUESTION: Question two, recall. Distinguish quality assurance from testing. Pause, then play.

NARRATOR: Model answer: quality assurance is the overall, process-oriented framework — criteria, reviews, continual checking, and compliance — that ensures the whole solution meets its quality requirements and prevents defects. Testing is one activity within QA: executing test cases against the built software to detect defects. QA is the process that prevents; testing is the activity that detects. Stating that QA is the preventive process and testing is a detecting activity within it is the mark.

QUESTION: Question three, the big one — code style. A function accepts a quantity between one and fifty inclusive and rejects anything else. Develop test data using boundary and path testing. Pause, write your test data, then play.

NARRATOR: Model answer: for boundary testing, select values at the edges of the valid range one to fifty: zero — invalid, just below minimum; one — valid, minimum; two — valid, just above minimum; twenty-five — valid, typical middle; forty-nine — valid, just below maximum; fifty — valid, maximum; fifty-one — invalid, just above maximum. Also include type and special cases — a non-numeric input and an empty input — which should be rejected. For path testing, ensure each branch is covered: at least one valid input that takes the accept path, at least one below-range input and one above-range input that take the reject paths, so every decision in the validation is exercised true and false. Presenting a labelled table of inputs with their expected accept-or-reject result, covering boundaries and both branches, plus invalid-type cases, is the full-mark answer — and noting that boundary data is black-box while path coverage is white-box earns the extra mark.

QUESTION: Question four, application. Describe a code optimisation technique and explain how you would apply it responsibly. Pause, plan, then play.

NARRATOR: Model answer: a code optimisation technique is memoisation — caching the result of an expensive or repeated computation so that subsequent calls with the same inputs return the stored result instead of recomputing it, which can reduce a naive recursive algorithm from exponential to linear time. It is language-dependent because a language such as Python provides a built-in tool, the functools cache decorator, to apply it in one line. To apply it responsibly, measure first — profile to confirm the function is actually the bottleneck and record a baseline time; apply the optimisation; then measure again to confirm it reduced the time and did not change the output. The rule is measure, optimise, measure, because optimising code that isn't the bottleneck wastes effort and can reduce readability for no gain. Naming a concrete language-dependent technique and the measure-optimise-measure discipline with before-and-after figures is the full-mark answer.
