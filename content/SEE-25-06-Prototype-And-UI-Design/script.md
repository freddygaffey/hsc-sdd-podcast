NARRATOR: Today's outcome is S-E-twelve-oh-six, and the dot-point is: propose an additional innovative solution using a prototype and user-interface design. The word "innovative" matters — you propose a new idea and explore it before investing in a full build. The home topic: prototyping and U-I design — said "you-eye design" — to validate an idea cheaply, and designing that interface for usability and accessibility.

NARRATOR: Objectives up front. By the end you'll be able to explain what a prototype is and why you build one first; walk the fidelity ladder from sketch to working prototype; apply usability principles; build in accessibility from the start; and answer "explain the purpose of prototyping in software development." Hear that question now.

NARRATOR: The core idea. A prototype is an early, incomplete model of a solution built to explore and validate an idea before committing to full development. The key word is cheaply — learn the maximum about whether an idea works while spending the minimum. You'd rather discover that a layout confuses users on a five-dollar sketch than on a five-week build. This is the same shift-left logic that runs through the course: catch problems early when they're cheap to fix, just like modelling before coding in twenty-three oh-two, and code review before integration in twenty-five oh-one. A prototype is shift-left applied to the user experience.

NARRATOR: Now the main hook, the fidelity ladder — how detailed and interactive the prototype is, from roughest to most polished. Four rungs, and the hook is S-W-M-P, said "swamp," because you climb out of the swamp of vague ideas into a real design. S, Sketch — a rough hand-drawn idea, seconds to make, throwaway. W, Wireframe — a low-fidelity layout: boxes, lines, and placeholder text showing structure and where elements go, deliberately with no colours or styling so you focus on layout and flow, not looks. M, Mockup — a high-fidelity static design: the real visual look, colours, fonts, branding, but not interactive. P, Prototype — an interactive model you can click through, simulating real flows for user testing. S-W-M-P: Sketch, Wireframe, Mockup, Prototype.

NARRATOR: Each rung tests a different thing, and you climb only as far as you need. A wireframe tests structure and user flow — does the journey make sense, is the information in the right place — without anyone arguing about the shade of blue. A mockup tests the visual design and brand. An interactive prototype tests the actual experience — can a real user complete the task. The trap, a common student error, is polishing a prototype like production: spending days making a throwaway mockup pixel-perfect when its job was just to test the layout. A prototype is meant to be cheap and disposable; over-polishing wastes the exact time prototyping was supposed to save.

QUESTION: Quick check. You want to test whether users can find the "submit assignment" button in a new layout, and you have one afternoon. Which rung of the fidelity ladder do you build, and which do you not bother with? Pause, decide, then play.

NARRATOR: You build a wireframe — low fidelity, boxes and labels, showing where everything sits and the flow to submit. That tests exactly what you care about — can users find and reach the submit button — and it's fast and throwaway. What you do not bother with is a polished, full-colour, branded mockup, let alone a fully coded interactive build. Visual styling is irrelevant to "can they find the button"; spending the afternoon on colours is polishing a prototype like production, the trap. Match the fidelity to the question you're asking.

NARRATOR: Now usability — designing the interface so people can actually use it. The framework here is Nielsen's ten usability heuristics, said "Nielsen's heuristics." You don't have to memorise all ten, but you should be able to name the principle and a few key heuristics — they're the language of good U-I. The headline ones: visibility of system status — always tell the user what's happening, show loading and confirmation; match between system and the real world — use the user's language and familiar concepts, not jargon; user control and freedom — give clear undo and escape; consistency and standards — same things behave the same way; error prevention — design so mistakes can't easily happen, like confirming destructive actions; and aesthetic, minimalist design — remove clutter, one idea at a time. That last one is the same minimalism as the slide-design rule from presenting, twenty-five oh-two: reduce cognitive load. Good interfaces and good slides fail for the same reason — too much at once.

NARRATOR: This is a heavy interleave with Programming for the Web, twelve oh-four — U-I and U-X principles. Everything you learned there about user experience lands here, in the prototype. U-X is the whole experience; U-I is the interface that delivers it; usability heuristics are how you judge whether the U-I serves the U-X. Prototyping is how you test U-X before you've built it — your web-design knowledge applied to your own project's idea.

NARRATOR: Now accessibility, and this is non-negotiable — both an ethical duty and a legal requirement. The framework is W-C-A-G, said "wuh-cag," the Web Content Accessibility Guidelines, organised around four principles you remember as POUR — Perceivable, Operable, Understandable, Robust. Perceivable — users can perceive the content: alt text on images, sufficient colour contrast, captions on video. Operable — they can operate it: full keyboard access, visible focus indicators, enough time. Understandable — clear language, predictable behaviour, helpful error messages. Robust — works with assistive technologies like screen readers, through proper semantic markup. POUR — Perceivable, Operable, Understandable, Robust. This is a callback to Programming for the Web twelve oh-four and to the accessibility-as-compliance point from quality assurance, twenty-four oh-six: failing W-C-A-G isn't just unkind, it can breach the Disability Discrimination Act. The trap is ignoring accessibility in mockups — bolting it on later is far harder than designing it in. Build accessibility into the wireframe, not after launch.

NARRATOR: One more interleave, tying the prototype to the people side: this connects to user-centred design from Secure Software Architecture, fourteen oh-three — designing around how humans actually behave, and the security-usability balance. A prototype is how you do user-centred design: you put a model in front of real users and watch, rather than guessing. And it connects forward to feedback, twenty-six oh-two: the prototype's whole purpose is to generate feedback you act on. You build it, test it with users, collect reactions, and refine — exactly the collect-synthesise-act loop we'll formalise next chapter. So a prototype isn't a deliverable; it's a question you ask your users.

NARRATOR: Worked example, weak versus strong. The brief: "explain how you'd use prototyping to propose a new feature — a quick-grade-entry mode for teachers." The weak answer: "I'd build the feature and show it to teachers." Band three — that's not prototyping, that's just building, with all the cost and risk prototyping exists to avoid. The strong answer climbs the ladder deliberately: "First, a sketch of the quick-entry idea to align the team in minutes. Then a wireframe showing the layout and flow — class list, tap a name, enter a mark, next — with no styling, to test whether the flow beats the current one. I'd put that wireframe in front of three teachers and watch them attempt the task, applying usability thinking — is the system status visible, can they undo a mistake, is it consistent with the rest of the app — and checking accessibility against POUR, keyboard operable and sufficient contrast. Only once the flow tests well would I build an interactive prototype to confirm the experience, then commit to the full build. Each fidelity level tests a specific thing, and I refine from real user feedback before investing in production code." Naming the fidelity ladder, tying each level to what it tests, applying usability and POUR accessibility, and feeding user testing back in — that's the band-six shape.

NARRATOR: Exam-style finish — four questions. Pause and attempt each before the model answer.

QUESTION: Question one, recall. Explain the purpose of prototyping in software development. Pause, then play.

NARRATOR: Model answer: the purpose of prototyping is to explore and validate a design idea cheaply and early, before committing to full development. A prototype is an early, incomplete model that lets developers test layout, user flow, and the overall experience with real users, gather feedback, and identify usability problems while they are still cheap to fix — rather than discovering them after an expensive build. This is the shift-left principle applied to the user experience. Naming "validate an idea cheaply and early" and the shift-left benefit is the mark.

QUESTION: Question two, application. Distinguish a wireframe from a mockup. Pause, then play.

NARRATOR: Model answer: a wireframe is a low-fidelity representation of an interface — boxes, lines, and placeholder text showing structure, element placement, and user flow, deliberately without colours, fonts, or styling, so it tests layout and flow. A mockup is a high-fidelity static design that shows the actual visual appearance — real colours, fonts, and branding — but is not interactive, so it tests the visual design. In short, a wireframe tests structure; a mockup tests look. Stating the fidelity difference and what each one tests is the mark.

QUESTION: Question three, the harder one. Explain how UI design supports usability, referring to recognised principles. Pause, plan a structured answer, then play.

NARRATOR: Model answer: U-I design supports usability by applying recognised principles, such as Nielsen's usability heuristics, that reduce errors and cognitive load. Visibility of system status keeps users informed with loading indicators and confirmations. Matching the system to the real world uses familiar language instead of technical jargon. User control and freedom provides clear undo and escape paths, so mistakes are recoverable. Consistency means similar elements behave the same way, so users transfer learning. Error prevention designs the interface so mistakes are hard to make, for example confirming destructive actions. And aesthetic, minimalist design removes clutter so attention falls on what matters. Together these make an interface intuitive, efficient, and forgiving — which is what usability means. Naming the principle set and explaining several heuristics with their usability effect is the full-mark answer.

QUESTION: Question four, extended. A team is prototyping a new student-portal feature. Explain how they should use the fidelity ladder and ensure the design is accessible. Pause, plan, then play.

NARRATOR: Model answer: the team should climb the fidelity ladder, building only as much fidelity as each question needs. They start with a sketch to align quickly, then a wireframe to test structure and user flow with real users, deliberately leaving out styling so feedback focuses on layout. Once the flow tests well, they create a mockup to test the visual design and branding, then an interactive prototype to test the actual experience through user testing, refining before any full build. To ensure accessibility, they design against W-C-A-G's POUR principles from the wireframe stage, not afterwards: Perceivable — alt text, sufficient colour contrast, captions; Operable — full keyboard access and visible focus indicators; Understandable — clear language, predictable behaviour, helpful error messages; Robust — semantic markup that works with screen readers. Designing accessibility in from the start is both an ethical duty and a legal requirement under disability discrimination law, and far cheaper than retrofitting. A full-mark answer walks the ladder with what each rung tests, ties prototyping to user feedback, and builds POUR accessibility in from the start.
