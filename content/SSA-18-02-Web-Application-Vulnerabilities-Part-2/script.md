NARRATOR: First — the B: broken authentication and session management. A family of failures in how a system proves identity and maintains the logged-in state. Name the common ones, because "describe broken authentication" wants examples. Weak controls: allowing weak passwords, no protection against credential stuffing — attackers replaying leaked username-password pairs — and no rate limiting on login, so brute force runs free. Session failures: session hijacking, where an attacker steals a live session identifier — through cross-site scripting, network sniffing, or a predictable I-D — and reuses it to impersonate the user; and session fixation, where an attacker plants a known session I-D before the victim logs in, then rides that same session once they do. And exposure failures: session I-Ds in the U-R-L where they get logged and shared, sessions that never time out, and tokens that aren't invalidated on logout.

NARRATOR: The fixes are exactly the S-A-F-E rules from seventeen-oh-one, now seen as the defence against named attacks. Strong, random session I-Ds defeat guessing. Cookie attributes — HttpOnly stops cross-site scripting stealing the I-D, Secure keeps it off plain H-T-T-P, SameSite blocks cross-site sending. Fixed lifetime — timeouts — limit a stolen session's window. And re-issue on Escalation — regenerate the session I-D at login — defeats session fixation directly, because the I-D the attacker planted is thrown away the moment the user authenticates. Add multi-factor authentication and rate-limited logins against credential attacks, and store passwords hashed-salted-slowed from fifteen-oh-two. The exam line: broken authentication and session management is the set of weaknesses in verifying identity and maintaining sessions — such as session hijacking and fixation — mitigated by strong random session I-Ds, secure cookie attributes, timeouts, regenerating the I-D on login, and multi-factor authentication.

QUESTION: Quick check. Distinguish session hijacking from session fixation. Pause, answer, then play.

NARRATOR: Session hijacking is an attacker stealing a victim's existing, valid session identifier — for example through cross-site scripting or sniffing — and reusing it to impersonate them. Session fixation is an attacker setting the victim's session identifier to a known value before they log in, then using that same identifier after the victim authenticates. Hijacking steals a session that exists; fixation plants one in advance. The shared fix is regenerating the session I-D at login, which kills a fixed I-D, plus HttpOnly and Secure cookies, which protect against theft. That's the mark.

NARRATOR: Second — the R: race conditions, the one that catches strong programmers off guard, so we'll do it carefully. A race condition is when the correctness of a program depends on the timing or interleaving of operations that can run concurrently — and an attacker manipulates that timing to slip between two steps. The classic security form has a name you should drop in an answer: T-O-C-T-O-U, time-of-check to time-of-use. The bug is a gap between when you check a condition and when you act on it; if state changes in that gap, the action runs on stale information.

NARRATOR: The textbook example is a double-spend. Your account has a hundred dollars. The withdraw code does two steps: step one, check the balance is at least the amount; step two, subtract the amount. An attacker fires two hundred-dollar withdrawals at the same instant. Both run step one — the check — before either runs step two — the subtract. Both see a hundred dollars. Both pass the check. Both subtract. The account goes to minus a hundred, and the attacker withdrew two hundred dollars they didn't have. The check and the use were not atomic — there was a window between them, and the attacker raced into it. The trap to flag, exactly the one that catches good programmers: "my code is single-threaded, so there's no concurrency." Wrong — your web server handles many requests at once, across threads, processes or machines, all hitting the same database. Concurrency is the default on the server, not the exception.

NARRATOR: The fix is to eliminate the gap — make the check and the use atomic, a single indivisible operation that can't be interrupted. Use a database transaction with proper locking, or an atomic conditional update — a single S-Q-L statement that subtracts only if the balance is sufficient, so the check and the change happen as one unit the database serialises. Other tools: locks, mutexes, or a database constraint that forbids a negative balance. The principle: never check-then-act on shared state with a gap between; do it as one atomic step. The exam line: a race condition, specifically time-of-check-to-time-of-use, is exploited by acting in the window between a check and the dependent action while state changes; it's prevented by making the check and action atomic, using transactions, locking, or an atomic conditional update.

QUESTION: This is the one to lock in — pause the player and answer it properly, because race conditions trip people up. Explain how a race condition can be exploited, using an example, and how to prevent it. Pause now, work it out, then play.

NARRATOR: Here's the full-mark answer. A race condition arises when a program's correctness depends on the timing of concurrent operations, and an attacker exploits the gap between a check and the dependent action — a time-of-check-to-time-of-use flaw. For example, a withdrawal that first checks the balance and then deducts the amount can be exploited by sending two withdrawals simultaneously: both pass the balance check before either deducts, so the account is overdrawn — a double-spend. It is prevented by making the check and the action atomic, so they cannot be interleaved: using a database transaction with locking, or a single atomic conditional update that deducts only if sufficient funds exist, ensuring concurrent requests are serialised. Naming the time-of-check-to-time-of-use gap, the concurrent-double-spend example, and the atomic-operation fix is the band-six answer.

NARRATOR: Third — file attacks, previewed in seventeen-oh-two, now treated fully. The headline is path traversal, or directory traversal: the program builds a file path from user input, and the attacker uses dot-dot-slash sequences to climb out of the intended folder up to sensitive files — dot-dot-slash climbs out. Beyond traversal, name unsafe file uploads: an attacker uploads a malicious file — a script disguised with a fake extension — that the server then executes or serves to other users. The mitigations, all variations on "filenames and paths are untrusted input": canonicalise the resolved path and verify it stays inside the intended directory; allow-list permitted filenames or reduce input to the base name only; for uploads, validate the actual content type, not just the extension, cap the size, rename to a server-generated name, and store outside any executable directory. The exam line: a file attack such as path traversal abuses unvalidated filenames to access or place files outside the intended location; it's mitigated by treating paths as untrusted — canonicalising and verifying them, and allow-listing.

NARRATOR: Fourth — side-channel attacks, also previewed in seventeen-oh-two, now a named vulnerability. A side-channel attack extracts secret information by measuring an indirect side effect of the computation — timing, power consumption, electromagnetic emissions, sound, or cache behaviour — rather than attacking the algorithm directly. The secret leaks through timing, power or sound, not the front door. The software-defensible classic is the timing attack: a comparison that exits as soon as characters differ takes measurably longer the more leading characters are correct, letting an attacker recover a secret token or password one character at a time by measuring response times. The fix is the constant-time comparison from the password and execution episodes — a comparison that always takes the same time regardless of how much matches, leaking nothing through timing. More broadly, ensure security-sensitive operations don't vary their timing, memory access or power based on the secret. The exam line: a side-channel attack infers secrets from an indirect effect like timing; a timing attack on a comparison is mitigated by a constant-time comparison.

NARRATOR: Worked example — the weak-versus-strong contrast on "how would you prevent a double-spend in a wallet." This is where the race-condition understanding shows. A weak answer says: "check the balance before allowing the withdrawal." That's actively the bug — checking before withdrawing is exactly what a race condition exploits; the weakness is the gap, not the absence of a check. A strong answer says: "the check and the deduction must be performed atomically so concurrent requests cannot interleave between them. Use a database transaction with row locking, or a single atomic conditional update that deducts the amount only if the balance is sufficient, in one statement — so the database serialises concurrent withdrawals and the balance can never go negative." Hear the difference: the weak answer names the very pattern that fails; the strong answer identifies the gap as the problem and makes the operation atomic. Atomicity, not just a check — that's the mark.

QUESTION: Question one, recall. Describe broken authentication and session management, with two examples. Pause, then play.

NARRATOR: Model answer: broken authentication and session management refers to weaknesses in how a system verifies identity and maintains a logged-in session. Examples include session hijacking, where an attacker steals a valid session identifier and reuses it to impersonate the user; session fixation, where an attacker plants a known session identifier before login; allowing weak passwords or no protection against credential stuffing and brute force; and failing to expire or invalidate sessions. QUESTION: Question two, the key one. Explain how a race condition can be exploited. Pause, then play.

NARRATOR: Model answer: a race condition occurs when a program's correctness depends on the timing of concurrent operations, and it is exploited by acting in the gap between a check and the dependent action — a time-of-check-to-time-of-use flaw. For example, if a withdrawal checks the balance and then deducts the amount as two separate steps, an attacker can send multiple withdrawals simultaneously so they all pass the balance check before any deduction occurs, overdrawing the account. Naming the timing-gap between check and use, with a concurrent example, is the mark.

QUESTION: Question three. Describe a file-based attack and its mitigation. Pause, then play.

NARRATOR: Model answer: a path traversal, or directory traversal, attack occurs when an application builds a file path from unvalidated user input, and the attacker includes "dot-dot-slash" sequences to climb out of the intended directory and access sensitive files elsewhere on the server. It is mitigated by treating filenames and paths as untrusted: canonicalising the resolved path and verifying it remains within the intended directory, rejecting traversal sequences, and using an allow-list or the base name only. Naming the traversal mechanism and the canonicalise-and-allow-list fix is the mark.

QUESTION: Question four. Describe a side-channel attack and how to mitigate it. Pause, then play.

NARRATOR: Model answer: a side-channel attack extracts secret information by measuring an indirect side effect of a computation rather than attacking it directly — most commonly timing. In a timing attack, a comparison that returns as soon as a character mismatches takes longer the more leading characters are correct, allowing the attacker to recover a secret incrementally by measuring response times. The mitigation is a constant-time comparison that always takes the same time regardless of how much matches, leaking no timing information. Naming the indirect-side-effect mechanism, the timing example, and the constant-time fix is the mark.

QUESTION: Question five, the extended one. A digital wallet lets users transfer funds and download their transaction history files. Identify the vulnerabilities and explain how to secure it. Pause, plan a structured response, then play.

NARRATOR: Model answer: the transfer feature is vulnerable to a race condition — concurrent transfers could pass the balance check before any deduction, causing a double-spend; secure it by making the check and deduction atomic using a database transaction with locking or an atomic conditional update so concurrent requests are serialised. The login and session are vulnerable to broken authentication and session management — hijacking or fixation; secure them with strong random session identifiers, HttpOnly, Secure and SameSite cookies, timeouts, regenerating the identifier on login, multi-factor authentication, and rate-limited logins. The file download is vulnerable to path traversal — a crafted filename could read files outside the intended directory; secure it by canonicalising the path, verifying it stays within the allowed directory, and allow-listing. Any token or password comparison should use a constant-time comparison to prevent a side-channel timing attack. A strong response identifies the specific vulnerability per feature, names the mechanism, and gives the targeted, named fix. That's full marks.
