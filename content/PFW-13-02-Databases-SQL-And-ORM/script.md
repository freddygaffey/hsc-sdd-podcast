NARRATOR: Objectives up front: write a SELECT with WHERE, GROUP BY and a JOIN; explain how parameterised queries prevent S-Q-L injection; and compare an O-R-M with raw S-Q-L. The middle one is the highest-value idea in the episode.

NARRATOR: Start with the shape of a query and the mnemonic, because the exam asks you to *write* S-Q-L and the marks evaporate if you get the clause order wrong. The clauses, in order, are SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY. The hook: "So Few Workers Go Home On-time" — So for SELECT, Few for FROM, Workers for WHERE, Go for GROUP BY, Home for HAVING, On-time for ORDER BY. So Few Workers Go Home On-time. That order is not optional: S-Q-L requires it, and writing GROUP BY before WHERE, or ORDER BY in the middle, is a syntax error that costs marks. Write the mnemonic down, then fill in each clause. Now the clauses that carry the marks.

NARRATOR: SELECT and FROM you know — choose the fields, name the table. Two exam points: select only the fields you need rather than SELECT-star, and you can rename a column with AS for clarity. WHERE filters *rows* — which records come back — using conditions, AND, OR, LIKE for pattern matching, BETWEEN for ranges. Then GROUP BY, where students stumble. GROUP BY collapses many rows into groups, used with an *aggregate function* — COUNT, SUM, AVG, MAX, MIN — to summarise each group. "How many posts has each user written?" is GROUP BY user with COUNT of posts. The trap, named: every non-aggregated field in your SELECT must appear in the GROUP BY, and GROUP BY without an aggregate is pointless — the whole reason to group is to count or sum something per group. And HAVING versus WHERE, a favourite distinction: WHERE filters individual rows *before* grouping; HAVING filters *groups after* aggregation. "Users with more than five posts" is HAVING COUNT greater than five, because you only know the count after grouping. WHERE before the group, HAVING after.

QUESTION: Pause the player. In one sentence each, what's the difference between WHERE and HAVING — and why can't you just use WHERE for both? Work it out, then play.

NARRATOR: WHERE filters rows before any grouping, so it can't see an aggregate like COUNT — that doesn't exist yet. HAVING filters after grouping, so it *can* test an aggregate like "COUNT of posts greater than five". You can't use WHERE for the second because the count isn't computed until the rows are grouped. The mark-earning line: WHERE filters individual rows before aggregation; HAVING filters grouped results based on aggregate values. It's a classic short-answer question and a place students routinely lose the mark by using WHERE on an aggregate.

NARRATOR: Now joins — combining data from multiple tables, the heart of a relational database. A relational database splits data into related tables — users in one, posts in another — linked by keys: a post carries an author-id that is a *foreign key* pointing at the *primary key* id in the users table. A JOIN recombines them. Two types you must know. An INNER JOIN returns only rows that match in *both* tables — posts that have an author, dropping any orphans. A LEFT JOIN returns *all* rows from the left table, plus matches from the right where they exist and nulls where they don't — all users, including those with zero posts. The hook: INNER keeps only matches in both; LEFT keeps everything on the left. So "all users and their post count, including users with no posts" needs a LEFT JOIN — an INNER JOIN would silently drop the post-less users. That choice is exactly what an exam tests. Interleave backward: this is the relationship modelling you met in Year 11 with data dictionaries and entity relationships — the foreign-key links between tables are those relationships made real.

NARRATOR: Now the worked example the exam rewards — a query that joins and groups, because "write an S-Q-L statement that joins two tables and groups the results" is a code-editor item. The task: list each user's username and how many published posts they have, most prolific first. Walk it with So Few Workers Go Home On-time. SELECT the username and a COUNT of posts; FROM users; JOIN posts on the author-id matching the user id — a LEFT JOIN so users with no posts still appear; WHERE the post is published, which filters rows before grouping, fine here; GROUP BY the user so COUNT counts per user; ORDER BY that count descending. That's a complete, mark-earning query, in Listing 1. The marks come from: correct clause order, the JOIN with the right key, COUNT as the aggregate, and GROUP BY the user. Get those four and you've got the marks.

NARRATOR: Now the centrepiece — parameterised queries and S-Q-L injection, the cash-in. Here's the danger, concretely. Say you build a query by gluing the user's input straight into the S-Q-L string — "SELECT star FROM users WHERE name equals, quote, plus the user's input, plus quote". Fine when the input is a name. But a malicious user doesn't type a name — they type something like: x, quote, OR, quote, one, quote, equals, quote, one. Now your WHERE clause reads "name equals x OR one equals one" — and one always equals one, so the query returns *every row in the table*. That's S-Q-L injection: the attacker's input broke out of the data and became part of the *command*. Worse versions delete tables or read other users' data. This is consistently one of the most common and damaging web vulnerabilities in the world.

QUESTION: This is the most important question in the episode — pause the player and really think. What is it about gluing strings together that lets the attack work, and how would you stop it? Then play.

NARRATOR: The attack works because the input and the command are mixed into one string, so the database can't tell where your instruction ends and the user's data begins — a quote in the input closes your string early and the rest is read as S-Q-L. The fix is a parameterised query, also called a prepared statement: you write the query with a placeholder — a question mark — where the value goes, and pass the actual value *separately*. The database treats that value strictly as *data*, never as part of the command, so a quote in the input is just a quote in a string — it can't break out. The mark-earning sentence: parameterised queries prevent S-Q-L injection by separating the S-Q-L command from the user-supplied data, so input is always treated as a literal value and never executed as part of the query. Listing 2 shows the vulnerable string-concatenated version next to the safe parameterised one.

NARRATOR: And here's the strong interleave. Remember episode six — never trust the client; validate server-side. Episode twelve — the handler reads untrusted input. This is *that rule, made into a defence*. Parameterised queries are how the Data stage of R-H-D-R defends itself. And it's taught in *full* in Secure Software Architecture, episode sixteen-one, as the primary mitigation for injection attacks — alongside its cousin, cross-site scripting, where the same untrusted input is dropped into H-T-M-L instead of S-Q-L. So when you write a back-end query in an exam, the marker wants a parameterised query and the sentence "this prevents S-Q-L injection by separating data from the command". That single habit is worth marks in both modules. The named trap, plainly: never build S-Q-L by string concatenation with user input — always parameterise.

NARRATOR: Now the second half of the dot-point — comparing an O-R-M to raw S-Q-L. An O-R-M, object-relational mapping, is a library — like SQLAlchemy in Python — that maps database tables to classes and rows to objects, so you work with Python objects instead of writing S-Q-L by hand. A class maps to a table, an instance to a row, and the O-R-M generates the S-Q-L under the hood. Interleave backward: this is your Year 11 object-oriented programming meeting the database — a User class whose instances are rows in the users table. To create a user you make a User object and save it; the O-R-M writes the INSERT.

QUESTION: So if the O-R-M writes the S-Q-L for me, why would I ever write S-Q-L by hand again?

NARRATOR: Because it's a trade-off, and "compare an O-R-M with raw S-Q-L" is a four-mark question that wants both sides. The O-R-M's strengths: faster development and less boilerplate, code that's object-oriented and readable, automatic handling of relationships and joins, and database portability — you can switch from SQLite to PostgreSQL with little change. Its costs: it adds an abstraction layer with some performance overhead, it can generate inefficient queries for complex reporting, and — the named trap — it does *not* remove the need to understand S-Q-L; when the O-R-M produces a slow query, only someone who understands the underlying S-Q-L can fix it. Raw S-Q-L is the opposite: maximum control and performance, full access to complex or database-specific features, but more verbose and more error-prone. The mark-earning comparison: an O-R-M improves productivity and maintainability by mapping objects to tables and generating S-Q-L automatically, at the cost of some performance and control, whereas raw S-Q-L offers maximum control and performance for complex queries at the cost of more code and a higher risk of errors. And the real-world answer is usually hybrid: O-R-M for routine CRUD, raw S-Q-L for complex reports. The same query is written both ways in Listing 3.

NARRATOR: Exam-style finish — five questions, recall to extended. Pause and attempt each, then the model answer.

QUESTION: Question one, recall. In the correct order, name the six main clauses of a SELECT statement. Pause, then play.

NARRATOR: Model answer: SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY — "So Few Workers Go Home On-time". Correct order is the mark; the database enforces it.

QUESTION: Question two. Distinguish between WHERE and HAVING in an S-Q-L query. Pause, then play.

NARRATOR: Model answer: WHERE filters individual rows before any grouping occurs and cannot reference aggregate values; HAVING filters groups after aggregation and is used to test aggregate values such as COUNT or SUM. For example, WHERE excludes inactive users before grouping, while HAVING keeps only groups with more than five posts. Naming "before grouping versus after, on an aggregate" is the mark.

QUESTION: Question three. Explain how a parameterised query prevents S-Q-L injection. Pause, give a real attempt, then play.

NARRATOR: Model answer: a parameterised query uses placeholders for user-supplied values and passes those values to the database separately from the S-Q-L command. The database treats the values strictly as data, never as executable S-Q-L, so malicious input such as a quote followed by extra S-Q-L cannot break out of the data context and alter the query. This separates code from data, which is what prevents injection. The "separates the command from the data" point is the discriminating mark.

QUESTION: Question four. Write, in words, an S-Q-L statement that lists each category's name and the number of published posts in it, for categories with more than three published posts, ordered from most to fewest. Pause, construct it, then play.

NARRATOR: Model answer: SELECT the category name and COUNT of posts; FROM categories; INNER JOIN posts on the post's category-id matching the category id; WHERE the post is published; GROUP BY the category; HAVING COUNT of posts greater than three; ORDER BY that count descending. The marks are in: the JOIN on the correct key, WHERE filtering published rows before grouping, GROUP BY the category, the aggregate COUNT, HAVING testing that aggregate, and ORDER BY descending — all in the So-Few-Workers order. That's a complete, well-formed query.

QUESTION: Question five, the extended one. Compare the use of an O-R-M with writing raw S-Q-L for a web application's data layer. Pause, plan a balanced response, then play.

NARRATOR: Model answer: an O-R-M, object-relational mapping, maps database tables to classes and rows to objects, automatically generating S-Q-L. Its benefits are higher productivity and readability, because developers work with familiar objects and write less boilerplate; automatic handling of relationships and joins; database portability; and built-in protection against injection through parameterisation. Its drawbacks are an abstraction layer that adds performance overhead and can generate inefficient queries for complex operations, and the fact that developers still need to understand the underlying S-Q-L to diagnose and optimise those queries. Raw S-Q-L, by contrast, gives maximum control and performance and full access to complex or database-specific features, but is more verbose, more error-prone, and requires careful use of parameterised queries for safety. In practice a hybrid approach is common: an O-R-M for routine CRUD operations and raw S-Q-L for performance-critical or complex reporting. That response gives both sides with concrete benefits and costs and resolves with the hybrid judgement — a full-mark "compare" answer.
