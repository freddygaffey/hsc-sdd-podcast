NARRATOR: Objectives up front, so your attention knows what to catch. By the end you'll distinguish a functional requirement from a non-functional one and write both so they're testable; explain how feasibility is assessed before a project starts; define what data the system handles using a data dictionary; and set boundaries so the project doesn't quietly grow forever. The classic exam questions on this dot-point are "distinguish functional from non-functional requirements" for around four marks, and "explain how feasibility is assessed for a software project" for around six. We'll drill both.

NARRATOR: The core idea. A requirement is a statement of something the software must do or must be. The whole game in requirements analysis is turning a fuzzy wish — "we want a better way to track assignments" — into specific, testable statements you could check off at the end. The syllabus dot-point: define and analyse the requirements of a problem, including demonstrating needs or opportunities, assessing scheduling and financial feasibility, generating requirements including functionality and performance, defining data structures and data types, and defining boundaries. That's our whole episode, in the syllabus' own words. In order.

NARRATOR: First, needs and opportunities. Before you write a single requirement, you justify that the project should exist at all — and "justify" is the S-E-twelve-oh-one verb. A need is a problem someone actually has: teachers spend two hours a day on manual data entry; students miss deadlines because reminders are scattered. An opportunity is a gap you could fill: no existing tool integrates with the school's systems. You demonstrate these with research — stakeholder interviews, surveys, observing how people do the task now, and reviewing competing solutions. The exam word here is stakeholder: anyone who uses, pays for, maintains, or is affected by the software. The four groups to name: end users; business stakeholders — who decide and hold the budget; technical stakeholders like developers and maintainers; and regulatory stakeholders like compliance and legal. A strong answer names stakeholder groups; a weak one just says "the users."

NARRATOR: Now the heart of the topic, and the most-examined part: the two kinds of requirements. Functional requirements describe what the system must do — the specific behaviours and features. "The system shall let a teacher record a grade." "The system shall send a deadline reminder." Functions, actions, verbs. Non-functional requirements describe how well the system must do it — the quality attributes. Speed, reliability, security, usability, scalability, maintainability. "Pages shall load in under two seconds." "The system shall support two hundred simultaneous users." Same system, two questions: what does it do, versus how well does it do it.

NARRATOR: Here's the memory hook — use it in the exam. Functional versus non-functional: remember it as FUN versus PERF. FUN — functional — is what it does, the features, the fun stuff the user asked for. PERF — performance, standing in for the whole non-functional family — is how well it does it. FUN is the verbs; PERF is the adjectives and numbers. When a question hands you a requirement and asks which kind, ask: is this a thing the system does, or a quality of how it does it? "Let users log in" is FUN. "Log in within three seconds even at peak load" is PERF. FUN is what, PERF is how well.

QUESTION: Quick check before we go further. I'll give you three requirements; you classify each as functional or non-functional. One: the system allows a student to view their own grades. Two: the system encrypts all stored data. Three: ninety-five percent of pages load in under two seconds. Pause the player, classify all three, then play.

NARRATOR: Here they are. One — view their own grades — is functional. It's a thing the system does, a feature. Two — encrypts all stored data — this one trips people up. Security is a non-functional requirement, a quality attribute of the whole system; it's the how-well, not a user-facing feature. PERF, in our hook's broad sense. Three — ninety-five percent of pages under two seconds — clearly non-functional, it's performance, and notice it's measurable. Now hold onto that word measurable, because it's the single biggest mark-earner in this whole topic.

NARRATOR: A good requirement is testable: at the end of the project you run a clear check and get a yes or a no — was this met? This is where weak answers die. Hear the contrast. A weak non-functional requirement says "the system should be fast." Useless — fast to whom, measured how, under what load? You can't test it, so you can't prove you delivered it. A strong one says "the ninety-fifth percentile page-load time shall be under three hundred milliseconds with two hundred concurrent users." Now it's measurable: you load-test it, read the number, get a yes or no. The exam phrasing that earns the mark is built around acceptance criteria — the specific, measurable conditions that define when the requirement is satisfied. The rule: every requirement, functional or not, comes with acceptance criteria you could actually test. "It should be fast" is band four. "P-ninety-five under three hundred milliseconds at two hundred users" is band six.

NARRATOR: Here's a clean interleave back to Secure Software Architecture — exactly the kind of link this module is built on. Remember the first SSA episode, the business case for security? Security was a business enabler, not a bolt-on. And security-by-design, shift-left — building security in from the start rather than bolting it on at the end? That lands right here: the way you build security in from the start is to write it as a requirement at this stage. A non-functional security requirement — "all sensitive data shall be encrypted at rest and in transit," "passwords shall be stored salted and hashed," your "hash is one-way, encrypt is two-way" line — written now, is what shifting-left actually looks like. Likewise, "the interface shall meet W-C-A-G two-point-one double-A," your accessibility standard from Programming for the Web — POUR, perceivable operable understandable robust — is a non-functional usability requirement. So the requirements stage is where the security and accessibility you learned all year get committed to. They're not separate topics; they're requirements.

NARRATOR: Next piece: defining data structures and data types. The syllabus wants you to specify, at requirements time, what data the system stores and in what form. The tool is the data dictionary — a direct callback to Year 11 programming fundamentals, where you first met data dictionaries to describe data and relationships. A data dictionary is a table listing every data field the system handles, with, for each: field name, data type, whether it's required, any constraints — length limits, valid ranges — and a short description. For an assignment-tracker: a field due-date, type Date, required yes; priority, type Integer, range one to five, required no; completed, type Boolean, required yes. There's a full data dictionary in Listing one if you want to read it later — but you don't need to; it's just a structured table that pins down every piece of data before you build. Choosing the right data type is part of this: a price is a Decimal, not a float, so money doesn't pick up rounding errors; an enrolment date is a Date, not a String, so you can do date arithmetic. The marker rewards naming appropriate types and justifying them.

NARRATOR: And the data dictionary is where this module quietly reuses your whole web stack. Those fields, types, and relationships between Student, Grade, and Assignment — that's the relational modelling from Programming for the Web, thirteen-oh-two, where primary keys link to foreign keys and a Student has many Grades. The data dictionary you write now becomes the database schema you build later. SQL is a spreadsheet of tables; the data dictionary is the spec for those tables. One continuous thread: requirements data dictionary, then relational schema, then the actual database.

NARRATOR: Now feasibility — the second half of the title, and the six-mark question. Feasibility assessment asks: can we realistically build this within our constraints? You can have perfect requirements for a system you have no chance of delivering. Feasibility is the honest gut-check before you commit. Here's the memory hook for the dimensions: TECO. T-E-C-O. Technical, Economic, Cost-and-schedule, Operational. Each one.

NARRATOR: Technical feasibility: do we have the skills, tools, and technology to build this — can it even be done with current technology and our team's ability? A two-person part-time team proposing a real-time machine-learning fraud-detection engine in six weeks is technically infeasible — not impossible in the universe, but impossible for them, then. Economic feasibility: do the benefits justify the cost? This is the cost-benefit analysis — development costs and ongoing operational costs like hosting and maintenance, weighed against expected benefits like saved time or new revenue. The textbook computes a return on investment: total benefits minus total costs, over costs. If the project costs more than it'll ever save, it's economically infeasible no matter how nice it'd be. Cost-and-schedule feasibility — the C — is can we finish in the time available on the budget we have: resource availability, task dependencies, and crucially a risk buffer, slack for things going wrong. And operational feasibility: once it's built, will it fit how people work, and will they use it? A technically perfect system teachers refuse to adopt has failed operationally — that's the security-usability paradox from SSA fourteen-oh-three pointed at a whole product: if it doesn't fit the humans, they route around it.

QUESTION: Pause-the-player, the highest-value retrieval in this episode. Without looking, write down the four dimensions of feasibility from the hook TECO, and give a one-line meaning for each. Pause now, dump them, then play to check.

NARRATOR: Here they are. T, Technical — do we have the skills, tools, and technology to build it. E, Economic — do the benefits outweigh the costs, the cost-benefit and return-on-investment question. C, Cost-and-schedule — can we deliver within the budget and time available, with a risk buffer. O, Operational — will it fit how people actually work, so they'll adopt it. Rattle off TECO with a meaning each, and you've got the six-mark feasibility answer. The mark-earning move on "explain how feasibility is assessed" is to name each dimension and say what question it answers — not just "we check if it's possible."

NARRATOR: Last piece: boundaries, also called scope. Defining boundaries means stating clearly what is in the project and what is explicitly not. The textbook splits it three ways: in scope — delivered now; out of scope — explicitly excluded, with a reason; and future scope — deferred to a later phase. For the school system: in scope, grade recording and viewing; out of scope, fee collection, handled by an existing system; future scope, a mobile app in phase three. Why does this matter so much? One word, and it's the trap: scope creep. Scope creep is the slow, uncontrolled expansion of a project as "just one more feature" gets added again and again, until the deadline blows out and quality collapses. Clear boundaries, written down and agreed, are the defence. When a stakeholder asks for something new mid-project, you point at the boundary document: "that's future scope, phase two." Boundaries aren't bureaucracy — they're how you protect the schedule you just said was feasible.

NARRATOR: And that connects forward — this module is one long forward-and-back weave. The requirements you write today become two things later. They become the quality criteria in quality assurance — episode twenty-four-oh-six, where "meeting the requirements" is literally the definition of quality. And they become the yardstick in your final evaluation — episode twenty-six-oh-three, where you evaluate the solution by comparing what it does against the requirements you wrote right now. So sloppy requirements here don't just hurt the build; they make it impossible to ever say whether you succeeded, because you defined no target to hit. Requirements are the spec, the quality bar, and the exam question rolled into one.

NARRATOR: Anchor this in a story — the most-cited requirements-and-feasibility failure in software history, a case study this module cashes in later. Denver International Airport, early nineteen-nineties. They commissioned a fully automated baggage-handling system — an enormous network of tracks and carts to route every bag automatically across the entire airport. The ambition was staggering, and so was the feasibility gap. Scope was set far beyond what the technology could reliably do, the timeline was fixed by the airport opening date, and the boundaries kept expanding as more airlines and requirements got bolted on. The result: bags shredded, mis-routed, and lost in testing; the system never worked reliably; the airport opened sixteen months late; and the overrun ran into hundreds of millions of dollars. The lesson, in this episode's exact terms: the requirements were not properly bounded, and the project was neither technically nor schedule-feasible, but nobody made the honest feasibility call early enough to stop it. We'll tell the full story at implementation methods, because the cut-over was a disaster too — for now, hold it as the cautionary tale: unbounded scope plus an infeasible plan equals a famous failure.

NARRATOR: Now the worked example — where the marks are. The brief: a school wants a system so students can see their grades and teachers can record them. Three functional and two non-functional requirements, done properly. Functional one: the system shall allow an authenticated teacher to record a grade for a student in their own class. Functional two: the system shall allow a student to view their own grades but not change them or view others' — that's authorisation, role-based access control, straight from CIA-AAA. Functional three: the system shall calculate a weighted average grade automatically from recorded assessments. Non-functional one — the weak-versus-strong moment — don't write "the system should be secure." Write: the system shall store passwords as salted, slow hashes and transmit all data over T-L-S, so a stolen database can't reveal credentials. That's a non-functional security requirement with a testable mechanism. Non-functional two: ninety-five percent of page loads shall complete in under three hundred milliseconds with two hundred concurrent users, verified by load testing. Five requirements, each testable, each tied to a stakeholder need. That structure — labelled requirements, classified, with acceptance criteria — is the band-six shape.

NARRATOR: Hear the weak-versus-strong contrast once more, sharply, because it's the difference between a four and a six. Weak answer to "write a non-functional requirement for performance": "the website should be quick and not crash." No metric, no condition, untestable — band four. Strong answer: "the system shall maintain a ninety-fifth-percentile response time under three hundred milliseconds while serving two hundred concurrent users, and shall maintain ninety-nine-point-five percent uptime, both verified by automated load and availability testing." Hear the difference: a number, a condition, and how you'd verify it. Measurable, testable, verifiable. That's the mark.

NARRATOR: Lock in today. A requirement is a testable statement of what the software must do or be. Functional versus non-functional is FUN versus PERF — what it does versus how well it does it — and the single most important property of any requirement is that it's measurable and testable, written with acceptance criteria. Feasibility is TECO — Technical, Economic, Cost-and-schedule, Operational — assessed before you commit. Boundaries define what's in, out, and future, and they exist to stop scope creep. The through-line: the requirements you write here become your quality criteria in twenty-four-oh-six and your evaluation yardstick in twenty-six-oh-three, and they're where the security and accessibility from all year get committed to. Traps: vague untestable requirements like "it should be fast"; and unbounded scope, the Denver baggage trap.

NARRATOR: Exam-style finish — four questions, recall up to extended. Pause and attempt each before the model answer; this is your retrieval over the whole episode.

QUESTION: Question one, recall. Distinguish between functional and non-functional requirements, with an example of each. Pause, then play.

NARRATOR: Model answer: a functional requirement specifies what the system must do — a specific behaviour or feature — for example, "the system shall allow a teacher to record a student's grade." A non-functional requirement specifies how well the system must perform — a quality attribute such as performance, security, reliability, or usability — for example, "the system shall load pages in under two seconds for ninety-five percent of requests." Functional is what it does; non-functional is how well it does it — FUN versus PERF. Stating both definitions, the what-versus-how-well contrast, and a clear example of each is the mark.

QUESTION: Question two, application. Rewrite this weak requirement so it is testable: "The system should be fast and secure." Pause, then play.

NARRATOR: Model answer: split it into two measurable non-functional requirements. For performance: "The system shall return a response within three hundred milliseconds for ninety-five percent of requests under a load of two hundred concurrent users, verified by load testing." For security: "The system shall store all passwords as salted, slow hashes and transmit all data over T-L-S, and shall log all access to sensitive records, verified by a security review." The mark is for replacing the vague adjectives with measurable acceptance criteria and a method of verification — naming the metric, the condition, and how you'd test it.

QUESTION: Question three, the six-marker. Explain how feasibility is assessed for a software engineering project. Pause, plan a structured answer, then play.

NARRATOR: Model answer: feasibility assessment determines whether the proposed solution can realistically be delivered within constraints, assessed across several dimensions — TECO. Technical feasibility asks whether the team has the skills, tools, and technology to build it, and whether it's achievable at all. Economic feasibility is a cost-benefit analysis: it weighs development and ongoing operational costs against expected benefits, often expressed as a return on investment, to confirm the benefits justify the spend. Cost-and-schedule feasibility asks whether the project can be completed within the available budget and timeframe given resource availability and task dependencies, including a risk buffer for the unexpected. Operational feasibility asks whether the finished system will fit how people actually work, so they adopt it rather than route around it. Assessing all four early lets an organisation make a justified go-or-no-go decision before committing resources — for example, the Denver airport baggage system failed largely because it was neither technically nor schedule-feasible, but that judgement wasn't made in time. Naming the dimensions, saying what question each answers, and framing it as an upfront go-or-no-go decision is the full-mark answer.

QUESTION: Question four, extended application. A small startup and a national hospital both want a patient-records system. Explain how their requirements and feasibility would differ. Pause, plan across the episode, then play.

NARRATOR: Model answer: both share the core functional requirements — record, store, and retrieve patient data — but their non-functional requirements and feasibility differ sharply. The hospital faces far stricter non-functional requirements: confidentiality and integrity of medical records are critical, so encryption, strict role-based authorisation, and full audit logging — the whole CIA-AAA set — are mandatory, and compliance with health-privacy legislation is a hard regulatory requirement, not a nice-to-have. Availability is also critical: the system must be reachable twenty-four-seven. The startup's non-functional requirements are lighter and it can trade some robustness for speed to market. On feasibility: the hospital has greater economic resources but tighter operational and regulatory constraints and a larger stakeholder set, so its boundaries must be drawn carefully to keep scope feasible; the startup has less budget and a smaller team, making cost-and-schedule feasibility its binding constraint, so it should narrow scope aggressively to what's deliverable. The judgement: same functional core, but feasibility and the non-functional bar are driven by context — risk, scale, and regulation. An answer that distinguishes shared functional requirements from context-driven non-functional requirements, and ties feasibility to each organisation's constraints, earns full marks.

NARRATOR: That's episode one of the Software Engineering Project — and the start of the synthesis. Requirements turn a fuzzy problem into testable statements: FUN is what it does, PERF is how well, and everything must be measurable. Feasibility is TECO, the honest can-we-do-this call. Boundaries stop scope creep. And remember the marker for this whole module: justify, manage, document — not just code. ### Listing 1 — Data dictionary for the assignment-tracker (referenced in narration)

| Field | Data type | Required | Constraints | Description |
|-------|-----------|----------|-------------|-------------|
| `assignment_id` | Integer | Yes | Primary key, auto-increment | Unique identifier for the assignment |
| `title` | String | Yes | 1–120 characters | Human-readable assignment name |
| `due_date` | Date | Yes | Must be today or later on creation | Deadline for submission |
| `priority` | Integer | No | Range 1–5 (default 3) | Relative importance, 1 = low, 5 = high |
| `completed` | Boolean | Yes | Default false | Whether the assignment is finished |
| `student_id` | Integer | Yes | Foreign key → Student.student_id | Owning student (relational link) |

Note the chosen types: `Date` (not `String`) so date arithmetic works; an `Integer` range constraint on `priority`; a `Boolean` for a yes/no flag; and the `student_id` foreign key that turns this dictionary into a relational schema (PFW 13-02). A monetary field, if present, would be `Decimal` — never a binary `float` — to avoid rounding errors.
