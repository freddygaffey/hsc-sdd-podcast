### PFW-01-Applications-Of-Web-Programming
- Title: Applications of Web Programming
- Terms: three syllabus application types — interactive website/webpage (defining trait = dynamic, client-side content updating without full reload), e-commerce (defining trait = secure financial transactions + protection of sensitive/personal data; security is the distinguishing requirement because the asset is money/identity), progressive web app/PWA (defining trait = installable + offline-capable via a service worker + web app manifest); trade-offs/constraints per type; exam skill = match a scenario's driving requirement (interaction / money / offline) to the right type and rule out the others.

### PFW-02-How-Data-Moves-On-The-Internet
- Title: How Data Moves on the Internet — Packets, IP and DNS
- Terms: data packets (numbered units; reasons to packetise = efficient sharing of links + independent routing/resilience + resend only the lost packet); packets arrive out of order → reassembled by sequence number; IP address = machine's numeric location; IPv4 = four numbers 0–255 with dots (each octet = 8 bits = 256 values); IPv6 exists to solve IPv4 address exhaustion; DNS = translates human-readable domain name → server IP address by walking a hierarchy (cache → resolver → root → TLD → authoritative), cached under TTL (time to live); request journey order = DNS → TCP → TLS → HTTP. (TCP/TLS/HTTP + ports taught in 11-03.)
- Mnemonics: "Dogs Take Tasty Hambones" = DNS → TCP → TLS → HTTP (request journey order — coined here, reuse in 11-03 & 13-01); IPv4 = "four numbers, 0–255, dots between"; DNS triple = "name, number, translator" (domain name / IP address / DNS).
- Traps: confusing the IP address with the DNS name (or saying "DNS is the address" — DNS is the translator between name and number); assuming packets arrive in order (they're numbered & reassembled); saying DNS "loads/fetches the page" (that's HTTP's job — DNS only resolves the name to an IP).
- Forward hooks planted: DNS as single point of failure → case_the_dyn_dns_ddos_2016 (+ SSA availability, SA insecure IoT); caching-closer-to-user → CDNs/big-data architecture in 11-04; TCP/TLS/HTTP + ports + "80 plain, 443 safe" → 11-03.

### PFW-03-Web-Protocols-Ports-And-Secure-Transport
- Title: Web Protocols, Ports and Secure Transport
- Terms: protocol = agreed rules; port = numbered service doorway; ports — HTTP 80, HTTPS 443, SSH/SFTP 22, FTP 21, DNS 53 (SMTP 25/587, POP3 110/995, IMAP 143/993); SSL = old, TLS = modern successor (current); plain text vs cipher text (encryption transforms plain→cipher, decryption reverses); symmetric = same key (fast), asymmetric = key pair public/private (solves key sharing, slow); TLS handshake = server sends CA-signed certificate (authenticates identity + carries public key) → asymmetric to agree session key → symmetric for bulk traffic; authentication = who you are, authorisation = what you can do; hash = one-way fixed-size fingerprint (integrity + password storage); digital signature = sign with private key / verify with public key → authenticity + non-repudiation + tamper detection. (All crypto taught in full in SSA; this is the PFW intro.)
- Mnemonics: "80 plain, 443 safe; 22 secure shell, 21 file" (ports — coined here, reuse); "Symmetric = Same key; Asymmetric = A pair"; "Authentication = who you are; Authorisation = what you can do"; "Hash is one-way, Encrypt is two-way" (SSA marquee — reuse verbatim).
- Traps: SSL vs TLS (TLS is the current one; don't treat as identical); a certificate AUTHENTICATES + carries the public key, it does NOT itself encrypt (the keys do); confusing hashing with encryption (one-way vs two-way); writing a checkout on port 80 (plain text) as secure.
- Forward hooks planted: full crypto + CIA-AAA triad + hash-vs-encrypt → Secure Software Architecture (15-01/15-02, 16-x); completes the "Dogs Take Tasty Hambones" journey (Take/Tasty/Hambones = TCP/TLS/HTTP); ties sideways to e-commerce security requirement from 11-01.

### PFW-04-Big-Data-And-Web-Architectures
- Title: Big Data and Web Architectures
- Terms: big data = the three Vs (Volume, Velocity, Variety) — NOT just "lots of data"; data mining = analysing large datasets to discover previously-unknown patterns/trends (≠ storage/retrieval); metadata = data about data (e.g. film genre/cast/year) → powers search/categorisation/recommendations; streaming service management = handling continuous high-velocity real-time data (buffering, throughput, reliability, overflow/drop); architecture patterns = monolithic (one deployable unit) / microservices (independent services, scale + fail in isolation, API gateway) / serverless (on-demand functions, auto-scale to zero, pay-per-use); CDN = edge servers caching content close to users (same "closer to the user" idea as DNS caching); exam skill = map a big-data property (volume/velocity/variety) → the architectural decision it forces.
- Mnemonics: "three Vs = Volume, Velocity, Variety" (define big data); "M-M-S" = Mining, Metadata, Streaming (the bag-of-M&Ms sorted by colour); architecture = "one block, many services, tiny functions" (monolith → microservices → serverless).
- Traps: defining big data as merely "lots of rows/data" (must give the 3 Vs); confusing data mining (pattern discovery) with data storage (retention/retrieval); treating "bigger faster server" as the architecture answer instead of microservices/CDN.
- Forward hooks planted: CDN/caching → performance toolkit in 13-03; centralisation → blast radius → case_the_cloudflare_2019_outage (+ SEE testing/deployment); metadata/machine-readable data → W3C in 12-01; big data as ML training data + bias → SA 22-03.

### PFW-05-The-W3C-And-Web-Standards
- Title: The W3C and Web Standards
- Terms: W3C = international body (founded 1994 by Tim Berners-Lee) developing open web standards → interoperability/consistency across browsers & devices; it RECOMMENDS (a finished standard = a "W3C Recommendation"), does NOT enforce; five concerns = WAI accessibility, Internationalisation, Privacy, Security, Machine-readable data; WCAG = Web Content Accessibility Guidelines (WAI) built on 4 principles Perceivable/Operable/Understandable/Robust; accessibility techniques = labels + alt text (Perceivable), ≥4.5:1 colour contrast (Perceivable), keyboard operability + visible focus (Operable); internationalisation (i18n) = preparing software for many locales vs localisation (l10n) = adapting to one (translate text, currency, date/number formats, RTL); privacy-by-design; machine-readable data = ARIA/microdata/JSON-LD (= metadata from 11.4 made into a standard).
- Mnemonics: "WIPS-M" = WAI accessibility, Internationalisation, Privacy, Security, Machine-readable (W3C's five concerns — wipe the M off the whiteboard); "POUR" = Perceivable, Operable, Understandable, Robust (WCAG principles); i18n vs l10n = "prepare vs adapt".
- Traps: writing the W3C "enforces" standards (it recommends; voluntary adoption); treating accessibility as "for blind users only" (covers motor/auditory/cognitive/situational); giving vague accessibility answers instead of naming technique + WCAG principle + the 4.5:1 ratio.
- Forward hooks planted: accessibility returns in 12-04 (CSS/UI/UX) + SEE 25-06 (prototype design); web security + privacy taught in full in SSA (16-02 privacy); machine-readable data ties back to metadata (11-04).

### PFW-06-Modelling-A-Web-Development-System
- Title: Modelling a Web Development System
- Terms: three tiers — client-side/front-end (browser; HTML structure, CSS style, JavaScript behaviour; presentation + interaction; visible & tamperable) / server-side/back-end (server; Python/Flask; routing, business logic, data access; trusted) / database = the Store (persists data; only the back end queries it); client-side processing runs on user's device (untrusted) vs server-side runs on developer's server (trusted) → consequence: validation/auth/sensitive logic MUST be server-side; SQL/relational = fixed schema tables + relationships + ACID (Postgres/MySQL/SQLite) vs NoSQL/non-SQL = flexible documents/JSON, schema-less, horizontal scaling (MongoDB/Redis); middleware = sits in request path doing cross-cutting jobs (auth/logging/rate-limit); API = defined contract between components (also a security boundary).
- Mnemonics: "Front, Back, Store" (the three tiers — coined here, reuse in 13-01/13-02); "SQL = a Spreadsheet of tables; NoSQL = a Notebook of documents".
- Traps: trusting client-side validation alone (can be bypassed — must re-validate server-side; SSA "validate everything"); thinking NoSQL = "SQL is obsolete/can't use SQL" (NoSQL = "not only SQL", tool for a data shape, most systems use both); shallow client-vs-server answers that omit the trust consequence.
- Forward hooks planted: HTTP request from "Dogs Take Tasty Hambones" (11-02) arrives at the Back tier; Back built for real in 13-01; Store/SQL = 13-02; client-vs-server trust + must-validate-server-side + secure API boundary → SSA 16-01/17-01.

### PFW-07-The-Browser-And-Developer-Tools
- Title: The Browser and Developer Tools
- Terms: a browser RUNS the page via a rendering engine + a JavaScript engine (Chrome=Blink/V8, Firefox=Gecko/SpiderMonkey, Safari=WebKit/JavaScriptCore) → same code can behave differently = cross-browser compatibility; responses = cross-browser testing, feature detection (test the capability, not the browser name), progressive enhancement (baseline + layered enhancements); same-origin policy = browser-enforced security boundary, cross-origin blocked unless CORS allows (prelude → SSA); dev-tool panels = Elements (live DOM + applied CSS → layout/style bugs), Network (every HTTP request: status code, timing, size → failed/slow requests), Storage (cookies/localStorage/session/service-worker caches → login/session/cache bugs), Performance (slow JS + paint/render → sluggish pages), plus Console (messages/errors); status codes localise faults: 404 = missing endpoint (request side), 401/403 = auth/authz (ep.3), 500 = server-side code error.
- Mnemonics: "E-N-S-P" = Elements, Network, Storage, Performance (dev-tool panels, + Console); status triage = "4xx = the request/permissions; 5xx = the server".
- Traps: thinking client-side code/console can hide secrets or enforce security (everything client-side is visible & editable via the same dev tools — never trust the client; mirror of 12-02's validate-server-side); using browser detection (UA string) instead of feature detection.
- Forward hooks planted: Network panel = the "Dogs Take Tasty Hambones" request journey (11-02) made visible; same-origin/CORS → SSA 17-01 secure APIs; service-worker caches (Storage) → PWAs 13-04; client-is-an-open-book reinforces SSA trust boundary; debugging echoes Year 11 debugging tools.

### PFW-08-CSS-UI-And-UX-Principles
- Title: CSS, UI and UX Principles
- Terms: separation of concerns = HTML structure / CSS presentation / JS behaviour; CSS → maintainability via external stylesheet (single source of truth: change once → updates everywhere) + design tokens (CSS custom properties for colour/spacing/font); responsive design = layout adapts to screen size via media queries, mobile-first (≠ a PWA, which = installable+offline); UI = the look (buttons/colour/layout) vs UX = the experience (easy/efficient/satisfying to use); UI/UX principles = Consistency, Feedback, Accessibility, Navigation, visual Hierarchy; accessibility/inclusivity techniques (fulfil WCAG/POUR) = ≥4.5:1 contrast, keyboard-operable + visible focus, never colour alone (add icon+text), text alternatives (alt text, captions/transcripts for audio/video), large touch targets (≥44px), designed in from the start.

- Traps: inline styles kill maintainability (use external stylesheet/tokens); accessibility as an afterthought (must be designed in = inclusivity); confusing "looks good" (UI) with "is usable" (UX); thinking responsive = PWA.

### PFW-12-05-Front-End-Libraries-And-Frameworks
- Title: Front-End Libraries and Frameworks

### PFW-12-06-Open-Source-And-Content-Management-Systems
- Title: Open-Source Software and Content Management Systems
- Terms: open source = source code public to view/modify/distribute, developed openly by a community ("free as in freedom, not necessarily price"; openness = strength, not weakness); pillars = Licence, Community (maintainers/contributors/users), Contribution (fork→branch→pull request = version control at scale); licence families = permissive (MIT/Apache/BSD — allow proprietary/closed commercial use, just attribution) vs copyleft (GPL — distributed derivatives must ALSO be open-sourced); software supply chain = depending on third-party code + its dependencies = an attack surface (xz backdoor 2024); benefits of OSS = free/cheap, fast (reuse mature code), transparent/inspectable, no lock-in; risks = supply-chain vulns, abandoned/unmaintained projects, licence (copyleft) obligations; CMS = software to create/manage/publish content via admin UI without coding (WordPress dominant; built from DB + template engine + admin = Front/Back/Store); hosted (provider manages updates/backups/security, limited control, ongoing fee) vs self-hosted (full control + data ownership, but YOU carry security/maintenance).
- Mnemonics: "L-C-C" = Licence, Community, Contribution (OSS pillars); "Permissive = Polite (use freely, say thanks); Copyleft = Contagious (derivative must share alike)"; CMS choice = "hosted trades control for managed convenience; self-hosted trades responsibility for control".
- Traps: "open source = free / insecure / low quality" myths (it's freedom not price; often MORE trustworthy via inspection); ignoring licence obligations (copyleft can force your proprietary product open); leaving a CMS + its plugins unpatched (commonest small-site hack).
- Forward hooks planted: supply-chain trust → case_the_xz_backdoor + SSA 18-x/19-x (supply-chain & patching); CMS/plugin patching → SSA security updates; contribution workflow = version control (Year 11 + SEE 25-04); CMS built from template engine (12-05) + Front/Back/Store (12-02). ### PFW-13-01-Server-Side-Python-And-The-Request-Flow
- Title: Server-Side Python and the Back-End Request Flow
- Terms: back-end request lifecycle = Route (URL→handler) → Handle (business logic, read request object) → Data (query/update DB) → Respond (render HTML via template engine / return JSON); web server software (nginx/Apache) accepts connections + parses HTTP + serves static files + TLS termination, FORWARDS dynamic requests to the web framework (Flask) where routes/handlers/business logic live — different layers; request/response/session objects; libraries (hashing, DB, external APIs); CRUD = Create/Read/Update/Delete maps to SQL INSERT/SELECT/UPDATE/DELETE and HTTP POST/GET/PUT/DELETE; untrusted input = anything from the request must be validated/sanitised server-side (concatenating it → SQL injection; echoing it into HTML → XSS); shell scripts (mkdir, touch, grep) = part of the dev toolkit/scripting.
- Mnemonics: "R-H-D-R" = Route, Handle, Data, Respond (back-end lifecycle); "CRUD = Create/Read/Update/Delete = INSERT/SELECT/UPDATE/DELETE = POST/GET/PUT/DELETE" (reuse in 13-02); reuse Front/Back/Store + "Dogs Take Tasty Hambones" (HTTP request arrives at the Back).
- Traps: confusing the web server (nginx) with the web framework (Flask) — different layers; trusting client input / relying on client-side validation (bypassable — must re-validate server-side); gluing user input into SQL (injection) or HTML (XSS).
- Forward hooks planted: parameterised queries + SQL/joins/ORM = 13-02 (+ "So Few Workers Go Home On-time"); never-trust-input + injection/XSS taught in full in SSA 16-01/17-01/18-x; CRUD↔HTTP↔SQL mapping → 13-02; shell scripting → CI/CD in SEE + Software Automation DevOps.

### PFW-13-02-Databases-SQL-And-ORM
- Title: Databases — SQL, Common Queries and the ORM
- Terms: SQL clause order = SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY (DB enforces it); WHERE filters rows BEFORE grouping (can't see aggregates) vs HAVING filters GROUPS after aggregation (tests COUNT/SUM); GROUP BY needs an aggregate (COUNT/SUM/AVG/MAX/MIN) + every non-aggregated SELECT field; relational tables linked by primary key ↔ foreign key; INNER JOIN = only rows matching in BOTH tables, LEFT JOIN = ALL left rows + matches (nulls otherwise, keeps zero-match rows); SQL injection = gluing user input into the command string lets input become SQL (x' OR '1'='1 → returns all rows); parameterised query / prepared statement = placeholder (?) + value passed SEPARATELY → DB treats it as data not command → prevents injection; ORM (SQLAlchemy) = maps classes↔tables, objects↔rows, auto-generates SQL (reuses Year 11 OOP); ORM trade-off = productivity/readability/portability/auto-parameterised vs performance overhead + can't escape needing to know SQL; hybrid common (ORM for CRUD, raw SQL for complex reports).
- Mnemonics: "So Few Workers Go Home On-time" = SELECT/FROM/WHERE/GROUP BY/HAVING/ORDER BY (clause order); "INNER keeps matches in both; LEFT keeps everything on the left"; reuse CRUD↔SQL↔HTTP (from 13-01).
- Traps: wrong clause order (syntax error); using WHERE on an aggregate instead of HAVING; GROUP BY without an aggregate / missing non-aggregated fields; building SQL by string concatenation (injection — always parameterise); "an ORM means I don't need to know SQL" (still must, to fix slow queries); using INNER JOIN when you need LEFT (silently drops zero-match rows).
- Forward hooks planted: parameterised queries = the marquee SQL-injection mitigation taught in FULL in SSA 16-01 (+ XSS = same untrusted input into HTML); joins/keys = Year 11 data dictionaries/relationships; ORM = Year 11 OOP; this completes the Data stage of R-H-D-R (13-01). ### PFW-13-03-Performance-And-Page-Load-Management
- Title: Performance and Page-Load Management
- Terms: measure-before-optimise = profile with dev tools (Network = big/slow requests, Performance = slow JS) to find the real bottleneck (avoid premature optimisation); core toolkit = Caching (browser cache via Cache-Control headers + server-side cache e.g. Redis for expensive query results; "store closer, reuse"), Compression (gzip/Brotli shrinks text bytes 60–80%; + minify CSS/JS), CDN (geographically distributed edge servers cache static assets near users → lower latency + less origin load), Lazy-load (defer non-critical/below-the-fold resources, loading="lazy", so initial render is fast); bundling/minification = combine files → fewer requests (heavy framework = big bundle = load cost, ties to 12-05); caching = same "closer to the user" idea as DNS caching (11-02) and CDN (11-04) — 3rd appearance.
- Mnemonics: "C-C-C-L" = Cache, Compress, CDN, Lazy-load (the four speed levers); reuse "store it closer, reuse it" (caching/CDN = DNS-caching idea).
- Traps: optimising before measuring (premature optimisation — profile first); caching dynamic/stale-prone content (set TTL / invalidate) or publicly caching private/per-user data in a shared cache/CDN (data leak — use Cache-Control: private, no-store).
- Forward hooks planted: profiling = E-N-S-P dev tools (12-03); CDN blast radius → case_the_cloudflare_2019_outage + SEE 26-01 availability/deployment; bundle size = framework cost (12-05); service worker turns this caching into OFFLINE in 13-04 (MISO). Next: PWA build 13-04.

### SSA-14-01-The-Business-Case-For-Security
- Title: The Business Case for Security

### SSA-15-01-CIA-Triad-And-AAA
- Title: Security Fundamentals — The CIA Triad and AAA
- Terms: the conceptual heart of SSA; outcome SE-12-07 (developing secure code); six foundational properties = CIA triad (what we protect) + AAA (how we protect it). Confidentiality = prevent unauthorised READING/disclosure (controls: encryption at rest + in transit/TLS, access control; threats: breach, eavesdropping, social engineering, insider). Integrity = prevent unauthorised MODIFICATION/tampering (controls: hashing, digital signatures, input validation; threats: tampering, injection, man-in-the-middle, malware). Availability = keep system REACHABLE for authorised users (controls: rate limiting, redundancy/failover, circuit breaker, backups; threats: DoS/DDoS, resource exhaustion, hardware failure, bugs). Authentication = verify WHO you are, comes first (factors: something you Know/Have/Are + somewhere you Are; MFA = combine 2+; passwords stored as salted hash). Authorisation = control WHAT you may do, comes second (role-based access control RBAC + least privilege). Accountability = LOG actions, traceable to a user (audit logging, non-repudiation, forensics, compliance). AAA pipeline = Authenticate → Authorise → Act → Log. Answer-shape for control questions = property, control, effect.
- Mnemonics: "CIA-AAA" said "see-eye-ay, triple-A" = Confidentiality/Integrity/Availability + Authentication/Authorisation/Accountability — DUMP IT IN THE EXAM MARGIN IN SECOND ONE (the marquee exam-dump hook, reuse verbatim everywhere); "CIA = what we protect; AAA = how we protect it"; triad one-word = "secrecy / accuracy / access"; "authentication = who you are; authorisation = what you can do" (reuse from PFW 11-03, now formal); authentication factors = "Know, Have, Are, plus Where"; "authenticate, authorise, act, log" (the AAA pipeline); reuse "Hash is one-way, Encrypt is two-way" (integrity tool).
- Traps: confidentiality vs integrity (reading vs changing — they fail independently); authentication vs authorisation (identity vs permission, who vs what, and the order); "logged in is NOT allowed" (authenticated ≠ authorised → broken access control in 18-02); blurring properties into vague "keeps data safe" instead of naming property + control + effect.

### SSA-15-02-Cryptography-And-Data-Protection-Part-1
- Title: Cryptography and Data Protection — Part 1: Hashing, Encryption and Password Storage
- Terms: outcome SE-12-07; dot-point = contribution of cryptography to 'security by design' (Part 1 = hashing + password storage; Part 2 = symmetric/asymmetric keys, TLS/certs, signatures, sandboxing). Encryption = TWO-WAY, reversible WITH a key, plaintext↔ciphertext → delivers CONFIDENTIALITY (use when you need the data back). Hashing = ONE-WAY, irreversible, produces a fixed-size fingerprint/digest → delivers INTEGRITY + verification (use when you only need to CHECK a match, never recover). Hash properties = one-way, deterministic (same input→same hash), fixed-size, avalanche effect (1-char change→totally different hash). Password storage = the marquee application: NEVER plaintext (breach=instant compromise); NOT encryption (reversible + key is a liability; and you never need to recover a password, only VERIFY it) → so HASH. Plain hashing insufficient → (1) SALT = unique random per-user value combined with pw before hashing → identical pws differ + defeats precomputed RAINBOW TABLES (stored next to hash, not secret); (2) SLOW hashing / key stretching = bcrypt/scrypt/Argon2/PBKDF2-high-iterations → beats brute force (fast hashes like SHA-256 are too fast). Order = password → salt → hash. Login = re-hash entered pw with same salt+algorithm, CONSTANT-TIME compare (timing side-channel → 17-02). Golden rule = never roll your own crypto; use audited libraries (= "secure default, easy right path"). Answer-shape for "role of a salt" = mechanism + effect + named attack defeated.
- Mnemonics: "Hash is one-way, Encrypt is two-way" (the marquee line — reuse verbatim relentlessly); "Salt stops the rainbow" (salting defeats rainbow tables); "Hash it, Salt it, Slow it" (the 3-step password-storage recipe); "password → salt → hash" (the order); hash = a fingerprint (one-way, deterministic). Reuse "secure default, easy right path" (lib makes secure=easy).
- Traps: calling hashing "reversible" or a kind of encryption (it's one-way — note the quiz Q1 mis-keys this; teach it correctly); "encrypting" passwords so they can be decrypted (reversible = key liability); storing unsalted and/or fast hashes; forgetting that you only need to VERIFY, not recover, a password.
- Forward hooks planted: encryption splits into symmetric/asymmetric + TLS/certs + digital signatures (a signature USES hashing → non-repudiation = accountability) + sandboxing → all in 15-02 Part 2 (next); hashing for file/message integrity returns in ch 18 testing/hardening; "use a library not your own" = open-source trust (PFW 12-06). NEXT: 15-02 Cryptography Part 2 (symmetric/asymmetric, keys, TLS, certificates, digital signatures, sandboxing).

### SSA-15-02-Cryptography-And-Data-Protection-Part-2
- Title: Cryptography and Data Protection — Part 2: Keys, TLS, Signatures and Sandboxing
- Terms: outcome SE-12-07; completes dot-point "contribution of cryptography AND sandboxing to security-by-design". Symmetric encryption = ONE shared key both ways, fast/bulk (AES), but key-distribution/sharing problem. Asymmetric = a key PAIR (public shared + private secret, RSA), solves sharing, slow/small-data only; encrypt-with-public/decrypt-with-private. HYBRID encryption = HTTPS move: asymmetric ONCE to agree a fast symmetric SESSION KEY, then symmetric for all bulk traffic (best of both). Key management lifecycle = Generate (secure RNG, never hard-code) / Store (access-controlled, separate from data) / Rotate (limits exposure window + data-per-key) / Destroy. CERTIFICATE = binds a public key to an identity (domain), signed by a Certificate Authority (CA); browser checks CA signature → AUTHENTICATES server + delivers its public key; the certificate does NOT encrypt. TLS handshake (4 steps) = certificate → verify against trusted CAs → asymmetric to agree symmetric session key → symmetric bulk encrypt. Digital signature = sign-with-PRIVATE / verify-with-PUBLIC (the inverse of encryption); mechanically = hash the doc then encrypt the hash with private key; gives authenticity + integrity + NON-REPUDIATION (=accountability) because only the holder's private key could produce a signature the public key verifies → can't deny. Sandboxing = run code in a restricted/isolated environment ("playpen the code can't climb out of") → least privilege + containment, limits blast radius (browser JS sandbox, mobile apps, malware detonation VM).
- Mnemonics: "Symmetric = Same key (fast, sharing problem); Asymmetric = A pair (slow, solves sharing)" (reuse verbatim); "encrypt with public, decrypt with private; but sign with private, verify with public"; "Good Spies Rotate Disguises" = Generate/Store/Rotate/Destroy (key-management lifecycle, coined here); "a sandbox is a playpen the code can't climb out of"; reuse "Hash is one-way, Encrypt is two-way" (a signature hashes-then-signs) + "secure default, easy right path" (use audited library) + CIA-AAA.
- Traps: certificate "encrypts" (WRONG — it AUTHENTICATES + carries the public key; the keys encrypt); confusing signing with encrypting (sign = proof of who + unchanged + non-repudiation, NOT secrecy; a signed doc is still readable); thinking strong crypto can't fail (Heartbleed = sound maths, buggy IMPLEMENTATION); hard-coding keys in source; treating asymmetric as the bulk encryptor (it's slow/small — symmetric does bulk).
- Forward hooks planted: case_heartbleed seeded (2014 OpenSSL memory over-read leaked private keys) → pays off in memory mgmt 17-02 + testing 18-01; HTTPS/TLS is the assumed transport for secure APIs 17-01; "use a library not your own" = open-source trust (PFW 12-06). ### SSA-16-01-Input-Validation-And-Sanitisation
- Title: Defensive Data Input Handling — Validation, Sanitisation and Error Handling
- Terms: outcome SE-12-07; dot-point = "defensive data input handling practices, including input validation, sanitisation and error handling" (marquee — injection = OWASP Top 10 #1). Governing rule = NEVER TRUST INPUT (treat all external data as hostile). VALIDATION = check input against rules, REJECT what doesn't conform (a gate); allow-list (whitelist, define what's permitted, reject rest) vs block-list (blacklist, define what's forbidden, allow rest) — ALLOW-LIST WINS (defining good is finite, defining bad is infinite/evadable via comments/case/encoding). SANITISATION = TRANSFORM/clean/escape input so it's SAFE TO USE (doesn't reject) — e.g. output encoding (HTML-escape "<" → "&lt;"). VALIDATION REJECTS, SANITISATION CLEANS (the key distinction). SQL injection = untrusted input concatenated into a query BECOMES SQL ("' OR '1'='1' --"); FIX = parameterised query / prepared statement = pass the value SEPARATELY from the command (placeholder ?), DB treats it as data not code → injection impossible. XSS = untrusted input echoed into a page BECOMES script; FIX = output encoding/escaping + Content Security Policy (CSP). Injection + XSS = SAME disease (input becoming code) in different interpreters (DB vs browser). ERROR HANDLING = security control: LOG PRIVATE (full detail internally = accountability), SHOW GENERIC (vague message to user, no stack trace/DB structure = no information disclosure), FAIL SECURELY (reject on doubt); login error = "invalid username OR password" (don't reveal which). Client-side validation = usability ONLY, never security (client is bypassable) → ALWAYS re-validate SERVER-SIDE.
- Mnemonics: "VSE" = Validate, Sanitise, handle Errors (the three practices — reuse verbatim, marquee module hook); "allow-list beats block-list" (define what's good, not what's bad); "validation rejects, sanitisation cleans"; "parameterise — pass the value separately so it can never become code"; "log it private, show it generic" (safe errors); "injection and XSS are the same disease — input becoming code". Reuse: never-trust-the-client/validate-server-side (PFW 12-02/12-03), Integrity from CIA-AAA.
- Traps: confusing validation (rejects) with sanitisation (cleans); block-list thinking (evadable, can't list every attack); leaking stack traces / DB structure in error messages (information disclosure); client-side-only validation (bypassable — must be server-side); revealing whether a username exists in a login error.
- Forward hooks planted: XSS + CSRF detailed in 18-02 (input-handling failures, VSE returns); a safe API validates every parameter at the boundary = this lesson applied = 17-01; pays off the unsanitised /search (PFW 13-01) + string-built SQL (PFW 13-02) IOUs. ### SSA-16-02-Privacy-By-Design
- Title: Privacy by Design — Proactive, Embedded, Respectful
- Terms: outcome SE-12-04 (safely/securely collect, use, store data); dot-point = "privacy by design approach" with 3 named principles: proactive not reactive / embed privacy into design / respect for user privacy. ### SSA-17-01-Secure-API-Design
- Title: Secure API Design — Authentication, Authorisation, Rate Limiting and CORS
- Terms: outcome SE-12-07; dot-point = "design, develop and implement a safe API to minimise software vulnerabilities" (+ session management). API = a door opened on purpose to the internet → every request is UNTRUSTED, check before acting. Secure it with A-A-R-T: AUTHENTICATE the caller (API key = identifies the app, simple, no fine-grained perms; JWT = signed stateless token carrying identity+permissions, signed-not-encrypted so NO secrets in payload; OAuth 2.0 = delegated authorisation, "log in with..."); AUTHORISE the specific action on EVERY request (RBAC + ownership check + least privilege) — AUTHENTICATED IS NOT AUTHORISED → forgetting this = BROKEN ACCESS CONTROL / insecure direct object reference (logged-in user changes the ID in the URL and reads others' data); RATE-LIMIT (cap requests/window per key or IP → protects AVAILABILITY vs DoS + slows brute-force/scraping; exceed → 429 too many requests; token bucket/sliding window); TRANSPORT-SECURE (everything over HTTPS/TLS or credentials cross in clear text). Underneath: validate every input param (VSE at the boundary) + fail securely (generic errors). SESSIONS (HTTP is stateless) = server issues a session ID (cookie) after login; secure them = S-A-F-E: Strong random IDs / cookie Attributes (HttpOnly=JS can't read→anti-XSS-theft, Secure=HTTPS only, SameSite=anti-CSRF) / Fixed lifetime (timeout) / re-issue on Escalation (regenerate ID on login → anti session fixation). SAME-ORIGIN POLICY = browser rule: a page's JS can only read responses from its OWN origin by default. CORS (cross-origin resource sharing) = a CONTROLLED RELAXATION of same-origin via response headers (server permits specific other origins) — CORS is NOT access control (only governs which BROWSER origins read responses; non-browser clients ignore it); allow-list origins, never wildcard with credentials.
- Mnemonics: "A-A-R-T" = Authenticate, Authorise, Rate-limit, Transport-secure (the 4 API measures — coined here, reuse); "authenticated is NOT authorised" (broken access control); "S-A-F-E sessions" = Strong IDs / Attributes (HttpOnly/Secure/SameSite) / Fixed lifetime / re-issue on Escalation (coined here); "CORS relaxes the same-origin policy — it is NOT access control"; reuse auth=who/authz=what (CIA-AAA), VSE, "allow-list beats block-list", "secure default easy right path".
- Traps: auth WITHOUT authz (broken access control — the classic API breach); treating CORS as access control; wildcard CORS + credentials; putting secrets in a JWT payload (signed ≠ encrypted); API on plain HTTP; relying on client/login alone.
- Forward hooks planted: broken authentication + session hijacking + session fixation = full treatment in 18-02-Part-2; CSRF (SameSite/CORS relate) = 18-02-Part-1; assumes TLS from 15-02-Part-2; API = the back-end boundary (PFW 13-01) hardened; same-origin previewed PFW 12-03. ### SSA-17-02-Secure-Execution-And-Resource-Management
- Title: Secure Execution and Resource Management — Memory, Sessions, Exceptions, and the Back-Door Attacks
- Terms: outcome SE-12-07; TWO dot-points = "efficient execution: memory management, session management, exception management" + "protect user file and hardware vulnerabilities from file attacks and side channel attacks". Secure execution = M-S-E: MEMORY management (buffer overflow = writing past a fixed buffer corrupts adjacent memory / can run attacker code; over-read = returning more bytes than you should leaks adjacent memory — THIS WAS HEARTBLEED: TLS trusted a client-supplied length without validating it → leaked private keys; high-level/managed langs (Python/Java) bounds-check arrays so classic overflows are rare BUT memory issues DON'T vanish — leaks, exhaustion, over-reads remain; FIX = validate sizes/lengths, never trust an input length, stream large input in chunks, cap sizes); SESSION management (secure lifecycle: strong random ID at login, validate per request, timeout, regenerate ID on login = anti-fixation, invalidate on logout = server-side, not just hidden; failures = session hijacking [steal live ID] + session fixation [plant known ID]); EXCEPTION management (handle errors to FAIL SECURELY: leaking a stack trace = INFORMATION DISCLOSURE; a caught security-check failure that continues = FAIL OPEN/grants access; FIX = log private, show generic, FAIL CLOSED/default-deny). Resource exhaustion = unbounded memory/connections/work → DoS-from-inside; FIX = limits + timeouts everywhere (the resource-level twin of rate limiting). FILE ATTACKS = path/directory traversal: "../" sequences in a user filename climb OUT of the intended folder to sensitive files; FIX = treat filenames as untrusted, canonicalise the real absolute path + verify it stays inside the base dir, allow-list/basename-only. SIDE-CHANNEL ATTACKS = extract secrets by measuring an INDIRECT side effect (timing, power, sound, EM emissions, cache) NOT the algorithm/front door; classic = TIMING ATTACK on a char-by-char comparison that exits early (response time reveals correct leading chars → recover secret incrementally); FIX = CONSTANT-TIME COMPARISON (same time regardless of match) — this is the attack the constant-time compare from 15-02-Part-1 defends.
- Mnemonics: "M-S-E" = Memory, Session, Exception management (the 3 execution areas — coined here, reuse); "Heartbleed was a memory over-read" (trusted an unvalidated length); "../ climbs out" = path traversal; "the secret leaks through timing, not the front door — compare in constant time" (side channel); reuse VSE / "log private, show generic" / "fail securely" / "allow-list beats block-list" / S-A-F-E sessions (17-01).
- Traps: assuming managed/high-level languages are immune to ALL memory attacks (over-reads/leaks/exhaustion remain); showing error messages/stack traces to users; trusting filenames/paths from users; ignoring timing leaks (use constant-time compare); no resource limits/timeouts.
- Forward hooks planted: case_heartbleed PAID OFF here (seeded 15-02-Part-2); session hijacking + session fixation + broken auth + race conditions + file & side-channel attacks = full vulnerability treatment in 18-02 (Part-2 esp.); exception handling echoes Year 11 error handling. ### SSA-18-01-Security-Testing-Fundamentals
- Title: Security Testing Fundamentals — Finding the Weaknesses and Surviving the Breach
- Terms: outcome SE-12-06 (justify/select tools); TWO dot-points = the 5 testing strategies (code review, SAST, DAST, vulnerability assessment, penetration testing) + survival (determine vulnerabilities, harden, handle breaches, business continuity, disaster recovery). FIVE METHODS (cheap+broad → expensive+deep): CODE REVIEW = human reads source line-by-line, uniquely catches FLAWED LOGIC/INTENT no tool sees (caught the xz backdoor); SAST (Static Application Security Testing) = analyses SOURCE without running it, finds insecure patterns EARLY + by exact line + all paths, BUT false positives + can't see runtime; DAST (Dynamic Application Security Testing) = attacks the RUNNING app from outside (simulated attacks), finds REAL exploitable runtime+config flaws, BUT late + needs running app + no line number; VULNERABILITY ASSESSMENT = broad scan cataloguing KNOWN weaknesses (unpatched libs, misconfig) = breadth not depth (would've caught Equifax's missing Struts patch); PENETRATION TEST = skilled human actively EXPLOITS + CHAINS weaknesses to prove real impact = depth not breadth. SURVIVING A BREACH: HARDEN (shrink attack surface BEFORE: patch, remove unused services, least privilege, change defaults, disable verbose errors); BREACH HANDLING/incident response (detect→contain→eradicate→recover→review; PLAN IT IN ADVANCE); BUSINESS CONTINUITY = keep the WHOLE business operating through a disruption (broad goal); DISASTER RECOVERY = the SUBSET restoring IT systems/data after (backups off-site+tested+regular, recovery objectives, redundancy/failover). black-box≈DAST, white-box≈SAST (Year 11 echo).

### SSA-18-02-Web-Application-Vulnerabilities-Part-1
- Title: Web Application Vulnerabilities Part 1 — XSS, CSRF and Invalid Redirects
- Terms: outcome SE-12-07; dot-point = "minimise vulnerabilities in user action controls" (split). Master list of user-action vulns = B-X-I-R = Broken auth/session, Cross-site (XSS/CSRF), Invalid forwarding/redirect, Race conditions. Part 1 = X + I. CROSS-SITE SCRIPTING (XSS) = attacker injects malicious SCRIPT that EXECUTES in another user's browser; root cause = untrusted input echoed into a page WITHOUT ENCODING → browser runs it as script (same disease as SQLi — input becoming code; THIS is the unsanitised PFW /search route, now named); types = Reflected (script in request, reflected back, needs a click), Stored/persistent (saved server-side, hits every viewer = worst), DOM-based (client-side JS); impact = run any JS as victim (steal session cookie, hijack account) = integrity+confidentiality break; FIX = output encoding/escaping (primary) + Content Security Policy (CSP) + allow-list validation + HttpOnly cookies. CROSS-SITE REQUEST FORGERY (CSRF, "see-surf") = NO script injected; tricks a LOGGED-IN user's browser into sending a FORGED unintended request to a site where they're authenticated, exploiting that the browser auto-attaches the session cookie ("forging a signature on a cheque"); attacker never sees password/cookie — just rides the existing session; FIX = anti-CSRF token (unpredictable, server-embedded+validated; attacker's cross-origin page can't read it due to same-origin policy → forged request rejected) + SameSite cookies + re-auth for sensitive actions. KEY DISTINCTION: XSS injects a SCRIPT (run code as you); CSRF forges a REQUEST (act as you, riding your session). INVALID FORWARDING/REDIRECTING (open redirect) = app redirects to an UNVALIDATED user-supplied URL → attacker crafts link starting at trusted domain that springboards victim to a malicious/phishing site (abuses brand trust); FIX = never redirect to a raw user URL — allow-list internal destinations / relative paths only.
- Mnemonics: "XSS injects a Script; CSRF rides a Session" (the key distinction — reuse verbatim); "B-X-I-R" = Broken-auth, Cross-site, Invalid-redirect, Race-conditions (user-action vuln master list — reuse, completed in Part 2); reuse "injection and XSS are the same disease — input becoming code", output encoding (16-01), SameSite/S-A-F-E (17-01), "allow-list beats block-list".
- Traps: confusing XSS with CSRF (inject-script vs forge-request); thinking "logged in" stops CSRF (logged-in is the very thing CSRF ABUSES); treating open redirects as harmless (real phishing enabler); relying on input validation alone for XSS (output encoding is the primary fix).
- Forward hooks planted: Part 2 completes B-X-I-R = broken authentication + session management (hijacking/fixation from 17-01/17-02), race conditions, file & side-channel attacks; these are exactly the bugs DAST/pen-test from 18-01 hunt. ### SSA-18-02-Web-Application-Vulnerabilities-Part-2
- Title: Web Application Vulnerabilities Part 2 — Broken Auth, Race Conditions, File and Side-Channel Attacks
- Terms: outcome SE-12-07; completes B-X-I-R (B + R) + the bundled file & side-channel attacks. BROKEN AUTHENTICATION & SESSION MANAGEMENT = weaknesses in proving identity + maintaining the session: weak passwords / credential stuffing / no login rate-limit (brute force); SESSION HIJACKING = steal a victim's LIVE valid session ID (via XSS, sniffing, predictable ID) + reuse it; SESSION FIXATION = plant a KNOWN session ID before the victim logs in, then ride it; IDs in URLs, no timeout, no logout-invalidation; FIX = the S-A-F-E rules as defences (Strong random IDs, HttpOnly/Secure/SameSite cookies, Fixed timeout, REGENERATE ID on login = kills fixation) + MFA + rate-limited logins + hashed-salted-slowed passwords. RACE CONDITION = correctness depends on TIMING of concurrent ops; classic = TOCTOU (Time-Of-Check to Time-Of-Use) — gap between a check and the dependent action; example = DOUBLE-SPEND (two simultaneous withdrawals both pass the balance check before either deducts → overdrawn); TRAP = "my code is single-threaded so no races" is WRONG (the web SERVER is concurrent — many requests hit the same DB); FIX = make check+act ATOMIC (DB transaction + locking, or a single atomic conditional update "subtract only if balance>=amount"). FILE ATTACKS = path/directory traversal ("../ climbs out" of the intended dir) + unsafe uploads (script with fake extension executed/served); FIX = canonicalise + verify path stays inside base dir, allow-list/basename-only, validate upload content-type not extension, rename + store non-executable. SIDE-CHANNEL ATTACKS = extract secrets via INDIRECT side effect (timing/power/EM/sound/cache) not the algorithm; classic = TIMING ATTACK on early-exit comparison (recover secret char-by-char); FIX = CONSTANT-TIME COMPARISON.
- Mnemonics: "B-X-I-R" COMPLETE (Broken-auth, Cross-site, Invalid-redirect, Race — reuse verbatim); "TOCTOU" = Time-Of-Check to Time-Of-Use (race conditions); "hijacking steals a live session; fixation plants one"; "the server is always concurrent" (never assume single-threaded); reuse S-A-F-E sessions (17-01), "../ climbs out" + "secret leaks through timing — constant-time compare" (17-02).
- Traps: confusing session hijacking (steal live) vs fixation (plant known); assuming single-threaded = no race (server is concurrent); "check the balance before withdrawing" IS the race bug (the gap is the problem — make it atomic); trusting filenames (path traversal); ignoring timing leaks.
- Forward hooks planted: these named vulns = what an SEE project must test for + what disciplined collaboration catches → Ch 19 (19-01 teams); xz backdoor's maintainer-trust lesson lands in 19-01; race conditions/concurrency return in Software Automation. ### SA-20-02-ML-Training-Models
- Title: Models of Training Machine Learning — Supervised, Unsupervised, Semi-Supervised, Reinforcement
- Terms: outcome SE-12-03; dot-point = explore models of training ML (supervised / unsupervised / semi-supervised / reinforcement). Each defined by the DATA it needs. SUPERVISED = LABELLED data (input + known correct output pairs) → predict a known answer; splits into classification (category) + regression (continuous number, → 21-01); learn-with-an-answer-key (spam filter, diagnosis, price); "only as good as its labels" → human/historical bias enters here (Amazon recruiting AI story). UNSUPERVISED = NO labels → discover hidden structure / groupings itself (clustering = customer segmentation; anomaly detection of "what's normal"). SEMI-SUPERVISED = a FEW (expensive) labels + a LARGE unlabelled pool, used together (medical imaging: 1k labelled + 50k unlabelled) — exists because labelling is costly. REINFORCEMENT = no fixed dataset; an AGENT acts in an ENVIRONMENT + learns by trial-and-error from a REWARD/penalty signal (games, robotics, self-driving); sequential decisions, delayed reward. DECISION FRAMEWORK = ask: do you have labels? how many? or is it an environment with rewards? Always JUSTIFY by the data (data situation → model → effect).
- Mnemonics: "SUSR" said "suss-er" = Supervised / Unsupervised / Semi-supervised / Reinforcement (the four training models — coined here, reuse verbatim); trigger words = "labelled / find-structure / few-labels / reward"; reuse "garbage labels in, bias out" (supervised learns its labels' bias).
- Traps: supervised vs unsupervised (the ONLY question is: do you have labels?); reinforcement is NOT supervised (a reward AFTER acting ≠ a labelled answer key — reward says how good, never what the right action was); recommending supervised when labelling is too expensive (→ semi-supervised).
- Forward hooks planted: supervised underpins regression (21-01) + the judgement ML adds to RPA/BPA (20-01); Amazon recruiting AI story → case_the_amazon_recruiting_ai + bias 22-03 (label/historical bias); reinforcement's act→feedback loop = Year 11 mechatronics control loop (sense-act-adjust); training on personal data → PER + "collect less keep less" (SSA 16-02). ### SA-21-01-Regression-And-Core-Algorithm-Types
- Title: Regression and the Core Algorithm Types — Linear, Logistic, Polynomial and K-Nearest Neighbour
- Terms: outcome SE-12-02; dot-points = develop ML regression models with OOP to predict numeric values (linear / polynomial / logistic) + types of ML algorithms (linear regression / logistic regression / K-nearest neighbour). LINEAR regression = predict a CONTINUOUS NUMBER by fitting the best straight LINE (output = slope*x + intercept; slope/intercept = parameters/weights; house size→price). LOGISTIC regression = DESPITE THE NAME it is CLASSIFICATION — predicts a CATEGORY (yes/no) via the SIGMOID (squashes any number to 0–1 = probability, threshold at 0.5); sigmoid = the neuron's activation (20-04); spam-or-not. POLYNOMIAL regression = fits a CURVE (bends where a line can't) — and is HOW YOU OVERFIT. K-NEAREST NEIGHBOUR = predict for a new point by finding the K most SIMILAR training examples + majority-vote/average (no line/curve, just comparison). OOP: models = fit/predict CLASSES (fit LEARNS the parameters from data; predict APPLIES them) — same as scikit-learn's .fit/.predict, reuses Year 11 OOP + PFW ORM. OVERFITTING = model fits training data TOO closely (memorises NOISE) → great on training, fails on NEW data (poor generalisation; underfitting = too simple); reduce by SIMPLER model (lower-degree poly / limit tree depth) + more data + regularisation. TRAIN-TEST SPLIT = NEVER evaluate on training data; train on one set, test on a separate UNSEEN set; a big train>>test gap = overfitting. Answer shape (distinguish) = output type + mechanism + example.
- Mnemonics: "L-L-K" = Linear regression, Logistic regression, K-nearest neighbour (core algorithm types — reuse verbatim; "+ polynomial" for the regression-building dot-point); "Linear predicts a number; Logistic predicts a yes/no" (the marquee distinction — reuse verbatim); "fit learns, predict applies"; "train on one set, test on a separate unseen set".
- Traps: logistic "regression" is CLASSIFICATION not number-prediction (the name misleads — output type decides: number=linear, yes/no=logistic); fitting to NOISE = overfitting (a perfect training fit is a RED FLAG); evaluating on TRAINING data tells you nothing (must use a held-out test set); "linear is a line, logistic is a curve" (WRONG — polynomial is the curve).
- Forward hooks planted: case_zillow_offers PAID OFF (overfitted/overconfident price model + changed market → ~$500M loss → validate on held-out data + human oversight); logistic regression ≈ ONE neuron (sigmoid activation) → bridges to neural networks 21-02 (stacked weighted-sum units); overfitting = deep-tree-memorising generalised (20-04); explainability + overfitting → 22-03 bias/transparency; forecasting = supervised regression (20-03 F-A-I, SUSR supervised). ### SA-21-02-Neural-Networks-Concepts-And-Implementation
- Title: Neural Networks — Concepts and Implementation
- Terms: outcome SE-12-02; dot-point = apply neural network models using OOP to make predictions (verb = APPLY → describe + trace + explain, not derive calculus). PERCEPTRON = the simplest network = a SINGLE NEURON: inputs × WEIGHTS, summed + a BIAS = weighted sum, through an ACTIVATION → output (learns logic gates AND/OR; weights = learned params, bias = adjustable threshold); ≈ logistic regression (one neuron + sigmoid). FORWARD PASS = run inputs through to a prediction (multiply, sum, activate; for multilayer, feed each layer's output forward) — this is what "trace" questions test. STRUCTURE (the "describe the structure" Q) = INPUT layer (one neuron/feature) → one+ HIDDEN layers (each neuron does weighted-sum+activation, passes forward) → OUTPUT layer; neurons joined by WEIGHTED links; "layers learn features" (early = edges, later = faces). TRAINING = P-L-A loop: PREDICT (forward pass) → LOSS (how far from the known label) → ADJUST (nudge every weight to reduce error), repeated over many EPOCHS; technically = BACKPROPAGATION (push error back to find each weight's contribution) + GRADIENT DESCENT (step downhill on error) — name for polish, but the loop is the substance; same optimisation as fitting regression, scaled to many weights. Big network can OVERFIT like a high-degree polynomial → still use train-test split. VOCAB precision: neuron (unit) / layer (group) / weight (number on a connection) / bias (per-neuron threshold).
- Mnemonics: "P-L-A" = Predict, Loss, Adjust (the training loop, repeated — coined here, reuse verbatim); "a perceptron = weighted sum + bias, then activation"; "layers learn features" (reuse from 20-04); reuse "a Net is weighted neurons" (20-04), "fit learns, predict applies" + train-test split + overfitting (21-01).
- Traps: hand-waving "it just learns the patterns" (band-4 — must give the predict→loss→adjust loop / forward pass + backprop); muddling neuron vs layer vs weight vs bias; "bigger network is better" (more capacity = more OVERFITTING, like high-degree polynomial → validate on unseen data).
- Forward hooks planted: explainability/black-box trade-off (a powerful net you can't read) → 22-03 bias + transparency + accountability (CIA-AAA); P-L-A = the gradient-descent fitting from 21-01 scaled up (logistic ≈ one neuron); perceptron/network = fit/predict OOP classes (Year 11 OOP). ### SA-22-03-Bias-In-Datasets-And-Models
- Title: Bias in Datasets and Models
- Terms: outcome SE-12-05; dot-point = INVESTIGATE the effect of HUMAN and DATASET SOURCE bias in ML/AI development. DEFINITION = bias = a SYSTEMATIC error producing unfair/skewed outcomes for particular groups (NOT random noise). MYTH TO KILL = "the algorithm is neutral/objective" — WRONG: the maths is impartial but it faithfully learns the data's patterns, so it MIRRORS and AMPLIFIES the biases in human-generated data (even when the sensitive attribute is NOT an input). FOUR SOURCES: SAMPLING = some groups missing/under-represented in training data (face recognition on light-skinned faces; tell = "who is missing?"; cousin = POPULATION DRIFT, population shifts after training → monitor+retrain). HISTORICAL = data is a FAITHFUL record of an UNJUST past, so the model perpetuates past discrimination (hiring model on male-dominated history favours men; tell = "true record of an unfair past?"; KEY: more data does NOT fix it). LABELER = humans' conscious/unconscious bias baked into the LABELS (ground truth) in supervised learning (annotators flag a dialect as "toxic"; defence = guidelines + multiple labelers + agreement checks). MEASUREMENT = a flawed/unequal PROXY distorts the data before learning (arrests as a proxy for crime → over-policed areas; tell = "did we measure what we care about, equally for all?"). MITIGATION = D-R-F: Diverse data (representative/stratified at the source → sampling), Reweight (in training, fairness constraints → can't collect more), Fairness metrics (AFTER, measure PER-GROUP not overall accuracy — demographic parity = similar positive rates; equalised odds = similar error rates; HIGH overall accuracy can still be UNFAIR to a minority). ACCOUNTABILITY (band-6 lift) = MODEL CARDS (disclose training data + per-group performance + limitations) + REPRODUCIBILITY/DATA PROVENANCE (audit trail) = the accountability "A" of CIA-AAA applied to AI; explainability (interpretable tree 20-04) is a precondition for accountability. Answer shape = named source + not-neutral mechanism + example + mitigation (+ accountability for the extended Q).
- Mnemonics: "S-H-L-M" said "SHaLuM" = Sampling, Historical, Labeler, Measurement (the 4 bias sources — coined here, reuse verbatim; "a bad SaMpLe of the world"); "D-R-F" = Diverse data, Reweight, Fairness metrics (the 3 mitigations — coined here, reuse; "Don't Reinforce unFairness"); "the algorithm isn't neutral — it mirrors and amplifies its data"; "model cards + provenance" (accountability); reuse "garbage labels in, bias out" (20-01/20-02 → labeler/historical bias made precise).

### SEE-23-03-Implementation-Methods
- Title: Implementation Methods
- Terms: outcome SE-12-06; dot-point = software implementation methods (direct/phased/parallel/pilot) = the CHANGEOVER/CUT-OVER strategy = the RISKIEST moment of a project. DIRECT ("big bang") = switch off old + on new in ONE changeover, all users at once → cheap+FAST but HIGH-RISK, NO fallback. PHASED = roll out the new system in STAGES BY FUNCTIONALITY (a feature at a time, to everyone), learn between phases → MEDIUM-risk, slower. PARALLEL = run old AND new SIMULTANEOUSLY (both process same data) → LOW-risk (live safety net + compare/verify outputs) but EXPENSIVE (two systems). PILOT = deploy the WHOLE system to a small representative USER GROUP first → LOW-risk (issues contained, real-world feedback) but delays full rollout. Risk order (riskiest→safest) = Direct > Phased > Parallel > Pilot. KEY DISTINCTIONS: Direct cheap/risky vs Parallel safe/costly (the exam pairing); PHASED splits by FEATURE (part of system to everyone) vs PILOT splits by USER (whole system to part of users) vs PARALLEL runs both at once. Recommend-and-justify skill = match risk profile to scenario stakes + weigh trade-off (hospital/bank = parallel+pilot; startup app = direct).

### SEE-24-01-Waterfall-Approach
- Title: The Waterfall Approach
- Terms: outcome SE-12-06; opens Ch 24 (research+planning); dot-point = research+use Waterfall (logical progression of steps; stages of 'falling water'; advantages/disadvantages; scale+types of developments). DEVELOPMENT APPROACH = overall model for how the whole project runs its lifecycle. WATERFALL = LINEAR/SEQUENTIAL: complete one phase fully + formal sign-off, then the next; water FALLS ONCE (going back up is expensive). 6 phases (= the SDLC run strictly linearly) = Requirements analysis → Design → Implementation (coding) → Testing → Deployment → Maintenance; heavy DOCUMENTATION + formal phase-gate reviews at each gate (Requirements Spec Doc, System Architecture Doc, test plans) = SE-12-09 manage+document. ADVANTAGES = predictability/clear structure + comprehensive documentation/audit trail + quality via formal reviews (reduces scope creep). DISADVANTAGES = rigid to change (late change = expensive, can't flow back) + LATE feedback + LATE integration + LATE testing + all-or-nothing delivery. SUITS = stable/well-understood requirements, REGULATED/compliance projects, mature tech, risk-averse (e.g. gov compliance, medical-device sw). POOR FIT = uncertain/evolving requirements, fast-moving market, needs early user feedback (e.g. startup app).
- Mnemonics: "Waterfall Falls once" (no cheap going back up — marquee, reuse verbatim, pairs with "Agile Goes around"); stages reuse the SDLC hook "Really Smart Developers Don't Ignore Testing In Maintenance" (SSA 14-02); advantage/disadvantage answer-shape = FEATURE + CONSEQUENCE.
- Traps: "waterfall is always bad/outdated" (it's RIGHT for stable + regulated — must give the balanced view); "explain advantage/disadvantage" with no mechanism (vague = band 4 — give feature+consequence); ignoring its strength for fixed/documented/regulated scope.
- Forward hooks planted: case_the_healthcare_gov_launch PAID OFF as illustration (waterfall + FIXED deadline + frozen-flawed requirements + late integration + NO load testing → launch meltdown → full cash-in 26-01 testing/load + 26-03 evaluation/post-mortem); direct contrast set up for Agile (24-02, "Goes around") + combined in WAgile (24-03); phases = SDLC (Year 11 + SSA 14-02); Requirements phase = FUN-vs-PERF (23-01). ### SEE-24-02-Agile-Approach
- Title: The Agile Approach
- Terms: outcome SE-12-06; dot-point = research+use Agile (rate of developing a final solution; method tailoring; iteration workflow; scale+types). AGILE = ITERATIVE + INCREMENTAL: do a bit of all phases repeatedly in short cycles, each producing WORKING/potentially-shippable software → rate of developing = continuous/frequent (not one big drop). ITERATION = SPRINT = TIME-BOXED (fixed length, ~2 weeks, can't extend; unfinished work goes back to backlog). PRODUCT BACKLOG = prioritised list of all to-build; SPRINT BACKLOG = the slice for this sprint; VELOCITY = how much reliably done per sprint (forecast). USER STORY format = "As a [user], I want [functionality], so that [benefit]" + ACCEPTANCE CRITERIA (testable conditions = requirements' measurable idea per story); INVEST = good-story check. CEREMONIES (P-S-R-R) = Plan (sprint planning: select+estimate stories, story points Fibonacci 1/2/3/5/8/13) / Standup (daily 15-min: did/will/blockers — surface not solve) / Review (demo to stakeholders + feedback = the loop Waterfall lacks; PRODUCT) / Retro (improve the TEAM PROCESS). METHOD TAILORING = adapt practices/frameworks (Scrum w/ Product Owner+Scrum Master; Kanban = continuous flow) to context, not one rigid recipe. ADVANTAGES = adaptable to change, early+continuous feedback, frequent delivery. DISADVANTAGES = less predictable scope/cost, lighter docs (bad for regulated), needs high client involvement. - Mnemonics: "Agile Goes around" (iterative loop — marquee, reuse verbatim, pairs with "Waterfall Falls once"); "P-S-R-R" = Plan, Standup, Review, Retro (the 4 ceremonies — coined here, reuse; Review = product, Retro = process); "As a [user], I want [x], so that [benefit]" (user-story format); reuse acceptance criteria (23-01), DevOps/CI-CD (SA 20-01 = the engine under frequent delivery), shift-left (security in definition-of-done).
- Traps: "Agile = no planning/no documentation" (FALSE — plans EVERY sprint, documents lightly+continuously; "over" ≠ "instead of"); cargo-cult Agile (ceremonies without the adaptive mindset); swapping Review (product) with Retro (process); "compare" answers that just say "old vs new/fast vs slow" (need named contrasts).
- Forward hooks planted: enabled by DevOps/CI-CD (SA 20-01); WAgile combines Agile sprints + Waterfall gates (24-03, next); sprint review feedback loop → feedback analysis (26-02); user stories+acceptance criteria → test plan (26-01). ### SEE-24-03-WAgile-Hybrid-Approach
- Title: The WAgile Hybrid Approach
- Terms: outcome SE-12-06; dot-point = research WAgile (it's a HYBRID; analysis of the 'WHEN' and 'HOW' intervention is applied during the lifecycle; scale+types). ### SEE-24-04-Project-Management-Tools-And-Gantt
- Title: Project Management Tools and Gantt Charts
- Terms: outcome SE-12-09 (the manage-and-document CORE of the module); dot-point = apply project management (scheduling+tracking using a software tool incl. Gantt charts; using collaboration tools). PROJECT MANAGEMENT TRIANGLE = Scope / Time / Cost (interdependent — change one, the others move; QUALITY in the middle; "pick two"; = TECO schedule+economic corners from 23-01). GANTT CHART = visual schedule: TASKS as horizontal BARS against a TIMELINE (left-right); bar length = duration; DEPENDENCIES = arrows (finish-to-start = B can't start till A finishes = default); MILESTONES = checkpoints/diamonds (WAgile stage-gates = milestones); shaded bars = progress. TRACKING = ongoing plan-vs-actual (schedule variance), alert on critical-path slips. COLLABORATION TOOLS = MS Project/Primavera P6 (large/complex), Jira (Agile teams), Trello (Kanban/small team), + VERSION CONTROL (fork/branch/pull-request = a PR is a review tool; PFW 12-06 + Year 11) — select+justify per project.

- Traps: a Gantt with NO dependencies (can't find critical path = useless); ignoring the critical path (treating all delays as equal); on a slip question, NOT checking whether the slipped task is ON the critical path (within slack = no effect; on it = whole project slips).
- Forward hooks planted: tracking data → feedback (26-02) + evaluation (26-03, "did we hit schedule"); collaboration tools = version control → deep dive 25-04; triangle = the thing you NEGOTIATE around (24-05, next); WAgile gates = Gantt milestones (24-03). ### SEE-24-06-Quality-Assurance
- Title: Quality Assurance
- Terms: outcome SE-12-06; dot-point = how solutions are quality assured (defining CRITERIA on which quality is judged; ensuring requirements met via a CONTINUAL CHECKING process; addressing COMPLIANCE+legislative requirements).
