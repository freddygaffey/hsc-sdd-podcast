# Podcast Plan — Programming for the Web (PFW)

> **What this file is.** The episode-by-episode plan an AI agent uses to write each
> script for this module. It is the bridge between the syllabus, the textbook, and
> `STYLE.md`. Read `../../STYLE.md` and `../../SUPPLEMENTARY.md` **before**
> writing any script — this file gives you the *seeds*; STYLE.md gives you the *form*.
>
> **How to use an episode entry.** Each entry below is a brief, not a script. It tells
> you: the exact filename, the syllabus dot-points and outcomes to hit, the one home
> topic to teach, what to recap (spaced repetition), what to interleave, the mnemonics
> to coin/reuse, worked-example and exam-question seeds, appendix listings, and the
> traps. Expand it into a full ~30–45 min script that satisfies the STYLE.md §9
> checklist. Do **not** merge two episodes; do **not** teach two home topics in one.

### Naming & layout (read once)

Each episode is a **folder** under `content/`, named with the episode key from
STYLE.md §8 (`MODULE-LL-Topic-Title`) — the **module acronym is in the folder name**,
e.g. `content/PFW-01-Applications-Of-Web-Programming/`. Inside it you write two files,
`script.md` and `supplementary.md` (the audio `<voice>.m4a` files are generated by the
pipeline — see `AUTHORING.md`). Module-review episodes: `PFW-99-Module-Review-…`.
Case studies: `case_…` (no number).

The textbook section (e.g. `11.1`) is recorded in the frontmatter `lesson:` field; the
folder name uses the zero-padded teaching-order `LL`, so syllabus reference and sort
order stay decoupled. **`AUTHORING.md` is the authoritative recipe — follow it.**

### Module-wide mnemonics (coin once, reuse in every recap/review)

- **OSI-ish layering for protocols** → teach the **"Please Do Not Throw Sausage Pizza Away"** only if you go down to layers; otherwise prefer the simpler grouping below.
- **Protocol/port pairs** → **"80 plain, 443 safe; 22 secure shell, 21 file"** chant (HTTP 80 / HTTPS 443 / SSH-SFTP 22 / FTP 21). Reuse verbatim.
- **The request journey** → **"DNS → TCP → TLS → HTTP"** = *"Dogs Take Tasty Hambones"* (resolve name, open connection, secure it, send request). Use in 11-02, 11-03, 13-01.
- **CRUD ↔ SQL/HTTP** → **C-R-U-D = INSERT/SELECT/UPDATE/DELETE = POST/GET/PUT/DELETE**. Reuse in 13-01 and 13-02.
- **Web dev tiers** → **"Front, Back, Store"** (client-side, server-side, database). Reuse in 12-02, 13-01.

## Episode briefs

### 11-01 — Applications of web programming
- **Source:** Ch 11 §11.1. **Outcomes:** SE-12-03.
- **Dot-points:** *Explore the applications of web programming, including interactive websites/webpages, e-commerce, progressive web apps (PWAs).*
- **Teach (home topic):** what kinds of things "web programming" actually builds, and the trade-offs that distinguish them (interactivity, payments/trust, offline/installable).
- **§5.1 recap:** This is episode 1 of the year — keep the opener to ~2 min: frame the four Year 12 modules and how PFW underpins the rest. No prior episodes to drill.
- **Interleave (≥2):** forward → e-commerce *"data protection and minimising attacks — we hit this hard in Secure Software Architecture"* (prelude it); forward → PWAs return in **13-04**; link → "interactive" depends on the client/server split taught in **12-02**.
- **Mnemonics:** application types **"I-E-P"** = *Interactive, E-commerce, PWA*.
- **Worked example seed:** classify three real sites (a news site, Amazon, Twitter/X-as-PWA) by type and justify; model the mark-earning sentence.
- **Exam seeds:** "Outline two characteristics of a progressive web app." / "Compare an interactive website with an e-commerce platform in terms of security requirements."
- **Appendix:** Listing 1 — a tiny static-vs-interactive HTML snippet (narrate the difference). No pseudocode needed.
- **Traps:** calling every dynamic site a PWA; PWA = installable + offline-capable (service worker), not just "mobile-friendly".

### 11-02 — How data moves on the internet
- **Source:** Ch 11 §11.2. **Outcomes:** SE-12-03.
- **Dot-points:** *Investigate and practise how data is transferred — data packets; IP addresses incl. IPv4; DNS.* (Protocols introduced; secure transport deepened in 11-03.)
- **Teach:** the journey of one request — packetisation, IP addressing/IPv4, and DNS name resolution.
- **§5.1 recap:** 11-01 application types ("I-E-P"); note every one of those depends on this plumbing.
- **Interleave:** forward → TLS/HTTPS in **11-03**; forward → DNS single-point-of-failure pays off in `case_the_dyn_dns_ddos_2016`; link → big-data routing/CDNs in **11-04**.
- **Mnemonics:** **"Dogs Take Tasty Hambones"** = DNS → TCP → TLS → HTTP (coin it here, reuse). IPv4 = "four numbers, 0–255, dots between".
- **Worked example seed:** trace `https://shop.example.com/cart` from typing it to bytes returning — name each step and protocol. This is the spine of a 4–6 mark question.
- **Exam seeds:** "Describe the role of DNS in retrieving a web page." (3) / "Explain why data is broken into packets." (4)
- **Appendix:** Listing 1 — `dig`/`nslookup` output annotated; Listing 2 — NESA pseudocode for a naive packet-reassembly loop (FOR each packet … order by sequence number).
- **Traps:** confusing IP address with DNS name; thinking packets arrive in order.

### 11-03 — Web protocols, ports and secure transport
- **Source:** Ch 11 §11.3. **Outcomes:** SE-12-04 (with SE-12-03).
- **Dot-points:** *Function of web protocols and their ports — HTTP/HTTPS, TCP/IP, DNS, FTP/SFTP, SSL/TLS, SMTP/POP3/IMAP.* *Processes for securing the web — SSL certificates, encryption algorithms, encryption keys, plain/cipher text, authentication & authorisation, hash values, digital signatures.*
- **Teach:** which protocol runs on which port, and how TLS turns plain text into cipher text using certificates and keys.
- **§5.1 recap:** "Dogs Take Tasty Hambones" (11-02); IPv4; I-E-P.
- **Interleave (heavy):** forward → **all of this is taught properly in Secure Software Architecture** — give a prelude and a forward pointer (CIA triad, hashing vs encryption, digital signatures = SSA 15-01/15-02); link back → DNS/TCP from 11-02; contrast → "hash value ≠ encryption" (set up the SSA trap now).
- **Mnemonics:** **"80 plain, 443 safe; 22 secure shell, 21 file"** (coin here). Symmetric vs asymmetric → **"Symmetric = Same key; Asymmetric = A pair"**.
- **Worked example seed:** walk a TLS handshake at "exam depth" — certificate presented, identity verified by a CA, session key agreed, traffic encrypted. Model a 5-mark "Explain how HTTPS secures data in transit" answer.
- **Exam seeds:** "Distinguish between authentication and authorisation." / "Outline the purpose of a digital signature." / port-matching objective items.
- **Appendix:** Listing 1 — protocol/port table; Listing 2 — symmetric vs asymmetric encrypt/decrypt in Python (narrate, don't read); Listing 3 — NESA pseudocode for a signature-verify check.
- **Traps:** "SSL" vs "TLS" (TLS is the modern one; say "ess-ess-ell" / "tee-ell-ess"); thinking a certificate *encrypts* (it *authenticates* + carries the public key).

### 11-04 — Big data and web architectures
- **Source:** Ch 11 §11.4. **Outcomes:** SE-12-03.
- **Dot-points:** *Effect of big data on web architecture — data mining, metadata, streaming service management.*
- **Teach:** how scale (big data + streaming) reshapes architecture — monolith vs microservices vs serverless, CDNs, where data mining/metadata fit.
- **§5.1 recap:** ports/TLS (11-03); the request journey; I-E-P.
- **Interleave:** forward → performance/caching in **13-03**; forward → bias & dataset provenance in **Software Automation 22-03** ("big data is where ML gets its training data — and its bias"); link → availability/blast radius set up `case_the_cloudflare_2019_outage`.
- **Mnemonics:** the three big-data terms → **"M-M-S"** = *Mining, Metadata, Streaming*.
- **Worked example seed:** Netflix-style streaming — why a CDN and microservices instead of one big server; what metadata drives recommendations.
- **Exam seeds:** "Explain how big data influences the architecture of a web application." (6, extended) / "Define metadata and give a web example."
- **Appendix:** Listing 1 — a metadata JSON record; no heavy code.
- **Traps:** "big data = lots of rows" only — it's volume/velocity/variety; conflating data mining with data storage.

### 13-02 — Databases: SQL and ORM
- **Source:** Ch 13 §13.2. **Outcomes:** SE-12-02.
- **Dot-points:** *Apply a web-based database and construct SQL — selecting fields, GROUP BY, common queries, WHERE constraints, table joins; parameterised queries.* *Compare ORM to SQL.*
- **Teach:** writing SQL that earns marks (SELECT/WHERE/GROUP BY/JOIN) and the ORM trade-off.
- **§5.1 recap:** R-H-D-R (13-01); Front/Back/Store; CRUD mapping.
- **Interleave (heavy):** forward → **parameterised queries prevent SQL injection — taught fully in SSA 16-01** (prelude + strong forward-ref, this is the marquee cross-link); link → ORM maps classes↔tables, reusing OOP from Year 11 (backward); link → joins model the relationships from data dictionaries (Year 11 PF11 03-03).
- **Mnemonics:** clause order → **"SELECT FROM WHERE GROUP-BY HAVING ORDER-BY"** = *"So Few Workers Go Home On-time"*. Join types → "INNER = both, LEFT = keep left".
- **Worked example seed:** build a `GROUP BY` aggregate with a join across two tables; then show the *string-concatenated* (vulnerable) query vs the parameterised one — model the SSA forward-link sentence.
- **Exam seeds:** "Write an SQL statement that … joins … and groups …" (code-editor item) / "Compare ORM with raw SQL." (4)
- **Appendix:** Listing 1 — schema + SELECT/JOIN/GROUP BY SQL; Listing 2 — vulnerable vs parameterised query (Python); Listing 3 — same query via an ORM; Listing 4 — NESA pseudocode for the search loop.
- **Traps:** string-building SQL (injection); forgetting GROUP BY needs aggregate logic; thinking ORM removes the need to understand SQL.
- **Split note:** if this runs long, split into `-Part-1` (SELECT/WHERE/GROUP BY/joins) and `-Part-2` (ORM comparison + parameterisation). Keep lesson `13.2` across both.

### 13-99 — Module review: Programming for the Web
