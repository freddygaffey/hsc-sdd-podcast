NARRATOR: Recap first. Last episode was the W3C and web standards. The W3C is the international body, founded by Tim Berners-Lee, that develops open standards to keep the web interoperable — and the key trap, it *recommends*, it does not enforce. Its five concerns were WIPS-M: WAI accessibility, Internationalisation, Privacy, Security, Machine-readable data. Accessibility runs on the WCAG guidelines and the four principles POUR — Perceivable, Operable, Understandable, Robust. Internationalisation prepares software for many locales; localisation adapts it to one. Two episodes back, big data: the three Vs and M-M-S, and the architecture patterns — monolith, microservices, serverless. Those patterns were about *how you deploy* the back end at scale. Today we zoom into the structure itself — the anatomy every web application has, no matter how it's deployed.

NARRATOR: Every system we've discussed — the interactive site, the e-commerce store, the streaming platform — is built from the same three parts. Learn those parts and where each responsibility lives, and you can model any web system. Today's dot-point, in the syllabus's words: model the elements that form a web development system, including client-side or front-end web programming, server-side or back-end web programming, and interfacing with databases based on Structured Query Language — S-Q-L — or non-S-Q-L. So the home topic is the three-tier model and the database choice behind it. It's foundational; the next several episodes build directly on it. By the end you'll be able to name the three tiers and what each does, distinguish client-side from server-side processing — a very common exam question — and choose between an S-Q-L and a non-S-Q-L database with justification. The word "distinguish" is doing real work: the exam loves to ask you to draw that client-server line.

NARRATOR: The three tiers, and the memory hook we coin now and reuse for the rest of the module: Front, Back, Store. Front is the client-side, the front end. Back is the server-side, the back end. Store is the database, where data is stored. Front, Back, Store — three words, the whole anatomy. Now each one's job, because the marks are in knowing *what lives where*.

NARRATOR: The Front — client-side, or front-end — is everything that runs in the user's web browser. Three technologies you already know: H-T-M-L for the structure and content of the page, C-S-S for visual styling and layout, and JavaScript for interactive behaviour. The front end's job is presentation and interaction — showing things to the user and responding to clicks and typing. The critical fact for the exam: client-side code runs on the user's own machine, so the user can see it and tamper with it. Hold that thought — it's the seed of a security point coming up.

NARRATOR: The Back — server-side, or back-end — runs on the server, not on the user's machine. Its job is request handling, routing, and business logic — receiving the H-T-T-P request, deciding which piece of code handles it, running the application logic, talking to the database, and sending a response back. This is where Python, with a framework like Flask, lives in this course. The user never sees this code; it runs on a machine you control. So the front end asks, and the back end decides and does. Remember the request journey from episode two — Dogs Take Tasty Hambones? The H-T-T-P request at the end of that chant arrives *here*, at the back end. Episode two delivered the request to the server's door; this is what's behind the door. And we build a real back-end request flow in episode twelve, the next chapter — today is the map, that episode is the territory.

NARRATOR: The Store — the database — is where data is persisted so it survives between requests and visits. Log in tomorrow and your data is still there: that's the Store. The back end is the only tier that talks to the Store directly; the front end never touches the database — it always goes through the back end. That boundary is both an architecture fact and a security one. Front, Back, Store: the browser presents, the server processes, the database persists.

QUESTION: This is the highest-value question in the episode, so pause the player. Distinguish between client-side and server-side processing — where each runs, and one consequence of the difference. Work out a full answer, then play.

NARRATOR: Here's the mark-earning version. Client-side processing runs in the user's browser, on their device — it handles the user interface and immediate interactivity, like instantly showing or hiding a menu, with no round trip to the server. Server-side processing runs on the web server, which the developer controls — it handles business logic, data access, and anything that must be trusted or kept secret. And the crucial consequence: because client-side code is visible to and modifiable by the user, it cannot be trusted for security or anything authoritative; server-side processing is where validation, authentication, and sensitive logic must happen. A weak answer says "client-side is the front and server-side is the back". True but shallow — one mark. The strong answer names *where each runs*, *what each is responsible for*, and the *trust consequence*. That trust point is the discriminator.

NARRATOR: Now the security link, because it's the biggest trap in this topic and it pays off in the next module. Imagine a sign-up form that checks the email and password length using JavaScript in the browser. Convenient — the user gets instant feedback. But the trap: a malicious user can bypass that JavaScript — disable it, or send a request straight to your server without ever loading your page. So if your *only* validation is client-side, an attacker walks right past it. The rule, write it down: client-side validation is for convenience; server-side validation is for security. Always validate on the server, because the client cannot be trusted. This is taught in full in Secure Software Architecture — the validate-everything principle — but the seed is here in the three-tier model: the front end is visible and tamperable, so the back end must re-check everything. A genuine forward link, worth real marks in both modules.

NARRATOR: Now the database choice — interfacing with S-Q-L or non-S-Q-L databases, the second half of the dot-point. S-Q-L databases — said "sequel" or "ess-cue-ell" — are relational: data lives in tables of rows and columns with a fixed structure, called a schema, and clear relationships between tables. PostgreSQL, MySQL, SQLite are examples. Non-S-Q-L databases, usually called NoSQL, store data more flexibly — commonly as documents, like J-S-O-N objects, without a rigid schema. MongoDB and Redis are examples. The memory hook: S-Q-L is a Spreadsheet of tables — rigid rows and columns; NoSQL is a Notebook of documents — flexible, free-form entries. Spreadsheet versus Notebook — S, S; N, N.

QUESTION: So when would you actually pick NoSQL over S-Q-L? Give me one clear situation.

NARRATOR: The mark-earning answer names a property of the data, not a vibe. Pick S-Q-L when your data is highly structured with clear relationships and you need reliable, consistent transactions — a banking system or an e-commerce orders table, where relationships between customers, orders, and products matter and you can't afford inconsistency. That reliability comes from ACID properties. Pick NoSQL when your data is flexible or its shape varies, and you need to scale to huge volume fast — storing social-media posts or activity feeds, where each item can have different fields and the volume is enormous. So "outline one situation where NoSQL is preferable to S-Q-L": NoSQL is preferable when storing large volumes of flexible, semi-structured data that doesn't fit a fixed schema, such as a social-media activity feed, because its flexible document model and horizontal scaling handle variety and volume better than a rigid relational schema. Name the data property — flexible and high-volume — and the example. The same data shown both ways, as an S-Q-L table and a J-S-O-N document, is in Listing 1.

NARRATOR: A trap on the database choice: don't say "NoSQL means you can't use S-Q-L" or "NoSQL is just better because it's newer". NoSQL means "not only S-Q-L" — a different tool for a different data shape, not a replacement. Most real systems use both: an e-commerce site might keep its orders in S-Q-L for reliability and its product-view logs in NoSQL for volume. Tool for the job, not newer-is-better.

NARRATOR: One more piece the textbook names, because it shows up in scenario questions: middleware and A-P-Is. Middleware sits between parts of the system and handles cross-cutting concerns — authentication, logging, rate-limiting — on the way through. An A-P-I, an application programming interface, is a defined way for components to talk to each other — the front end calls the back end's A-P-I, and the back end might call a third-party A-P-I like a payment service. You should be able to say middleware sits in the request path doing shared jobs, and an A-P-I is the contract between components. That A-P-I boundary between front and back is also a security boundary — securing it is an SSA topic, another forward link.

NARRATOR: Consolidating the interleaving. Back: the H-T-T-P request from the Dogs Take Tasty Hambones journey in episode two arrives at the Back tier — this episode is what's behind that door. Forward, in the module: the Back tier gets built for real in episode twelve, the back-end request flow, and the Store tier — S-Q-L specifically — is the whole of episode thirteen, databases and queries. Forward, across modules: client-versus-server trust and the validate-server-side rule are core Secure Software Architecture. Back to the request journey, forward to the build episodes, forward to security — three real links.

NARRATOR: Lock in today. Every web system is three tiers — Front, Back, Store. The Front, client-side, runs in the browser using H-T-M-L, C-S-S, and JavaScript, handling presentation and interaction — but it's visible and tamperable. The Back, server-side, runs on the server using something like Python and Flask, handling routing, business logic, and data access — and it's where trust lives. The Store, the database, persists data, and only the Back talks to it. S-Q-L is a Spreadsheet of structured tables for reliable, related data; NoSQL is a Notebook of flexible documents for high-volume, varied data. Traps: never trust client-side validation alone — validate on the server; and NoSQL is "not only S-Q-L", not "S-Q-L is obsolete".

NARRATOR: Exam-style finish — five questions, recall to extended. Pause and attempt each, then the model answer.

QUESTION: Question one, recall. Name the three tiers of a web development system and the main technology or role of each. Pause, then play.

NARRATOR: Model answer: the client-side or front end, which runs in the browser using H-T-M-L, C-S-S, and JavaScript for presentation and interaction; the server-side or back end, which runs on the server using a language like Python for routing and business logic; and the database, which stores and retrieves persistent data. Front, Back, Store — three tiers, three roles. Full marks.

QUESTION: Question two. Distinguish between client-side and server-side processing. Pause, then play.

NARRATOR: Model answer: client-side processing runs in the user's web browser and handles the user interface and immediate interactivity, but because it is visible to and modifiable by the user it cannot be trusted for security. Server-side processing runs on the web server, which the developer controls, and handles business logic, data access, and security-critical operations such as authentication and validation. The distinction is where the code runs and, critically, whether it can be trusted. That trust point lifts it from two marks to full marks.

QUESTION: Question three. A developer adds password-strength checking in JavaScript in the browser. Explain why this alone is not sufficient. Pause, then play.

NARRATOR: Model answer: client-side JavaScript runs on the user's own device and can be disabled, altered, or bypassed by sending a request directly to the server, so a malicious user could submit a weak value despite the check. The same validation must therefore run server-side, where the user cannot tamper with it. Client-side validation improves user experience; server-side validation provides security. Naming the bypass and the server-side requirement is the mark.

QUESTION: Question four. Outline one situation where a NoSQL database would be preferable to an S-Q-L database. Pause, then play.

NARRATOR: Model answer: a NoSQL database is preferable when storing large volumes of flexible, semi-structured data that does not fit a fixed schema — for example, a social-media activity feed where each post may contain different fields and the data volume is very high — because NoSQL's flexible document model and horizontal scaling handle that variety and volume more easily than a rigid relational schema. Name the data property and the example; don't just say "it's faster".

QUESTION: Question five, the extended one. A team is designing an online bookstore. Describe how the three tiers of the web development system would work together to display a book and process an order, and justify one database choice. Pause, plan a structured response, then play.

NARRATOR: Model answer: the client-side front end, built with H-T-M-L, C-S-S, and JavaScript, renders the book page and sends requests when the user views a book or clicks to order. The server-side back end, written in Python, receives each request, routes it to the correct handler, applies business logic such as validating the order and checking stock, and is the only tier that queries the database. The Store — the database — persistently holds the books, customers, and orders; an S-Q-L relational database is a strong choice because the data is highly structured with clear relationships between customers, orders, and products, and its ACID properties keep transactions reliable and consistent, which is essential when handling payments. Critically, all validation and payment logic runs server-side, because client-side code cannot be trusted. That response names all three tiers with responsibilities, justifies the database by the data's structure and the need for reliable transactions, and includes the trust point — a full-mark integrated answer that ties straight back to the e-commerce theme from episode one.

NARRATOR: That's episode six. Front, Back, Store — the browser presents, the server processes and is the only one you trust, and the database persists. S-Q-L is a Spreadsheet for structured, related data; NoSQL is a Notebook for flexible, high-volume data. Never trust the client alone.
