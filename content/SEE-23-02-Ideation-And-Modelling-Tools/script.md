NARRATOR: Recap first. Last episode opened the Software Engineering Project module — the synthesis module about the process of building software, not another pile of features. The marker for the whole module is three verbs: justify, manage, document. The home topic was requirements and feasibility. A requirement is a testable statement of what the software must do or be. Functional versus non-functional is FUN versus PERF — what it does versus how well it does it — and the single most important property of any requirement is that it's measurable, written with acceptance criteria. "It should be fast" is band four; "ninety-fifth percentile under three hundred milliseconds at two hundred users" is band six. Feasibility is TECO — Technical, Economic, Cost-and-schedule, Operational — the honest can-we-build-this call before you commit. Boundaries define what's in, out, and future, to stop scope creep. And we met the data dictionary — exactly where today picks up.

NARRATOR: One expanding-interval callback that matters today. Back in Programming for the Web we modelled a web system — Front, Back, Store, the three tiers — and a data dictionary describes the Store, the database. We'll lean on that. Where today fits: requirements told us what to build. We have testable statements and a data dictionary — but you don't jump straight from a requirements list to code. That's a classic trap. Between "here's what it should do" and "here's the code" sits a modelling stage: working the idea out on paper first. That's today. The outcome is S-E-twelve-oh-six — justifies the selection and use of tools to design, develop, manage and evaluate software. Notice the verb: justifies. This dot-point isn't "list some tools," it's "pick the right tool and say why."

NARRATOR: By the end you'll name the ideation tools and say what each is for; explain the purpose of a data dictionary in design; describe how algorithm design, code generation, testing, installation, and maintenance fit into developing a solution; and — the exam skill — justify choosing a particular tool for a particular job. The dot-point is: explore tools used to develop ideas and generate solutions, including brainstorming, mind-mapping and storyboards; data dictionaries, including selecting appropriate data types; algorithm design; code generation; testing and debugging; installation; and maintenance. That's a long list, so it needs hooks. Let's build them.

NARRATOR: Start with ideation — generating ideas. Three named tools, and here's the hook: B-M-S. Brainstorm, Mind-map, Storyboard. It's the order you'd actually use them: brainstorm a pile of ideas, mind-map to organise them, storyboard the one you picked to see how it'll feel to use. Take each.

NARRATOR: Brainstorming generates many possible solutions without judging them yet. The governing rule is defer judgement — quantity first, accept wild ideas, build on each other's suggestions, filter later. Widen the field before you narrow it. If someone says "that won't work" in the first five minutes, the brainstorm is dead. So: quantity over quality, no criticism during generation. The exam wants the why — premature judgement kills the unusual idea that might've been the good one.

NARRATOR: Mind-mapping is a visual tool: the central concept in the middle, branching outward into related concepts and sub-concepts. For an assignment-tracker you'd branch into user interface, data storage, integration, and features, then sub-branch each. Its value is showing relationships and connections that linear lists hide, and it's collaborative — several people can build one map. Picture it in words: a hub in the centre, spokes radiating out, smaller spokes off each spoke. Use it to organise the brainstormed pile into a structure.

NARRATOR: Storyboarding visualises how a user moves through the system over time — the sequence of actions, step by step, often with how the user feels and where the pain points are at each stage. For the assignment-tracker: morning, the student checks the app and sees "history essay due in two days," feeling anxious but prepared; lunchtime, updates progress, feeling accomplished; evening, plans tomorrow, feeling organised. A storyboard is a narrative of the user journey, frame by frame. Its value is catching usability problems before you build — you spot the awkward step in the story, not after you've coded it. First interleave: a storyboard is user-experience thinking, U-X from Programming for the Web, twelve-oh-four — UI is the look, UX is the experience. Storyboarding is UX done on paper before any UI exists.

QUESTION: Quick check on the ideation set. Name the three ideation tools from the hook B-M-S, and give the one-line purpose of each. Pause the player, say all three, then play.

NARRATOR: B-M-S. Brainstorm — generate many ideas without judging, quantity first. Mind-map — organise ideas visually around a central concept, showing relationships. Storyboard — walk through the user's journey step by step to find usability issues early. Generate, organise, walk-through. If you've got B-M-S, you've got the ideation half of this dot-point.

NARRATOR: Now the modelling tools — turning the chosen idea into a concrete, buildable design. First and most examinable: the data dictionary, which we met last episode. The dot-point pairs it with "selecting appropriate data types," so that's the exam angle. A data dictionary is a table documenting every data element the system handles: for each field, its name, data type, whether it's required, any constraints — length, valid range — and a description. The purpose, in exam terms: to define the structure and meaning of all the system's data in one authoritative place, before you build, so everyone agrees on what's stored and developers know exactly what types to use. QUESTION: Here's a common exam question. Explain the purpose of a data dictionary in project design. Pause, attempt a two-sentence answer, then play.

NARRATOR: Model answer: a data dictionary documents the structure and meaning of every data element in the system — for each field, its name, data type, constraints, and a description — providing a single authoritative reference for the data the solution handles. Its purpose is to remove ambiguity before development: all stakeholders and developers share one consistent understanding of the data and its types, which prevents mismatches, supports accurate database design, and feeds directly into testing and validation. Naming what it contains and why it's done first — to remove ambiguity and drive consistent design — is the mark. Weak answer: "it lists the data." Strong answer names what it removes — ambiguity, and the cost of fixing a data-type mistake after you've built.

NARRATOR: Selecting appropriate data types is a mark-earner on its own. A price should be a Decimal with fixed precision, never a floating-point — floats give rounding errors on money, and there's your interleave back to handling money in e-commerce. A date should be a Date type, not a String, so you can do date arithmetic like "days until due." A student I-D is an Integer if it's a key, or a fixed-length String if it has leading zeros. A "completed" flag is a Boolean. The exam move is to name the type and justify it — "Decimal, to avoid floating-point rounding on currency" earns more than just "Decimal."

NARRATOR: Next: algorithm design. Before you write code, you design the logic — a direct callback to Year 11 programming fundamentals, where you learned pseudocode, flowcharts, and structure charts. Designing the algorithm means expressing the steps in a language-independent form — NESA pseudocode with its BEGIN and END, IF-THEN-ELSE-ENDIF, FOR-NEXT, the left-arrow for assignment — so you can reason about and check the logic before the syntax of a real language gets in the way. A flowchart shows the same logic as a diagram of decisions and flows. A structure chart breaks a system into modules and shows how they call each other — your object-oriented design from Year 11, one task per subroutine, decompose the problem. The exam point: you design the algorithm in pseudocode first because logic errors are far cheaper to fix on paper than in code. The feature's algorithm is in Listing two, in the exact NESA form the exam wants.

NARRATOR: The dot-point also lists code generation, testing and debugging, installation, and maintenance — the tools across the rest of the build. The syllabus is reminding you they're all part of "developing a solution," not just the typing. The hook for this memorisable list: C-T-I-M, said almost like "see-tim." Code generation, Testing and debugging, Installation, Maintenance. Code generation is tools that produce code for you — scaffolding from a framework, an O-R-M generating SQL from your classes (Programming for the Web thirteen-oh-two), an I-D-E autocompleting boilerplate; the routine code is generated so you focus on the logic. Testing and debugging tools — unit-test frameworks, debuggers, the browser dev tools, your E-N-S-P panels from Programming for the Web — verify the code and locate faults. Installation tools — installers, deployment scripts, package managers — get the software onto the target environment reliably. Maintenance tools — version control, issue trackers, monitoring — support the software after it ships, the longest phase of its life. The marker's point: a "tool" here isn't just an idea-generator; it's anything that supports any stage of producing the solution, and choosing well at each stage is what S-E-twelve-oh-six is about.

NARRATOR: The key trap of this dot-point: jumping to code before modelling. The temptation, especially for a strong programmer — exactly who's listening — is to skip the brainstorm, the mind-map, the data dictionary and the pseudocode, and just start coding because you can. It's marked down, and it costs you in a real project, because mistakes caught at the modelling stage are nearly free while the same mistakes caught after you've built are expensive. This is the shift-left idea from Secure Software Architecture fourteen-oh-two — fixing a flaw at design is roughly sixty-three times cheaper than fixing it in production. Modelling is shift-left applied to the whole solution, not just security. A second, related trap: treating the data dictionary as an afterthought — writing it up at the end to satisfy the documentation requirement, instead of using it up front to drive the design. The data dictionary is a design tool, not a paperwork tax. Use it first.

NARRATOR: The worked example — one feature, end to end, the tools chained together. Feature: deadline reminders for the assignment-tracker. Ideation — a quick brainstorm gives push notification, email reminder, in-app badge, a nagging chatbot; mind-map groups those under a "notifications" branch with sub-branches for channel and timing; storyboard the chosen one: two days before the due date, at eight a.m., the student's phone shows "history essay due in two days," and they feel prepared rather than ambushed. Now model it. Data dictionary: an Assignment has a due-date of type Date, a title of type String, a completed flag of type Boolean, plus a reminder-sent flag, Boolean, so we don't send twice. Algorithm design, in pseudocode: for each assignment, compute days-until-due as due-date minus today; if that equals two and completed is false and reminder-sent is false, send the reminder and set reminder-sent to true. That's Listing two. Then code generation produces the boilerplate; testing checks the boundary — exactly two days, plus the edge cases of zero, one, and overdue; installation schedules it to run each morning; maintenance monitors that reminders actually go out. One feature, the full toolkit, idea to maintained code — every stage modelled before it was built.

NARRATOR: The weak-versus-strong contrast, because the verb is justify. Question: "describe a tool used to develop software ideas." Weak answer: "you can use brainstorming to think of ideas." True but thin — no purpose, no reasoning, band four. Strong answer: "brainstorming generates a large quantity of candidate solutions by deliberately deferring judgement during the generation phase, which prevents promising but unconventional ideas from being dismissed too early; the team then filters and refines afterward." The strong answer names the tool, states its mechanism, and justifies why it's done that way. Name it, say how it works, say why. That structure is the mark on any "describe a tool" or "justify a tool" question.

NARRATOR: Exam-style finish — four questions, recall up to application. Pause and attempt each before the model answer.

QUESTION: Question one, recall. Describe two tools used to generate and develop software ideas. Pause, then play.

NARRATOR: Model answer: brainstorming generates a large number of candidate ideas by deferring judgement during the generation phase, aiming for quantity so that unconventional but valuable ideas aren't dismissed early; the team filters and refines afterward. Mind-mapping organises ideas visually around a central concept, branching outward into related concepts so that relationships and connections between ideas become clear, supporting collaborative exploration. Naming two tools and stating the purpose and mechanism of each — not just "they help with ideas" — is the mark.

QUESTION: Question two, the classic. Explain the purpose of a data dictionary in software design. Pause, then play.

NARRATOR: Model answer: a data dictionary documents every data element the system handles — for each field its name, data type, whether it's required, its constraints, and a description — giving a single authoritative reference for the solution's data. Its purpose is to remove ambiguity before development: it ensures developers and stakeholders share one consistent understanding of the data and the appropriate data types, which supports accurate database and validation design and reduces costly data-type mistakes discovered later. Stating what it contains and the why — removing ambiguity, driving consistent design up front — earns full marks.

QUESTION: Question three, application. Explain why a developer should design an algorithm before writing code, and name a tool used to do so. Pause, then play.

NARRATOR: Model answer: designing the algorithm first — in a language-independent form such as NESA pseudocode, a flowchart, or a structure chart — lets the developer reason about and verify the logic before committing to the syntax of a real language. This matters because logic errors are far cheaper to identify and fix at the design stage, on paper, than after the code is written and integrated — the shift-left principle. Pseudocode is the standard tool: it expresses the steps using BEGIN and END, conditionals, and loops without language-specific detail. Naming the tool and justifying with the cheaper-to-fix-early reasoning is the mark.

QUESTION: Question four, application. A team is modelling a new feature. Justify the use of a storyboard over jumping straight to coding the user interface. Pause, plan an answer, then play.

NARRATOR: Model answer: a storyboard models the user's journey through the feature step by step — the sequence of actions, and ideally how the user feels and where pain points occur — before any interface is built. Justifying it over immediate coding: it surfaces usability problems early, while they're cheap to fix, rather than after effort has been spent building the wrong flow; it keeps the design focused on user experience, the UX, not just the look; and it gives stakeholders something concrete to give feedback on before commitment. Coding the interface first risks an awkward flow that's expensive to rework. An answer that names the storyboard's purpose, contrasts the cost of fixing early versus late, and ties it to UX and stakeholder feedback earns full marks.
