NARRATOR: On the evening of the second of November, nineteen eighty-eight, a graduate student at Cornell University sat down at a terminal and, from a computer at M-I-T — not his own, so it couldn't easily be traced back to him — released a small program onto the internet. Except in nineteen eighty-eight, almost nobody called it "the internet" yet. It was a quiet, trusting little network of universities, research labs, and a few military sites — perhaps sixty thousand computers in the whole world. The people on it mostly knew each other. It was built by academics, for academics, on the assumption that everyone connected was a colleague acting in good faith. There were locks on some doors, but plenty were left open. Why wouldn't they be? You were among friends.

NARRATOR: The student's name was Robert Tappan Morris. He was twenty-three. And the program he released that night was, by his own later account, an experiment — he wanted to measure the network, to count how many computers were really out there by writing something that would quietly spread from machine to machine and tally them up. That was the idea. What actually happened: within hours his little program had spread out of control, clogged and crashed computers across the country, and brought a sizeable chunk of the connected world grinding to a halt. It was the first time the network had ever been attacked at scale, and the first time most people realised it could be. Almost everything we now call cybersecurity traces a line back to that one night.

QUESTION: How does a program copy itself from one computer to another, with no human helping it along — no one clicking, no one installing it?

NARRATOR: That question is what makes this historic, because answering it is the definition of a worm. We throw the word "virus" around for any nasty software, but there's a real distinction, and Morris's program is the textbook case of the other kind. A virus needs a host and needs help — it hides inside a file or program, and it only spreads when a person runs that thing, carrying it along like a passenger. A worm needs no passenger at all. A worm spreads itself across a network on its own — it finds another computer, breaks in, copies itself over, then from that new machine goes looking for the next one, and the next, automatically, with no human in the loop. It propagates by itself. That self-spreading is the whole essence of it, and the Morris worm is where the world learned the word in earnest. So the real question is how. How did it break into each new machine, with no person to let it in? The answer is that it didn't have one trick. It had a small toolkit of them — and each one teaches a lesson that is still, decades later, exactly what goes wrong today.

NARRATOR: The first way in is the most technically famous, with a name you'll hear again and again in security: a buffer overflow. Picture it, because the idea is simpler than it sounds. Imagine a form with a box that says "name," with room for, say, twenty letters. You're meant to write your name in it. But the box is badly designed: if you keep writing past the end, your extra letters aren't rejected — they spill out and write over whatever was printed on the rest of the form below. Now imagine the stuff printed below the box is the form's own instructions — what to do next. By overflowing the box, you overwrite those instructions with your own. That is a buffer overflow. A "buffer" is just a fixed-size space in memory set aside to hold incoming data — the box on the form. When a program carelessly accepts more data than the buffer was built to hold, the excess spills out and overwrites the adjacent memory — and in that adjacent memory sits, among other things, the program's record of what to run next. So a carefully crafted overflow — too much data, precisely shaped — doesn't just crash the program. It overwrites the "what to do next" instruction with a pointer to the attacker's own code. The input they sent becomes the code that runs.

NARRATOR: The Morris worm found a program on these computers — a small network service that reported information about a user, the kind of thing you'd ask to see who else was logged in — and it had exactly this flaw. It accepted an input without checking that the input fit the space reserved for it. So the worm sent it an input that was far too long, deliberately crafted, that overflowed the buffer and overwrote the program's instructions with a command to run the worm's own code. The service, trying to be helpful, ran the attacker's code instead. And just like that, the worm was in — no password, no permission, no human involved. It simply exploited a program that trusted its input not to be too big.

QUESTION: Where have you heard that shape before — a program trusting input that turns out to be malicious?

NARRATOR: Everywhere. It's the same fundamental shape as so many attacks across the decades: a program trusts data coming in from outside, and that trust is betrayed. The buffer overflow is the memory-level version — the input is too big and spills into territory it should never reach. It comes down to memory safety: whether a program rigorously guards the boundaries of its memory, refusing to let any input run past the edges of the space set aside for it. The languages and tools of the time made this terribly easy to get wrong, and the program the worm attacked got it wrong. Decades later we're still fighting buffer overflows. It's one of the oldest and most persistent classes of vulnerability there is, and the Morris worm is its grand debut.

NARRATOR: But the worm didn't rely on that alone. Its second route in is, if anything, even more relatable. It tried to guess passwords. The worm carried a built-in list of a few hundred common passwords — obvious, predictable ones, the kind people picked because they were easy to remember. It tried them one by one against accounts on each machine, looking for any password weak enough to guess. It also tried the username itself as the password, and simple variations, because people did that constantly. And it worked, again and again, because in nineteen eighty-eight — and, frankly, in twenty twenty-six — an enormous number of people protect their accounts with passwords a machine can guess in seconds. Once the worm guessed a valid password, it walked straight in the front door, no clever exploit needed. Weak authentication — relying on a secret that's trivially guessable — handed it the keys.

NARRATOR: So the worm spread, machine to machine — breaking in through the overflow, guessing its way past weak passwords, copying itself across, then launching again from each new foothold. And here is where the experiment turned into a disaster, through a single design decision. Morris had thought about the obvious risk: what if his worm reaches a machine that's already infected? If it politely declined to reinfect, a clever administrator could fake being infected — set a flag that says "already got it" — and immunise their machine, ruining the count. So Morris made the worm sometimes reinfect a machine even when a copy was supposedly already there: roughly one time in seven, it installed another copy regardless. He meant that to defeat the fake-out. What it actually did was turn every infected machine into a place where copies of the worm piled up. One copy, then another, then another, each consuming memory and processing power, multiplying, until machines slowed to a crawl and then seized up entirely under the weight of dozens of copies all running at once. It didn't damage files. It didn't steal anything. It just multiplied, and the multiplication itself was the catastrophe. It ate the computers alive with copies of itself.

NARRATOR: So across that night and the next day, system administrators all over the country watched their machines bog down, freeze, and fall over, with no idea why. The network that connected them was the very thing spreading the infection, so as they tried to coordinate and warn each other, the worm raced along the same wires faster than the humans could talk. Some sites, in desperation, simply cut themselves off — pulled the plug — which protected them but also cut them off from any help and any warnings. It was chaos, and the first time the connected world had ever seen anything like it. By the estimates of the time, the worm hit around ten percent of the computers on the internet — somewhere near six thousand machines — and the lost time and frantic recovery ran into serious money. For a network of sixty thousand machines, taking out six thousand was an extraordinary blow.

NARRATOR: Morris, for his part, seems to have been horrified at what he'd unleashed. As it became clear how badly it was spreading, he had a friend send out an anonymous message explaining how to stop it — but the network was so clogged by then that the message barely got through. He was identified, and what followed was itself a first: he became the first person ever convicted under a brand-new American law against computer crime — the law that made unauthorised access to computers a federal offence. He was sentenced to probation, community service, and a fine. Releasing a program onto the network had become, formally, a crime, with a person standing in court answering for it. And in a strange epilogue, Robert Morris went on to a long, respected career as a computer scientist — a professor at the very kind of institution his worm had once brought to its knees.

NARRATOR: Now, what this old story is really about — because for something that happened before most of the internet existed, it is astonishingly modern.

NARRATOR: The clearest lesson is about availability. We tend to think of security as keeping secrets safe — stopping data being stolen or read. But the Morris worm stole nothing and read nothing. It didn't break confidentiality at all. What it destroyed was availability — the simple ability to use the systems. The computers were still there, the data still on them, untouched — but you couldn't get any work done, because the machines were drowning under copies of the worm. A system you cannot use is, for all practical purposes, a system that has failed. Keeping a system up and responsive — defending its availability against the thing that floods it until it collapses — is one of the core pillars of security, every bit as much as keeping its secrets. The Morris worm is the founding example of an availability attack, and you can draw a straight line from it to the modern attacks that knock services offline by burying them in traffic.

NARRATOR: Then the two ways it got in, both lessons that never went out of date. The buffer overflow is the ancestor of an entire dynasty of memory-safety bugs — the failure to guard the boundaries of memory, to check that input actually fits the space you set aside for it before accepting it. The weak passwords are the eternal reminder that authentication is only as strong as the secret behind it: a password a machine can guess from a short list is barely a lock at all. The very oldest internet attack we have a record of came in through a memory bug and a guessable password — and those two doors are still, decades later, among the most common ways attackers get in. The technology changed beyond recognition. The mistakes did not.

NARRATOR: And finally, the thing that makes this the true origin story: this was the night the world learned the network needed defending at all, and the night the defence was born. Until then the connected world ran on trust, and the worm broke that trust in a single evening. The reaction was immediate and lasting. In direct response to the chaos, the first coordinated incident-response team was created — a central group to be the place you call when this happens again, to coordinate the response, warn everyone, organise the cleanup. We now call these C-E-R-Ts — computer emergency response teams — and nearly every country and major organisation has one, all descended from that first one, stood up in the aftermath of this single worm. The idea that when an attack hits you don't panic alone, you have a planned, coordinated response and people to call — incident response as a discipline — was born in the wreckage of the Morris worm. The attack and the field that fights attacks were, in a real sense, born on the same night.

NARRATOR: So here it is, whole, in one breath. A trusting little network of academics. A curious twenty-three-year-old who wanted to count it, and wrote a program that could spread on its own — a worm — to do the counting. Two old, eternal mistakes for it to ride in on: a program that didn't check whether its input fit the memory it had, and accounts protected by passwords a machine could guess. And one fatal decision — reinfect anyway, just in case — that turned a quiet census into a flood, piling copies upon copies until the machines drowned. It stole nothing, read nothing, broke no secrets. It simply took away the ability to use the computers at all — and in doing so it taught the world, all at once and forever, that the network had enemies, that availability was something you had to fight to keep, and that someone needed to be ready to answer the call when the next one came. The first internet attack and the first organised defence against attacks were born the very same night.
