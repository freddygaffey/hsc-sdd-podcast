NARRATOR: Here's the shape of the whole module, because the arc is itself a memory aid. A request starts as packets, gets an address from D-N-S, travels over ports through T-L-S using H-T-T-P, arrives at a three-tier system — front, back, store — gets handled by back-end Python, queries a database with S-Q-L, comes back through an accessible standards-compliant front end, gets made fast, and finally becomes an installable offline app. Packet, to address, to transport, to architecture, to back end, to store, to front end, to performance, to P-W-A. That sentence is the module.

NARRATOR: Start at the top — chapter eleven, applications of web programming. There are three syllabus application types and the exam skill is matching a scenario's driving requirement to the right one. Interactive website or webpage — the defining trait is dynamic, client-side content updating without a full page reload. E-commerce — the defining trait is secure financial transactions and protection of sensitive personal data; the distinguishing feature is security, because the asset is money and identity, not "it has a shopping cart." And a progressive web app, or P-W-A — the defining trait is installable plus offline-capable. The mnemonic is "I Eat Pizza" — I-E-P — Interactive, E-commerce, P-W-A. The trap: a P-W-A is not just "mobile-friendly" or "responsive." Responsive is about appearance; progressive is about capability — install and offline.

NARRATOR: Those P-W-A essentials get their own mnemonic — MISO. Manifest, Installable, Service worker, Offline. The web app manifest is the J-S-O-N file with the name, icons, theme; installable means it can be added to the home screen and launch standalone; the service worker is the background script; and offline is what the service worker delivers. MISO — Manifest, Installable, Service worker, Offline. Hold that, because chapter thirteen builds it for real.

NARRATOR: Next, how data moves on the internet. Data is broken into packets — numbered units. Why packetise? Three reasons: efficient sharing of links, independent routing for resilience, and you only resend the lost packet. Packets can arrive out of order and are reassembled by sequence number — classic trap, don't say they arrive in order. An I-P address is a machine's numeric location. I-P-v-four is four numbers, zero to two-fifty-five, with dots — each chunk is eight bits, two hundred fifty-six values. I-P-v-six solves I-P-v-four address exhaustion. And D-N-S translates a human-readable domain name into a server I-P address by walking a hierarchy — cache, resolver, root, T-L-D, authoritative — cached under a T-T-L, a time to live. Mnemonic for D-N-S: name, number, translator — the domain name, the I-P address, and D-N-S the translator between them. The trap is calling D-N-S "the address" or saying it "loads the page" — no, D-N-S only resolves the name to a number; H-T-T-P fetches the page.

NARRATOR: And the marquee one for this module — the request journey order. "Dogs Take Tasty Hambones" — D-N-S, T-C-P, T-L-S, H-T-T-P. First D-N-S resolves the name to an address, then T-C-P establishes the connection, then T-L-S encrypts it, then H-T-T-P carries the actual request and response. Dogs Take Tasty Hambones. That single hook covers the order of the next chapter too.

NARRATOR: Dogs Take Tasty Hambones. D-N-S — resolve the domain name to an I-P address. T-C-P — open a reliable connection to that address. T-L-S — encrypt the connection so it's private and tamper-proof. H-T-T-P — send the actual request and get the response. If you wrote those four with a job each, that's the mark.

NARRATOR: Chapter eleven, web protocols, ports and secure transport. A protocol is an agreed set of rules; a port is a numbered service doorway on a machine. The ports you must know, as a chant: eighty plain, four-four-three safe, twenty-two secure shell, twenty-one file. H-T-T-P is port eighty, H-T-T-P-S is four-four-three, S-S-H and S-F-T-P are twenty-two, F-T-P is twenty-one, and D-N-S is fifty-three. Then transport security. S-S-L is the old protocol; T-L-S is its modern successor and the current one — don't treat them as identical. Plain text versus cipher text: encryption transforms plain text into cipher text, decryption reverses it. Symmetric encryption uses the same key both ways and is fast; asymmetric uses a key pair, public and private, which solves the key-sharing problem but is slow. Mnemonic: Symmetric is the Same key; Asymmetric is A pair.

NARRATOR: Still in that chapter — the T-L-S handshake. The server sends a certificate signed by a certificate authority; that certificate authenticates the server's identity and carries its public key; the two sides use asymmetric crypto to agree a session key, then switch to fast symmetric encryption for the bulk traffic. Critical trap: a certificate authenticates and carries the public key — it does not itself encrypt; the keys do that. Then two distinctions you reuse all year. Authentication is who you are; authorisation is what you can do. And hashing — a one-way fixed-size fingerprint, used for integrity and password storage — versus encryption. The marquee line, carrying straight into Secure Software Architecture: "Hash is one-way, Encrypt is two-way." A digital signature signs with the private key and verifies with the public key, giving authenticity, non-repudiation, and tamper detection. All of this is just the P-F-W intro; it's taught fully in S-S-A.

NARRATOR: Chapter eleven, big data and web architectures. Big data is the three Vs — Volume, Velocity, Variety — not merely "lots of data"; if you only say "lots of rows" you lose the mark. Data mining is analysing large datasets to discover previously unknown patterns and trends — that's pattern discovery, not storage. Metadata is data about data — a film's genre, cast, year — and it powers search, categorisation, and recommendations. Streaming service management is handling continuous high-velocity real-time data — buffering, throughput, reliability, handling overflow. Mnemonic for those three: M-M-S — Mining, Metadata, Streaming; picture a bag of M&Ms sorted by colour. Then architecture patterns: monolithic is one deployable unit; microservices are independent services that scale and fail in isolation behind an A-P-I gateway; serverless is on-demand functions that auto-scale and you pay per use. The hook: one block, many services, tiny functions. And a C-D-N is edge servers caching content close to users — the same "closer to the user" idea as D-N-S caching.

NARRATOR: Chapter twelve, the W-three-C and web standards. The W-three-C is the international body, founded in nineteen ninety-four by Tim Berners-Lee, that develops open web standards for interoperability across browsers and devices. The key verb: it recommends — a finished standard is called a W-three-C Recommendation — it does not enforce. Its five concerns: W-A-I accessibility, Internationalisation, Privacy, Security, and Machine-readable data. Mnemonic: WIPS-M — wipe the M off the whiteboard. Accessibility runs through W-C-A-G, the Web Content Accessibility Guidelines, built on four principles — Perceivable, Operable, Understandable, Robust — mnemonic POUR. Concrete techniques: labels and alt text, colour contrast of at least four-point-five to one, keyboard operability with a visible focus. Internationalisation, i-eighteen-n, is preparing software for many locales; localisation, l-ten-n, is adapting to one — prepare versus adapt. The trap: don't say the W-three-C enforces, and don't treat accessibility as "for blind users only" — it covers motor, auditory, cognitive, and situational needs too.

NARRATOR: Chapter twelve, modelling a web development system — the three tiers. Front-end, the client side: the browser running H-T-M-L structure, C-S-S style, JavaScript behaviour; it's visible and tamperable. Back-end, the server side: Python and Flask doing routing, business logic, and data access; it's trusted. And the database is the Store: it persists data, and only the back end queries it. Mnemonic: Front, Back, Store. The consequence you must state: client-side runs on the user's untrusted device, server-side on the developer's trusted server — therefore validation, authentication, and sensitive logic must be server-side. Storage: S-Q-L or relational means fixed-schema tables with relationships and A-C-I-D — Postgres, MySQL, SQLite — while NoSQL is flexible schema-less documents that scale horizontally — MongoDB, Redis. Mnemonic: S-Q-L is a Spreadsheet of tables; NoSQL is a Notebook of documents. NoSQL doesn't mean S-Q-L is obsolete — it means "not only S-Q-L." Middleware sits in the request path doing cross-cutting jobs; an A-P-I is a defined contract and a security boundary.

QUESTION: Retrieval. Pause and name the three tiers, and for each one say whether it's trusted or untrusted and what that means for where you validate. Then play.

NARRATOR: Front, Back, Store. The front end is untrusted — it runs on the user's device and can be edited, so you never trust it for security. The back end is trusted — it's your server. The store is queried only by the back end. The consequence: validation, authentication, and sensitive logic must live server-side, because anything client-side can be bypassed. Saying "Front, Back, Store" plus the trust consequence is the full-mark answer.

NARRATOR: Chapter twelve, the browser and developer tools. A browser runs a page with a rendering engine and a JavaScript engine — Chrome uses Blink and V-eight, Firefox uses Gecko and SpiderMonkey, Safari uses WebKit and JavaScriptCore — so the same code can behave differently, which is cross-browser compatibility. The responses: cross-browser testing, feature detection — test the capability, not the browser name — and progressive enhancement. The same-origin policy is a browser-enforced security boundary; cross-origin is blocked unless C-O-R-S allows it. The dev-tool panels, mnemonic E-N-S-P, plus Console: Elements shows the live D-O-M and applied C-S-S for layout bugs; Network shows every H-T-T-P request with status code, timing, and size for failed or slow requests; Storage shows cookies, local storage, and service-worker caches for login and cache bugs; Performance shows slow JavaScript and paint for sluggish pages. Status-code triage: four-x-x is the request or permissions side — four-oh-four is a missing endpoint, four-oh-one and four-oh-three are auth and authorisation — and five-x-x is the server, five hundred being a server-side code error. The trap: nothing client-side can hide secrets or enforce security — everything is visible and editable in those same tools.

NARRATOR: Chapter twelve, C-S-S, U-I and U-X principles. Separation of concerns: H-T-M-L is structure, C-S-S is presentation, JavaScript is behaviour — mnemonic Structure, Style, Behaviour. C-S-S delivers maintainability through an external stylesheet — one source of truth, change once and it updates everywhere — plus design tokens, which are C-S-S custom properties for colour, spacing, and fonts. Responsive design means the layout adapts to screen size via media queries, mobile-first — and responsive is not a P-W-A. U-I is the look — buttons, colour, layout; U-X is the experience — easy, efficient, satisfying to use. The principles, mnemonic C-FAN-H: Consistency, Feedback, Accessibility, Navigation, and visual Hierarchy — a C-FAN cooling you, plus H. Accessibility techniques fulfil W-C-A-G and POUR: at least four-point-five-to-one contrast, keyboard operable with visible focus, never colour alone, text alternatives, large touch targets, designed in from the start. The trap: inline styles kill maintainability, and accessibility as an afterthought fails inclusivity.

NARRATOR: Chapter twelve, front-end libraries and frameworks. The core distinction is inversion of control: you call a library, but a framework calls you — the Hollywood principle. The three syllabus types, mnemonic F-T-C — for the code: Frameworks like React, Vue, Angular, with component architecture, state management, routing, for complex single-page applications; Template engines like Jinja with Flask, server-side rendering that fills H-T-M-L templates with data, living on the Back tier; and predesigned C-S-S classes like Bootstrap and Tailwind, ready-made styled components for consistency off the shelf. Then the adopt-versus-build decision: BUILD versus BUY. Build when it's bespoke, unique, performance-critical, or you need control; buy when it's standard, the timeline is tight, or the team already knows it. Benefits of adopting: speed, tested code, consistency, community. Costs: bundle size and slower load, learning curve, lock-in, over-engineering. The trap: a heavy framework on a simple page is over-engineering — "a framework is always better" is wrong.

NARRATOR: Chapter twelve, open-source software and content management systems. Open source means the source code is public to view, modify, and distribute, developed openly by a community — "free as in freedom, not necessarily price"; openness is a strength, not a weakness. The pillars, mnemonic L-C-C: Licence, Community, Contribution — and contribution is fork, branch, pull request, which is version control at scale. Licence families: permissive — M-I-T, Apache, B-S-D — allow proprietary commercial use with just attribution; copyleft — G-P-L — means distributed derivatives must also be open-sourced. Mnemonic: Permissive is Polite — use freely, say thanks; Copyleft is Contagious — the derivative must share alike. The software supply chain is the third-party code you depend on, plus its dependencies — an attack surface, like the x-z backdoor of twenty twenty-four. A C-M-S is software to create, manage, and publish content through an admin interface without coding — WordPress dominant — built from a database, a template engine, and an admin: Front, Back, Store again. Hosted trades control for managed convenience; self-hosted trades responsibility for control. The trap: leaving a C-M-S and its plugins unpatched is the commonest small-site hack.

NARRATOR: That closes chapters eleven and twelve. Now chapter thirteen — building the back end for real, where the densest exam mnemonics live, so we slow down. Server-side Python and the request flow. The back-end lifecycle, mnemonic R-H-D-R: Route — map the U-R-L to a handler; Handle — run the business logic, read the request object; Data — query or update the database; Respond — render H-T-M-L through a template engine or return J-S-O-N. Crucial layer distinction: the web server software, like nginx or Apache, accepts connections, parses H-T-T-P, serves static files, and terminates T-L-S, then forwards dynamic requests to the web framework, Flask, where the routes, handlers, and business logic live — different layers, don't confuse them. C-R-U-D maps cleanly: Create, Read, Update, Delete equals S-Q-L's INSERT, SELECT, UPDATE, DELETE equals H-T-T-P's POST, GET, PUT, DELETE. And untrusted input: anything from the request must be validated and sanitised server-side — concatenate it into a query and you get S-Q-L injection; echo it into H-T-M-L and you get cross-site scripting.

NARRATOR: Chapter thirteen, databases — S-Q-L, common queries, and the O-R-M. The marquee hook here is the clause order: "So Few Workers Go Home On-time" — SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY. The database enforces that order. WHERE filters rows before grouping, so it can't see aggregates; HAVING filters groups after aggregation, so it can test a COUNT or a SUM. GROUP BY needs an aggregate and every non-aggregated selected field. Joins: an INNER JOIN keeps only rows matching in both tables; a LEFT JOIN keeps all left rows plus matches, with nulls otherwise — mnemonic, INNER keeps matches in both, LEFT keeps everything on the left. S-Q-L injection is gluing user input into the command string so the input becomes S-Q-L — the classic input of quote-O-R-quote-one-equals-one returns every row. The fix is a parameterised query, also called a prepared statement: a placeholder and the value passed separately, so the database treats it as data, never command. An O-R-M, like SQLAlchemy, maps classes to tables and objects to rows and auto-generates the S-Q-L, reusing your object-oriented programming — the trade-off is productivity, readability, and auto-parameterisation against performance overhead, and you still need to know S-Q-L to fix slow queries.

QUESTION: High-value retrieval. Pause the player and write out the full S-Q-L clause order from the mnemonic, then say the one-sentence difference between WHERE and HAVING. Then play.

NARRATOR: So Few Workers Go Home On-time — SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY. WHERE filters individual rows before any grouping happens, so it cannot reference an aggregate; HAVING filters the groups after aggregation, so it's where you test a COUNT or a SUM. Using WHERE on an aggregate when you needed HAVING is a classic lost mark — so is getting the clause order wrong, which is a straight syntax error.

NARRATOR: Chapter thirteen, performance and page-load management. Rule one: measure before you optimise — profile with the dev tools, the Network panel for big or slow requests and the Performance panel for slow JavaScript, to find the real bottleneck and avoid premature optimisation. The core toolkit, mnemonic C-C-C-L: Cache — browser cache via Cache-Control headers plus a server-side cache like Redis for expensive query results, "store closer, reuse"; Compress — gzip or Brotli shrink text bytes sixty to eighty percent, plus minify your C-S-S and JavaScript; C-D-N — geographically distributed edge servers caching static assets near users for lower latency; and Lazy-load — defer non-critical below-the-fold resources so the initial render is fast. Bundling and minification combine files into fewer requests. And caching is the same "closer to the user" idea as D-N-S caching and the C-D-N — its third appearance. The trap: don't optimise before measuring, and don't put private per-user data in a shared public cache.

NARRATOR: Chapter thirteen, designing and implementing a progressive web app — where MISO finally gets built. A P-W-A is distinguished from a standard website by MISO: Manifest — the J-S-O-N file with name, icons, theme, and standalone display; Installable — added to the home screen, launching standalone, app-like; Service worker — a background script that acts as a programmable proxy intercepting every network request; and Offline — the service worker serving the cached app shell, the H-T-M-L, C-S-S, JavaScript, and icons, when there's no network, with offline-created data queued and synced on reconnect. The service worker requires H-T-T-P-S, because it can intercept all traffic and so must be tamper-proof. That offline caching is the same caching idea from the performance lesson, repurposed — its fourth appearance. And you still apply U-I and U-X — C-FAN-H — and accessibility — POUR — to the app shell, especially Feedback: clearly signal the offline state, including to screen readers. The trap, one more time: a P-W-A is not a responsive website — responsive is appearance, progressive is capability.

A certificate authenticates and carries the public key — it does not encrypt. Authentication versus authorisation: who you are versus what you can do. I-P versus D-N-S: D-N-S is the translator, not the address. Packets are numbered and can arrive out of order. Responsive is not a P-W-A. NoSQL doesn't mean S-Q-L is obsolete. NARRATOR: To finish, a short exam-style set across the module, real N-E-S-A verbs, model answer after each. Your last self-test.

QUESTION: Question one. Distinguish between a responsive website and a progressive web app. Pause, attempt it, then play.

NARRATOR: Model answer: a responsive website adapts its layout to different screen sizes using media queries — that's about appearance. A progressive web app adds capability: it's installable via a web app manifest, launching standalone from the home screen, and it works offline because a service worker intercepts network requests and serves a cached app shell — MISO. So responsive is how it looks at different sizes; progressive is what it can do — install and run without a network. Naming the manifest, service worker, and offline capability is the mark.

QUESTION: Question two. A developer builds a database query by inserting the user's search text directly into the S-Q-L string. Identify the vulnerability and describe the correct fix. Pause, then play.

NARRATOR: Model answer: this is S-Q-L injection — because the user input is concatenated into the command string, malicious input can become part of the S-Q-L itself, for example a value that always evaluates true to return every row. The fix is a parameterised query, also called a prepared statement: use a placeholder in the query and pass the user's value separately, so the database treats it strictly as data and never as command, making injection impossible. Hear the marking boundary: a weak answer says "validate the input" or "sanitise it" and stops — that names a symptom, not the mechanism, and caps you low. The strong answer names parameterised queries and explains that the value is treated as data, not code. That mechanism is the mark.

QUESTION: Question three, the harder one. A streaming-video startup expects sudden, unpredictable spikes in users worldwide and slow page loads. Recommend an architecture and two performance techniques, justifying each. Pause the player, plan an answer, then play.

NARRATOR: Model answer: for unpredictable worldwide spikes, recommend microservices rather than a monolith — independent services scale and fail in isolation, so the video-delivery service can scale without scaling everything, and one failing service doesn't take the whole platform down; serverless functions are defensible for spiky on-demand work that scales to zero. For performance, from C-C-C-L: use a C-D-N so static assets and video segments are cached on edge servers geographically close to users, cutting latency and origin load — directly relevant for a worldwide audience; and use caching plus compression — cache expensive results and gzip or Brotli the text assets — but measure first with the dev tools to find the real bottleneck before optimising. A weak answer just says "use the cloud" or "make it scalable" with no named pattern; the strong answer names the architecture with isolation reasoning, two C-C-C-L techniques with justification, and the measure-first principle. That's the whole of Programming for the Web — make it work on the internet — and next in the course it's made trustworthy in Secure Software Architecture, where every deferred security promise here gets paid off.
