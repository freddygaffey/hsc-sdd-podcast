NARRATOR: Start with version control. A version control system — Git is the standard — tracks every change to your files over time, so multiple developers can work together and you keep a complete, recoverable history. You've met this before: Year eleven mentioned online code-collaboration tools, and Programming for the Web, twelve oh-six, covered version control for web apps — fork, branch, pull request. Today we make it a project discipline. Five core concepts, and here's the hook: R-C-B-M-T, said as "Really Clean Builds Manage Trust." Repository — the store of the whole project and its history. Commit — a saved snapshot with a descriptive message. Branch — a parallel line of work, isolating a feature so it doesn't disturb the working code. Merge — combining a branch's changes back. Tag — a marker on a specific version, used for releases. R-C-B-M-T: Repository, Commit, Branch, Merge, Tag.

NARRATOR: Now the workflow, because "use Git" isn't an answer — the marker wants discipline. The rule of thumb: commit small, branch per feature, never commit straight to main. The main branch holds production-ready code and is protected — no direct commits, changes only via review. You branch off for each feature, build there, and when it's done and reviewed, you merge it back. Bigger teams add a develop branch for integration and hotfix branches for urgent production fixes; that's the Git-flow model. The key exam point: a pull request — a request to merge a branch — is a review gate, tying straight back to code review and continual checking from quality assurance, twenty-four oh-six. Branching plus pull requests is how a team builds incrementally — REFF from twenty-five oh-one — without stepping on each other. Listing one is a real Git branch-and-tag command sequence in bash.

NARRATOR: Tagging and releases — semantic versioning. A release is tagged with a version number in three parts: major-dot-minor-dot-patch. Major, the first number, means breaking changes — not backward-compatible. Minor, the second, means new features that are backward-compatible. Patch, the third, means backward-compatible bug fixes. So two-point-one-point-three to two-point-one-point-four is a bug fix; to two-point-two-point-zero is a new feature; to three-point-zero-point-zero is a breaking change. The hook: "Major breaks, Minor adds, Patch fixes." Tagging a release means you can always go back to exactly what was running on any date — the foundation of rollback, the thing Knight Capital lacked.

QUESTION: Quick check. You ship version one-point-four-point-two, then fix a login bug. What's the next version number, and why? Pause, answer, then play.

NARRATOR: It's one-point-four-point-three. A backward-compatible bug fix bumps the patch number — the third — and only the patch. You don't touch minor or major because you added no feature and broke no compatibility. "Major breaks, Minor adds, Patch fixes." Had you added a whole new dashboard, it'd be one-point-five-point-zero — minor up, patch reset to zero. And if you'd changed the A-P-I in a way that breaks existing clients, two-point-zero-point-zero. Getting the right digit is the mark on a semantic-versioning question.

NARRATOR: Now data backup, the other safety net. Version control protects code; backup protects data — the database, user files, the things that aren't in your repository. The single most important hook in this episode, and you should dump it in any backup question, is the three-two-one rule. Three-two-one: three copies of your data, on two different types of media, with one copy off-site. Three copies means the original plus two backups, so a single failure can't wipe you out. Two media types means not all eggs in one technology — not three copies on one disk that can all die together. One off-site means a fire, flood, or theft at one location can't destroy everything — the trap-killer for "backup equals it's on my laptop," which is zero of the three. Say it with me: three copies, two media, one off-site.

NARRATOR: And the types of backup, because the HSC asks you to choose and justify. Full backup — a complete copy of everything; safe and simple to restore, but slow and storage-heavy. Incremental backup — only what changed since the last backup of any kind; fast and small to make, but slower to restore because you replay a chain. Differential backup — everything changed since the last full backup; a middle ground. The standard strategy combines them: a weekly full, daily incrementals, plus continuous transaction logs for a critical database — and you test your restores, because a backup you've never restored is a hope, not a plan. NARRATOR: Now rollback, and the Knight Capital story in full, because it ties the whole episode together. Rollback means reverting to a previous known-good version when a deployment goes wrong. With tagged releases and a clean version-control history, rollback is one command: redeploy the last good tag. Knight Capital, the first of August twenty-twelve. They were deploying new trading software to eight servers. The deployment failed to update one server, which still held old, dead code. Worse, a configuration flag the new code reused had previously controlled that dead code — so on the un-updated server, the flag switched the dead code back on. When trading opened, that server started firing millions of unintended orders into the market. For forty-five minutes, with no automated kill switch and no rehearsed rollback, Knight Capital bled about four hundred and forty million dollars. The firm was effectively destroyed within days.

NARRATOR: Map that onto today's content — that's the exam skill. Failure one: inconsistent deployment — the release didn't reach all servers, which proper release hygiene and automated, verified deployment would have caught. Failure two: leftover dead code in the repository that should have been removed — a version-control and code-review hygiene failure. Failure three, the fatal one: no rollback plan. With a tagged last-good release and a tested rollback, they could have reverted in seconds instead of bleeding for forty-five minutes. And underneath it all, this was a direct, big-bang deployment of a critical system — DiP-PP from twenty-three oh-three, the cheapest and riskiest choice, made where the cost of failure was catastrophic. Knight Capital is the cautionary tale for direct cut-over without a safety net — and why version control, tagged releases, and rollback aren't bureaucracy, they're survival.

QUESTION: Question one, recall. Explain the role of version control in a software project. Pause, then play.

NARRATOR: Model answer: version control systematically tracks all changes to a project's files over time, maintaining a complete, recoverable history. It enables multiple developers to work simultaneously without overwriting each other's work, by isolating work on separate branches and merging through reviewed pull requests; it allows any previous version to be recovered, supporting rollback when a change causes problems; and it provides accountability through commit messages showing who changed what and why. That makes it foundational to collaboration, quality, and recovery. Naming history, collaboration, branching and merging, and rollback is the mark.

QUESTION: Question two, application. Your project is at version two-point-three-one. You add a new export feature without breaking anything. What is the new version number and why? Pause, then play.

NARRATOR: Model answer: the new version is two-point-four-point-zero. Adding a new, backward-compatible feature increments the minor version — the second number — from three to four, and resets the patch to zero. The major stays at two because nothing breaks backward compatibility. The rule is Major breaks, Minor adds, Patch fixes — a new feature is an "add," so minor goes up. Getting the minor bump with the patch reset is the mark.

QUESTION: Question three, the harder one. Describe a data backup strategy for a school's student-records system and justify it. Pause, plan a structured answer, then play.

NARRATOR: Model answer: the strategy should follow the three-two-one rule — three copies of the data, on two different media types, with one copy off-site. Concretely: the live student-records database; a daily incremental backup to a separate on-site server for fast recovery of recent data; and a weekly full backup, encrypted, pushed to off-site cloud storage. Retain the database transaction log for point-in-time recovery, and test restores monthly. This is justified because the multiple media protect against hardware failure, the off-site copy protects against site disaster such as fire or theft, the incremental-plus-full mix balances backup speed against restore speed, and tested restores guarantee the backups are usable. Because student records are sensitive and the system must be available, this protects confidentiality through encryption and availability — the A in CIA-AAA — through recoverability, and forms the disaster-recovery plan. Naming three-two-one, giving concrete copies, and justifying each against a named risk is the full-mark answer.

QUESTION: Question four, extended. Using the Knight Capital incident, explain how version control and backup practices could have prevented or limited a deployment disaster. Pause, plan, then play.

NARRATOR: Model answer: Knight Capital lost about four hundred and forty million dollars in forty-five minutes when a deployment left old, dead code on one server and a reused configuration flag reactivated it, flooding the market with erroneous trades, with no way to stop or revert. Several practices could have prevented or limited this. Proper version-control hygiene — removing dead code and reviewing changes through pull requests — would have eliminated the leftover code that caused the fault. Release hygiene with a verified, automated deployment would have caught that one server was not updated, since inconsistent deployment was the trigger. Tagged releases with a rehearsed rollback plan would have let them revert to the last known-good version in seconds rather than bleeding for forty-five minutes — the most important missing safety net. More broadly, deploying a critical financial system by a direct, big-bang cut-over with no fallback was the riskiest implementation choice from DiP-PP, when the catastrophic cost of failure justified a safer, staged approach with rollback. The judgement: version control, tagged releases, tested rollback, and disciplined deployment are not overhead — for a high-stakes system they are the difference between a recoverable mistake and a company-ending one. A full-mark answer maps specific practices onto specific failures and connects to the implementation-method risk.
