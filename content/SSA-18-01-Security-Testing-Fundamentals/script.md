NARRATOR: First — code review. Code review is a human reading the source code, line by line, looking for security flaws. It's the oldest and, for certain bugs, still the most powerful method, because a human understands intent and context a machine can't. A reviewer spots logic errors — an authorisation check that's subtly wrong, a discount that can go negative, a flow that fails open — the business-logic flaw no scanner will ever flag, because the code is syntactically fine, it's just wrong. The cost: slow, expensive, and only as good as the reviewer. Code review uniquely catches flawed logic and intent. This is where the xz backdoor story pays off. In twenty-twenty-four an attacker spent two years patiently becoming a trusted maintainer of xz, a compression library buried deep in Linux, then slipped in a backdoor for remote access to vast numbers of servers. It was caught — barely, days before going worldwide — not by a scanner, but by one engineer who noticed the software ran half a second slower than it should and dug into the source until he found it. That's human attention catching what every automated tool missed. We tell it fully in its own case study; for now, hold it as proof that humans-reading-code is irreplaceable.

NARRATOR: Second and third are the pair the exam loves to contrast, so do them together — S-A-S-T and D-A-S-T. S-A-S-T is Static Application Security Testing. Static means the application is not running — the tool analyses the source code itself, scanning for dangerous patterns: string-concatenated S-Q-L, hard-coded credentials, weak cryptography, an unsafe eval. Like reviewing the blueprints before the building goes up. D-A-S-T is Dynamic Application Security Testing. Dynamic means the application is running — the tool attacks it from the outside like an attacker would, firing malicious inputs at the live endpoints and watching how it responds. Like breaking into the finished building while the alarms are on. The memory hook to dump in the exam: Static equals Source — the code, not running; Dynamic equals Doing — the app, running. Static reads the Source; Dynamic watches it Doing.

NARRATOR: Now the part that earns the "evaluate" marks — what each catches that the other can't. S-A-S-T runs early, even on incomplete code, finds the exact file and line, and covers all code paths — but it can't see runtime or configuration problems, and it produces false positives, flagging things that aren't really exploitable. D-A-S-T finds real, exploitable vulnerabilities as they behave at runtime, including server and configuration issues S-A-S-T can't see — but it runs late, needs a running app, only tests the paths it reaches, and can't pinpoint the code. So they're complementary: static finds it in the source early; dynamic proves it's exploitable at runtime. Use both. The exam line: S-A-S-T analyses source code without running it, finding flaws early and by location but with false positives; D-A-S-T tests the running application from the outside, finding real exploitable behaviour but late and without pinpointing the code.

QUESTION: Quick check, the classic. Distinguish S-A-S-T from D-A-S-T. Pause, give me the core difference and one strength of each, then play.

NARRATOR: S-A-S-T is static — it analyses the source code without executing it, so it finds vulnerabilities early in development and points to the exact line, but it can't see runtime issues and yields false positives. D-A-S-T is dynamic — it tests the application while it's running by simulating real attacks from the outside, so it finds genuinely exploitable flaws and configuration issues, but it needs a running app and can't pinpoint the code. Static reads the Source; Dynamic watches it Doing. Naming static-versus-running and a strength of each is the mark.

NARRATOR: Fourth — vulnerability assessment. A vulnerability assessment is a broad, systematic scan to identify and catalogue known weaknesses across a system — out-of-date libraries, missing patches, misconfigurations, components with publicly known vulnerabilities. The emphasis is breadth and known issues: it produces a prioritised list of what's wrong, by checking your software and its dependencies against databases of published vulnerabilities. It tells you what's vulnerable, not how far an attacker could get. This is where the Equifax breach pays off. In twenty-seventeen Equifax, a credit agency holding data on a hundred and forty-seven million people, was breached through a single unpatched vulnerability in Apache Struts, a web framework. A patch existed. They didn't apply it. A vulnerability assessment would have flagged that exact missing patch — that's the whole point of the method, and the whole tragedy of the breach. Vulnerability assessment uniquely catches the known-and-unpatched. Its own case study tells it fully; hold it as the lesson that patch management is a security control.

NARRATOR: Fifth — penetration testing, or pen testing. A penetration test is a simulated real attack by a skilled human — an ethical hacker — who actively tries to break in, chaining weaknesses together exactly as a real attacker would, to prove what's genuinely reachable and how much damage is possible. The difference from a vulnerability assessment is depth versus breadth: the assessment lists what might be wrong; a pen test proves what can actually be exploited and how far it goes. A pen tester might start from the assessment's list, then demonstrate "I used this flaw to get in, then chained it with this one to reach your customer database." Pen testing uniquely catches the real, chained, demonstrated exploit. The cost: it's the most expensive and skill-intensive, and it's a snapshot in time. So the five form a layered strategy, cheap-and-broad to expensive-and-deep: code review and S-A-S-T early in development; D-A-S-T and vulnerability assessment on the running system; penetration testing as the deep, periodic human attack.

QUESTION: Quick retrieval. For one web app, name what S-A-S-T, vulnerability assessment, and penetration testing each catch that the others would miss. Pause, say all three, then play.

NARRATOR: S-A-S-T catches insecure coding patterns in the source — a string-built query, a hard-coded password — early and by line, before the app runs. Vulnerability assessment catches known weaknesses in the deployed system — an unpatched library, a missing update, a misconfiguration — across breadth. Penetration testing catches what's actually exploitable, chaining flaws the way a real attacker would, proving real-world impact in depth. Source patterns, known issues, real chained exploits — different lenses, which is exactly why you use all of them. That's the evaluate-the-tools answer.

NARRATOR: Now the second half of the dot-point — surviving an incident, because finding weaknesses isn't enough; you plan for the day one gets through. Four ideas to name, and here's the hook: Harden, then Breach-handle, then Disaster-recover, while keeping the business Continuing — H-B-D, with continuity running through it. Hardening means reducing the attack surface before anything happens: remove unused services and accounts, close unnecessary ports, apply least privilege, patch everything, change default passwords, turn off verbose error messages. Hardening makes the system a smaller, tougher target — the proactive, secure-by-default mindset from earlier in the module, applied to the deployed system. Breach handling, or incident response, is what you do when an attack succeeds: detect it, contain it to stop the spread, eradicate the cause, recover the systems, and review so it can't recur. The insight the marker wants: you plan this before a breach, not during — a rehearsed incident response plan is the difference between controlled recovery and chaos.

NARRATOR: Then the two continuity concepts, which students mix up. Business continuity is keeping the business operating during a disruption — the plans, backups, redundancy and alternative processes so that, even mid-incident, critical functions keep running. Disaster recovery is the specific subset focused on restoring the I-T systems and data after a disruption — restoring from backups, failing over to a secondary site, getting the systems back. The clean distinction: business continuity is keeping the whole business running through the disruption; disaster recovery is restoring the technology after it. Continuity is the broad goal; recovery is the technical restore. Both depend on something concrete and examinable: backups — tested, off-site, regular — because an untested backup is a guess, and a disaster recovery plan you've never rehearsed is a wish.

QUESTION: This is the one to lock in — pause the player and answer it properly, because it's a near-certain extended question. Outline a disaster recovery plan for a web service, and explain how it relates to business continuity. Pause now, work it out, then play.

NARRATOR: Here's the full-mark answer. A disaster recovery plan for a web service defines how the technology is restored after a disruption: regular, automated backups of the database and application, stored off-site so a single failure doesn't destroy both the system and its backup; a documented restore procedure with defined recovery objectives — how quickly service must be back and how much data loss is acceptable; redundant infrastructure or a secondary site to fail over to; and regular testing of the plan, because an untested backup or restore may not work when needed. This sits inside business continuity, the broader objective of keeping critical functions operating throughout a disruption — disaster recovery restores the I-T systems, while business continuity keeps the business as a whole, including manual fallback processes and communication, functioning. Naming backups off-site, recovery objectives, redundancy, testing, and the continuity-versus-recovery relationship is the band-six answer.

QUESTION: Question two, the classic. Distinguish between static and dynamic application security testing. Pause, then play.

NARRATOR: Model answer: static application security testing analyses an application's source code without executing it, identifying insecure coding patterns early in development and pinpointing the exact location, though it cannot detect runtime or configuration issues and produces false positives. Dynamic application security testing tests the running application from the outside by simulating attacks, identifying genuinely exploitable vulnerabilities and configuration flaws as they behave at runtime, though it requires a running application, runs later, and cannot identify the exact code. Static reads the Source; Dynamic watches it Doing. Naming the not-running-versus-running distinction with a strength of each is the mark.

QUESTION: Question three. Distinguish between a vulnerability assessment and a penetration test. Pause, then play.

NARRATOR: Model answer: a vulnerability assessment is a broad, systematic scan that identifies and catalogues known weaknesses across a system, such as unpatched components and misconfigurations, producing a prioritised list of what is vulnerable. A penetration test is a deeper, simulated attack by a skilled tester who actively exploits and chains weaknesses to demonstrate what an attacker could really achieve and how far they could reach. The difference is breadth versus depth: the assessment lists what might be wrong; the pen test proves what can actually be exploited. Naming the breadth-versus-depth, list-versus-exploit distinction is the mark.

QUESTION: Question four. Explain the role of penetration testing and why it is used alongside other methods. Pause, then play.

NARRATOR: Model answer: penetration testing simulates a real attack by a skilled ethical hacker who actively attempts to exploit and chain vulnerabilities, demonstrating which weaknesses are genuinely reachable and how much damage is possible — providing evidence of real-world impact that scanners cannot. It is used alongside other methods because it is expensive, skill-intensive and a point-in-time snapshot, and because each method finds different flaws: code review and static testing catch source-level issues early, dynamic testing and vulnerability assessment catch runtime and known issues, and penetration testing confirms real exploitability. Together they provide layered, complementary coverage. Naming the simulated-real-attack role and the complementary-coverage justification is the mark.

QUESTION: Question five, the extended one. A company is launching an online banking service. Describe how it should test the application's security and prepare to survive a breach. Pause, plan a structured response, then play.

NARRATOR: Model answer: testing should be a layered strategy. During development, conduct code review to catch business-logic flaws a human spots, and static application security testing to find insecure code patterns early and by line. On the running system, run dynamic application security testing to find exploitable runtime and configuration flaws, and a vulnerability assessment to identify known issues such as unpatched components. Before launch and periodically, commission a penetration test in which a skilled tester actively exploits and chains weaknesses to prove real-world impact. Each method is used because it catches what the others miss. To survive a breach, harden the system first — patch everything, remove unused services, apply least privilege, and disable verbose errors — to shrink the attack surface. Prepare an incident response plan in advance for detecting, containing, eradicating and recovering from a breach. And implement disaster recovery with regular, tested, off-site backups and defined recovery objectives, within a broader business continuity plan that keeps critical banking functions operating throughout any disruption. A strong response names all five testing methods with their distinct value, then covers hardening, breach handling, and disaster recovery within business continuity. That's full marks.
