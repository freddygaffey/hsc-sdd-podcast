NARRATOR: Start with protocols and ports — the most directly examinable thing in the episode. It shows up as matching questions and short-answer recall, and it's free marks if you've memorised it. A protocol is an agreed set of rules for how two machines talk. A port is a numbered doorway on a machine — a server has thousands, and different services listen on different port numbers so traffic knows which service it's for. You know this from any networked program you've written; the HSC just wants the standard numbers.

NARRATOR: Here are the ones you must know, and the chant to lock them in — coined now, reused in every recap: "eighty plain, four-four-three safe; twenty-two secure shell, twenty-one file". Unpack it. H-T-T-P, the web's plain unencrypted protocol, runs on port eighty — "eighty plain". H-T-T-P-S, the secure encrypted version, runs on port four-four-three — "four-four-three safe". S-S-H, the secure shell for remote login, and S-F-T-P which rides on it, run on port twenty-two — "twenty-two secure shell". And plain F-T-P, file transfer, runs on port twenty-one — "twenty-one file". Eighty plain, four-four-three safe; twenty-two secure shell, twenty-one file. Say it three times and it's yours. A couple more worth knowing by name: D-N-S runs on port fifty-three — the Dogs from last episode answer on fifty-three — and the email protocols, S-M-T-P for sending, P-O-P-three and I-M-A-P for retrieving. But the four in the chant are the high-value ones. The full table is in Listing 1.

QUESTION: Why does it even matter which port a protocol uses — isn't that just trivia? Pause if you like, then play.

NARRATOR: Two reasons, both exam-relevant. First, it's the cleanest possible recall question — "match the protocol to its port" is a gift if you've memorised the chant, a guaranteed loss if you haven't. Second, the port tells you the security posture at a glance: traffic on port eighty is plain H-T-T-P, readable by anyone in the middle; traffic on four-four-three is H-T-T-P-S and encrypted. So "the site uses port eighty" is itself a security finding — the data is in plain text. That connects to the plain-text-versus-cipher-text idea we'll hit in a moment, and to the e-commerce security requirement from episode one: a checkout page on port eighty instead of four-four-three is a serious flaw.

NARRATOR: A terminology trap before we go further, because it costs marks. You'll see "S-S-L" and "T-L-S" used almost interchangeably. S-S-L — secure sockets layer — is the old protocol; T-L-S — transport layer security — is its modern successor and what's actually used today. People still say "S-S-L certificate" out of habit, but the live protocol is T-L-S. For the exam: T-L-S is the current one, S-S-L its predecessor, and don't write as if they're the same thing. I'll say "ess-ess-ell" and "tee-ell-ess" so you hear the difference.

NARRATOR: The centrepiece: how H-T-T-P-S secures data in transit. This is the "explain how H-T-T-P-S secures data" five-marker, and it's the Tasty in Dogs Take Tasty Hambones — the T-L-S step. First, two encryption concepts you need named. Plain text is readable data — your password sitting there legible. Cipher text is that same data after encryption — scrambled, random-looking, useless to anyone who intercepts it. Encryption is the transformation from plain text to cipher text; decryption is the reverse. The job of T-L-S is to make sure that, between your browser and the server, everything travels as cipher text.

NARRATOR: Now the two kinds of encryption, because the handshake uses both and the exam asks you to distinguish them. Symmetric encryption uses the same key to encrypt and decrypt — fast, but both sides need that one secret key, which is a problem if you've never met. Asymmetric encryption uses a pair of keys: a public key anyone can have, which encrypts, and a private key kept secret, which decrypts. Slower, but it solves the key-sharing problem because you can hand out the public key freely. The memory hook: "Symmetric is the Same key; Asymmetric is A pair". Same key, A pair — even the first letters help.

QUESTION: Here's a genuinely worthwhile one — pause the player for this. If asymmetric encryption solves the key-sharing problem, why don't we just use it for everything and forget symmetric? Work out the trade-off, then play.

NARRATOR: Because asymmetric is computationally expensive — slow for bulk data. So T-L-S does something clever, and this is the heart of the mark-earning answer: it uses asymmetric encryption once, at the start, purely to agree on a shared secret key — then switches to fast symmetric encryption for the actual conversation. Best of both: the public-private pair solves "how do we agree a secret without having met", and the symmetric key makes the ongoing traffic fast. If you can say that sentence in an exam — asymmetric to exchange the key, symmetric for the bulk data — that's a band-six discriminator most students miss.

NARRATOR: Let me walk the full H-T-T-P-S handshake at exam depth, because narrating this is the five-marker. Step one: your browser connects to the server on port four-four-three and asks to start a secure session. Step two: the server sends back its S-S-L certificate, which contains the server's domain name, its public key, and a digital signature from a Certificate Authority — a C-A. Step three: your browser checks the certificate — is it for the right domain, is it still valid and not expired, and is it signed by a Certificate Authority the browser trusts, like Let's Encrypt or DigiCert? This proves the server is who it claims to be — it's authentication of the server's identity. Step four: once trusted, the browser and server use the certificate's public key, asymmetrically, to agree on a shared session key. Step five: from there on, all traffic is encrypted symmetrically with that session key — cipher text both ways. So a strong answer to "explain how H-T-T-P-S secures data in transit": the browser and server perform a T-L-S handshake in which the server presents a certificate, verified against a trusted Certificate Authority to authenticate its identity; they then use asymmetric encryption to agree a shared session key, and encrypt all subsequent data symmetrically, so any intercepted traffic is unreadable cipher text. That hits identity, key exchange, and encryption — the three things the marker wants.

NARRATOR: A trap right here, because it catches people: a certificate does not encrypt anything by itself. Its job is to authenticate — prove identity — and carry the server's public key. The encryption is done with the keys, not by the certificate. Students write "the certificate encrypts the data" and lose the mark. Say it correctly: the certificate authenticates the server and provides its public key; the keys do the encrypting.

NARRATOR: Now authentication versus authorisation. "Distinguish between authentication and authorisation" is one of the most common short-answer questions in the course, and the words look so similar that students blur them. Authentication answers "who are you?" — proving identity, like logging in with a password. Authorisation answers "what are you allowed to do?" — your permissions once you're identified, like whether you can delete a record or only read it. The hook: authentication is who you are; authorisation is what you can do. Identity, then permission. A worked contrast: when you log into an online store, entering your correct password is authentication; being able to view your own orders but not the admin dashboard is authorisation. Weak answer: "authentication is logging in and authorisation is permissions" — partial. Strong answer: "authentication verifies a user's identity, for example by checking a password, whereas authorisation determines what an authenticated user is permitted to do, such as which resources or actions their role grants" — that names the distinction and gives the example. These two are part of the security foundation the next module formalises as the C-I-A triad and friends — you'll meet the full dump-on-the-page hook, C-I-A-A-A, in Secure Software Architecture. For now, just nail the pair.

NARRATOR: Two more security concepts to name, both pointing forward to Secure Software Architecture. First, hash values. A hash function takes any data and produces a fixed-size fingerprint of it. Two properties: it's one-way — you can't reverse a hash back to the original — and it's deterministic — the same input always gives the same hash, but the tiniest change to the input gives a completely different hash. That's why hashes store passwords safely and check data integrity — if the data was tampered with, the hash won't match. And here's the single most important distinction in the whole security strand, which we'll repeat all year: hashing is not encryption. Encryption is two-way — you encrypt so you can later decrypt. Hashing is one-way — there's no "un-hashing". The hook, straight from the security module: "Hash is one-way, Encrypt is two-way". Write that down now; it's the trap that separates bands in Secure Software Architecture.

NARRATOR: Second, digital signatures. A digital signature uses the private key to sign and the matching public key to verify, and it gives you two things the exam names: authenticity — proof of who created the message — and non-repudiation — the signer can't later deny they signed it. It also detects tampering, because the signature is tied to a hash of the content; change the content and the signature no longer verifies. So "outline the purpose of a digital signature": a digital signature verifies the authenticity of a message and provides non-repudiation, proving it came from the claimed sender and was not altered, by signing with the sender's private key and verifying with their public key. Notice signatures and certificates are cousins — the Certificate Authority digitally signs the certificate, which is exactly why your browser can trust it. There's a signature-verification routine in NESA pseudocode in Listing 3 so you can rehearse the exam form.

NARRATOR: Let's lock in today. Protocols run on ports: eighty plain, four-four-three safe; twenty-two secure shell, twenty-one file — and D-N-S on fifty-three. T-L-S, the modern successor to S-S-L, secures H-T-T-P-S by having the server present a certificate that a Certificate Authority has signed — authenticating identity and carrying a public key — then using asymmetric encryption to agree a session key and symmetric encryption for the fast bulk traffic, turning plain text into cipher text. Symmetric is the same key; asymmetric is a pair. Authentication is who you are; authorisation is what you can do. Hashing is one-way; encryption is two-way. And a digital signature gives authenticity and non-repudiation by signing with a private key and verifying with a public one. The traps: S-S-L versus T-L-S, the certificate authenticates rather than encrypts, and never confuse hashing with encryption.

NARRATOR: Now the exam-style finish — five questions, recall up to extended. Pause and attempt each, then I give the model answer with the mark-earning phrasing.

QUESTION: Question one, recall. State the default port numbers for H-T-T-P, H-T-T-P-S, S-S-H, and F-T-P. Pause, then play.

NARRATOR: H-T-T-P is port eighty, H-T-T-P-S is port four-four-three, S-S-H is port twenty-two, and F-T-P is port twenty-one. That's the chant — eighty plain, four-four-three safe, twenty-two secure shell, twenty-one file. Four numbers, four marks.

QUESTION: Question two. Distinguish between authentication and authorisation. Pause, then play.

NARRATOR: Model answer: authentication is the process of verifying a user's identity — confirming they are who they claim to be, for example by checking a username and password. Authorisation is the process of determining what an authenticated user is permitted to do — which resources or actions their role grants, such as read-only versus full administrative access. The distinction is identity versus permission: authentication is who you are, authorisation is what you can do. Both definitions and the contrast is the full mark.

QUESTION: Question three. Distinguish between symmetric and asymmetric encryption, and state one advantage of each. Pause, then play.

NARRATOR: Model answer: symmetric encryption uses the same single key to both encrypt and decrypt, and its advantage is speed, making it efficient for large amounts of data. Asymmetric encryption uses a pair of keys — a public key to encrypt and a private key to decrypt — and its advantage is that it solves the key-distribution problem, since the public key can be shared openly without compromising security. Same key versus a pair; fast versus secure key exchange. For a bonus, note T-L-S uses both: asymmetric to exchange the key, symmetric for the data.

QUESTION: Question four, five marks. Explain how H-T-T-P-S secures data transmitted between a browser and a web server. Pause, give a real attempt, then play.

NARRATOR: Model answer: when a browser connects to a server over H-T-T-P-S on port four-four-three, the two perform a T-L-S handshake. The server presents an S-S-L certificate containing its domain name and public key, signed by a trusted Certificate Authority; the browser verifies this certificate to authenticate the server's identity and confirm it is valid and not expired. Using the certificate's public key, the browser and server use asymmetric encryption to securely agree on a shared session key. All subsequent data is then encrypted symmetrically with that session key, so it travels as cipher text and cannot be read or altered if intercepted. Five marks live in five named ideas: the certificate, C-A verification and authentication, asymmetric key agreement, symmetric bulk encryption, and the plain-to-cipher-text result.

QUESTION: Question five, the harder one. A small business is setting up an online store. Explain three security measures, with the correct terminology, that should be in place to protect customers' payment details, and justify each. Pause, plan three, then play.

NARRATOR: Model answer: first, the site must use H-T-T-P-S on port four-four-three with a valid T-L-S certificate, so that payment details are encrypted into cipher text in transit and cannot be intercepted in plain text — a checkout served over plain H-T-T-P on port eighty would expose card numbers. Second, the certificate must be issued by a trusted Certificate Authority, authenticating the store's identity so customers and browsers can verify they are not dealing with an imposter site. Third, customer passwords must be stored as salted hash values rather than plain text, because hashing is one-way and protects credentials even if the database is breached. Each measure names the technique and justifies it by the threat it counters — encryption against interception, certificates against impersonation, hashing against credential theft. That structure — measure plus justification, three times — is exactly what an extended-response marker rewards.
