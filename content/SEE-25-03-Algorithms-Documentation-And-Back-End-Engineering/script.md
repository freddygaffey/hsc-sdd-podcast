NARRATOR: First half: artefacts and documentation. A development artefact is any document, diagram, or resource produced across the lifecycle — requirements specs, design documents, the data dictionary, A-P-I specifications, code documentation, test plans, bug reports. The dot-point verb is document, and the SEE marker rewards it because S-E-twelve-oh-nine and oh-six are about managing and documenting, not just coding. The principle to name is living documentation: documents kept current as the code changes, not written once and abandoned. Stale documentation is worse than none — it actively misleads. So you version it, review it, and where you can, generate it from the code — a docstring becomes A-P-I docs automatically. The artefact-maintenance lifecycle is in Listing one: create, update, review, archive.

NARRATOR: Now documenting an algorithm specifically — examinable, and it reuses Year eleven directly. You document an algorithm three ways, and the contrast is itself the teaching point. One, the real code in the actual language. Two, the NESA pseudocode version — BEGIN, END, IF-THEN-ELSE-ENDIF, the exact form you write for marks. Three, a comment header — what the function does, its inputs, its outputs, any assumptions. Listing two is a documented Python function with a docstring; Listing three is the same algorithm in NESA pseudocode. Why both? The code shows how it actually runs, the pseudocode is what the exam marks, and writing them side by side checks your logic before you ever run it — logic errors are cheap on paper. That's a callback to ideation and modelling, twenty-three oh-two: model first, build second. The trap is the undocumented clever one-liner — code so terse nobody, including future-you, can maintain it. Clever and undocumented loses marks; clear and documented earns them.

NARRATOR: Last piece of the first half: citing sources and libraries, and allocating resources. Every external library you use, you document — name, version, licence, and purpose. Why does the HSC care? Two reasons. Legal compliance: licences impose obligations, and an incompatible licence — say a copyleft G-P-L library inside a closed-source product — is a real legal problem. And integrity: you credit others' work, and you can track which dependency introduced a vulnerability. That last point is a hard interleave with Secure Software Architecture, chapter nineteen — supply-chain risk: a known vulnerability in a dependency, a C-V-E, is your problem the moment you ship it. Remember the xz backdoor — a malicious dependency nearly compromised the world's S-S-H. Documenting your dependencies is how you even know what you'd have to patch. Listing four is a dependency record with name, version, licence, and purpose. Allocating resources means assigning the right people, time, and tools to each part of the build — the scheduling thinking from the Gantt episode, twenty-four oh-four, applied during construction.

QUESTION: Quick check before the second half. Why document an algorithm in both real code and NESA pseudocode — what does each give you that the other doesn't? Pause, answer, then play.

NARRATOR: The real code is executable and precise to the language — it shows how the algorithm actually runs. The NESA pseudocode is language-independent and is the exact form the exam marks your algorithm answers against, so writing it rehearses the marks and lets you reason about the logic before committing to any language — catching logic errors cheaply on paper. Together they cross-check: if the pseudocode and the code disagree, one of them is wrong. Showing both is itself a documentation-quality signal the marker rewards.

NARRATOR: Now the second half, the bigger half: back-end engineering, and why it decides project success. The front end is what users see; the back end is the engine behind it — servers, business logic, databases, A-P-Is, security. Most users never think about it — which is exactly why it's underrated and over-examined: a beautiful front end on a broken back end is a failed project. The dot-point gives you four contributions, and here's the memory hook: T-E-I-S, said "tease," because the back end teases the front end into working. T, Technology used. E, Error handling. I, Interfacing with the front end. S, Security engineering. Each is a callback — let me take them in turn.

NARRATOR: T, technology. The back end is web servers, databases, and A-P-I frameworks — and choosing them is a justify decision, S-E-twelve-oh-six. Web servers like Nginx; frameworks like Django, Flask, or Express. Databases split into relational — MySQL, PostgreSQL, with tables and S-Q-L, said "sequel" — and non-relational, NoSQL, like MongoDB or Redis. The selection factor is the data: structured, related data with integrity needs wants relational — that's the relational schema from Programming for the Web thirteen oh-two, the same primary-key-to-foreign-key modelling, now realised. And it's a direct callback to the back-end request flow from Programming for the Web thirteen oh-one: a request arrives, the server routes it, business logic processes it, the data-access layer queries the database, a response goes back. The back end is that whole journey.

NARRATOR: E, error handling — where good back ends separate from bad. The back end must handle errors gracefully: catch exceptions, return sensible responses, never crash or leak. That's a callback to Year eleven programming fundamentals — exception handling, try-and-except, validating before you process. But there's a critical security twist from Secure Software Architecture: error handling is part of secure coding, the E in V-S-E — Validate, Sanitise, Errors. The rule is fail securely: when something goes wrong, the user sees a clean, generic message; the detailed stack trace, the database error, the file path go to a private log, never to the user. Why? A leaked stack trace is reconnaissance for an attacker — it reveals your stack, your queries, your structure. Listing five shows the fail-securely pattern: log the detail internally, return a generic message externally. Weak back ends print the raw exception to the screen — bad user experience and a security hole at once.

NARRATOR: I, interfacing with the front end — the contract between the two halves. The back end exposes A-P-Is the front end calls, and the quality of that interface decides how easily the two teams work together. A clean, well-documented A-P-I — clear endpoints, predictable responses, versioning — lets front-end and back-end developers work in parallel against an agreed contract. That's a callback to Programming for the Web's point that collaborative front-end and back-end work practices improve a web solution, and it's why the dot-point says back-end engineering eases development, not just enables it. A messy, undocumented interface is where projects bog down: every change breaks the other team.

NARRATOR: S, security engineering — the heaviest interleave, because the back end is where security actually lives. The front end can validate for convenience, but it can never be trusted — anyone can bypass it and call your A-P-I directly. So real security must sit in the back end: authentication and authorisation, your CIA-AAA from Secure Software Architecture; input validation and parameterised queries to stop SQL injection; passwords stored salted and hashed, where hash is one-way and encrypt is two-way; and data encrypted in transit over T-L-S. Every secure-coding principle from chapter sixteen and the crypto from chapter fifteen lands here. The back end isn't invisible plumbing — it's the security boundary of the entire system. Treating it as plumbing is the trap that gets projects breached.

NARRATOR: Let me anchor the whole episode in a real disaster that's pure back-end failure — the perfect story because the front end was irrelevant. Knight Capital, twenty-twelve, a trading firm. They deployed new code to their servers but left old, dead code on one of eight servers, and a reused flag accidentally switched that dead code back on. The back end started firing millions of erroneous trades. No documentation caught it, no error handling stopped it, no rollback. In forty-five minutes they lost about four hundred and forty million dollars and the firm effectively died. The lesson in today's exact terms: back-end engineering — the technology, the error handling, the deployment discipline — is not invisible plumbing; it's the difference between a company and a smoking crater. We'll tell the full deployment-and-rollback story next episode, the marquee version-control cautionary tale.

NARRATOR: Worked example, weak versus strong. The brief: explain the contribution of back-end engineering to a solution's success. The weak answer: "The back end stores the data and makes the app work." True, vague, band three — it names no contributions and shows no understanding. The strong answer uses T-E-I-S: "Back-end engineering contributes through the technology chosen — an appropriate server and database, such as a relational database for structured data with integrity needs; through error handling that fails securely, logging details internally while returning safe generic messages so the system stays robust and doesn't leak information; through a clean, documented interface with the front end, which lets front-end and back-end developers work in parallel and eases development; and through security engineering — authentication, authorisation, input validation, parameterised queries, and hashed passwords — because the back end is the real security boundary that a front end can never be." Four named contributions, each explained, each tied to success or ease. That's the six-mark shape.

NARRATOR: Exam-style finish — four questions. Pause and attempt each before the model answer.

QUESTION: Question one, recall. Why should a developer document an algorithm using both source code and pseudocode? Pause, then play.

NARRATOR: Model answer: source code shows how the algorithm actually executes in a specific language, while NESA pseudocode is language-independent and is the form against which exam algorithm answers are marked; writing both lets the developer reason about the logic before committing to a language, catch logic errors cheaply on paper, and provides clear documentation that future maintainers can understand regardless of language. The two also cross-check each other. Naming both purposes is the mark.

QUESTION: Question two, application. A back end prints the full database error to the user's screen when a query fails. Explain why this is a problem and what should happen instead. Pause, then play.

NARRATOR: Model answer: printing the full database error is both a usability and a security failure. For usability, a raw error message is meaningless and alarming to a user. For security, it leaks internal information — the database structure, query text, and technology stack — which an attacker can use to plan an attack such as SQL injection. Instead, the back end should fail securely: log the full error detail to a private internal log for developers, and return only a clean, generic message to the user, such as "something went wrong, please try again." This is the error-handling part of secure coding — the E in Validate, Sanitise, Errors. The mark is for identifying the information leak and stating the log-internally, generic-message-externally pattern.

QUESTION: Question three, the six-marker. Explain the contribution of back-end engineering to the success and ease of software development. Pause, plan a structured answer, then play.

NARRATOR: Model answer: back-end engineering contributes in four main ways. Technology — selecting and justifying an appropriate server, framework, and database, for example a relational database for structured, related data, which determines whether the system can store and process data correctly and at scale. Error handling — handling exceptions gracefully and failing securely, so the system stays robust and does not leak sensitive information, which directly affects reliability and security. Interfacing with the front end — exposing a clean, well-documented A-P-I so that front-end and back-end developers can work in parallel against an agreed contract, which eases and speeds development. And security engineering — implementing authentication, authorisation, input validation, parameterised queries, and password hashing on the server, because the back end is the real security boundary that a front end can never enforce. Together these make the back end the foundation that determines whether a solution is reliable, secure, scalable, and maintainable. A full-mark answer names all four T-E-I-S contributions and ties each to success or ease, not just lists them.

QUESTION: Question four, extended. A team has a polished front end but their project keeps failing in production. Explain how weak back-end engineering could be the cause, referencing at least three back-end contributions. Pause, plan, then play.

NARRATOR: Model answer: a polished front end cannot compensate for weak back-end engineering, and three contributions explain the failures. Poor technology choices — an unsuitable or unscalable database — could mean the system slows or crashes under real load even though the interface looks fine. Poor error handling — unhandled exceptions or leaked stack traces — means small faults cascade into crashes and expose internal information, as in the Knight Capital failure where dead back-end code with no error handling or rollback caused catastrophic losses in minutes. And weak security engineering — relying on the front end for validation, since a front end can always be bypassed — leaves the system open to SQL injection or broken authentication, so it gets breached in production. The judgement: because users only see the front end, teams under-invest in the back end, yet the back end is where reliability, security, and scalability are actually engineered, so weakness there is the most common hidden cause of production failure. A full-mark answer references at least three named contributions and ties them to concrete production failures.
