NARRATOR: The integrated scenario — the centrepiece. The real H-S-C extended question hands you one brief and makes you reach across every module, so here's a deliberately four-module stimulus. A state education department wants a new online platform: students and parents log in to see grades, book parent-teacher interviews, and receive an A-I-powered early-warning flag for students at academic risk. It handles sensitive data on minors, must comply with privacy and accessibility law, has a fixed eighteen-month deadline with executive reporting, but the parent-facing interface must be refined through real feedback. Plan, secure, automate, build, test, and evaluate it. Let me walk the whole course through one brief.

NARRATOR: First, the project process — SEE. Requirements, FUN versus PERF: functional — "a parent can book an interview," "the system flags an at-risk student"; non-functional, measurable — confidentiality via encryption and role-based access, availability twenty-four-seven, page loads under two seconds, W-C-A-G double-A accessibility. Feasibility, TECO: technically demanding because of the A-I component, economically justified against staff time saved, schedule tight at eighteen months so build a risk buffer, operationally it must fit how teachers and parents actually work. Boundaries: grades, booking, and the flag in scope; payments out of scope; a mobile app future scope. Implementation method, DiP-PP: sensitive and high-stakes, so not direct cut-over — recommend a pilot in a few schools then a parallel or phased rollout, justified by the risk profile. Development approach: the textbook WAgile case — Waterfall stage-gates for the regulated records core and executive milestones, Agile sprints for the parent-facing booking interface that needs feedback, with the when and how explained. Manage it on a Gantt with the gates as milestones, watching the critical path, around the Scope-Time-Cost triangle. Communicate with I-F-N. Quality-assure with the three Cs.

NARRATOR: Second, make it work — Programming for the Web. It's a web app, so it runs on the request journey — Dogs Take Tasty Hambones: D-N-S resolves the address, T-C-P connects, T-L-S encrypts on port four-four-three, H-T-T-P carries the request. The architecture is Front-Back-Store: a front-end the parents see, a back-end with the business logic and the A-I flag, a data store for the relational records — the data dictionary from twenty-three oh-two realised as a schema. The back end is engineered with T-E-I-S — chosen technology, error handling that fails securely, a clean A-P-I interface, and security engineering.

NARRATOR: Third, make it secure — Secure Software Architecture. Sensitive data on minors, so CIA-AAA is mandatory: Confidentiality through encryption and role-based access — a parent sees only their own child; Integrity of records; Availability of the platform; Authentication and Authorisation; Accountability through audit logging. Passwords hashed and salted — hash is one-way, encrypt is two-way. All input validated and queries parameterised — V-S-E and SQL-injection defence, especially on a public-facing system. T-L-S in transit. Security is shift-left — written as a non-functional requirement at the start, not bolted on. Security testing: S-A-S-T on the source, D-A-S-T on the running app.

NARRATOR: Fourth, make it smart — and make the smart responsible — Software Automation. The early-warning flag is machine learning: A-I is the goal, M-L is one way. This is the high-risk part, because an A-I flag on a child is a fairness and ethics minefield. The bias sources, S-H-L-M: if the training data over-represents one group — Sampling and Historical bias — the model could unfairly flag students from particular backgrounds, a self-fulfilling label. So you mitigate: diverse representative data, fairness metrics measured per group, and human oversight — a teacher reviews every flag, never an automated consequence. That's PABT's Bias and Transparency, and the impact assessment S-P-E-E. The model is a decision support, not a decision maker — the responsible-A-I judgement the integrated question rewards.

NARRATOR: Fifth, build, test, evaluate — back to SEE. Build incrementally — REFF — with quality — S-R-I — version-controlled — R-C-B-M-T — and backed up — three-two-one — with a rollback plan, because this is high-stakes. Test with O-S-A-E-S, path-and-boundary data, and — the Healthcare-dot-gov lesson — a real load test before launch. Collect and analyse feedback on the parent interface with C-S-A. Evaluate the whole thing with C-E-R: criteria from the requirements and QA, evidence from the tests and feedback with actual-versus-expected, reflection on maintainability, deployment, and lessons. That single answer — process, web, security, responsible A-I, build, test, evaluate — walks nearly every mnemonic in the course through one brief. Breadth across modules, tied together by the project process, is the whole point of the synthesis module.

QUESTION: Your turn to integrate, the way the exam will make you. A council wants an online system for residents to report and track potholes, including an A-I feature that auto-prioritises reports by severity from a photo. In one structured answer, choose a development approach and implementation method, name two security measures, and state one responsible-A-I safeguard. Pause the player, plan across all four modules, then play.

NARRATOR: Model answer. Development approach: WAgile, defensibly, or Agile — the council reporting and any compliance need governance, Waterfall-heavy, while the resident-facing reporting interface benefits from iterative feedback, Agile-heavy; explain the when and how. Or argue Agile if it's a small, low-regulation build. Implementation method, DiP-PP: moderate stakes, so a pilot in one suburb then a phased rollout is reasonable; reject direct cut-over for a public system. Security, CIA-AAA, two measures: authentication and role-based authorisation so residents see only their own reports and only staff can change status; and input validation plus parameterised queries to stop SQL injection on the public report form — V-S-E. Plus T-L-S in transit. Responsible-A-I safeguard: because severity from a photo is an A-I judgement that affects resource allocation, keep a human in the loop — staff review the auto-priority before dispatch — and check the model for bias, S-H-L-M, so it doesn't systematically deprioritise reports from particular areas. Hear the marking boundary: a weak answer says "use Agile and add security and A-I" — vague, no justification, one module. A full-mark answer names the approach with justification, matches the implementation method to the risk, gives two concrete CIA-AAA-grounded security measures, and gives a real responsible-A-I safeguard — human oversight plus bias check. That's all four modules in one answer.

NARRATOR: Now the big cross-module traps in one place, because the integrated exam is where students confuse modules. Trap one, from SEE: "evaluation equals it works" — no, evaluation is C-E-R, a justified judgement against criteria with evidence and reflection. Trap two: untestable requirements — every requirement is measurable, FUN versus PERF with acceptance criteria. Trap three, the implementation methods: phased splits by feature, pilot splits by user group, parallel runs both at once, and never direct cut-over for a high-risk system — the Knight Capital and Denver lessons. Trap four: "Waterfall is always bad" and "Agile means no docs" — both false; and WAgile needs the when-and-how, not "best of both." Trap five: QA versus testing — QA prevents, testing detects. Trap six, cross-module: "hash is encryption" — no, hash is one-way, encrypt is two-way. Trap seven: trusting the front end for security — never; the back end is the security boundary, and a front end can always be bypassed. Trap eight: "the algorithm is neutral" — no, M-L mirrors its data, S-H-L-M; high overall accuracy can hide deep unfairness to a group. Trap nine: optimising without measuring, and skipping the load test — the Healthcare-dot-gov failure. And the meta-trap that costs the most integrated marks: answering a four-module scenario from only one module. The exam rewards reaching across, so always ask "where do the other three modules touch this?"

NARRATOR: Pull the three SEE case studies together one last time — the same lesson in three costumes, mapped onto the whole module. Denver airport: unbounded requirements and an infeasible plan — a TECO failure — plus a direct big-bang cut-over of a giant untested system, the worst DiP-PP choice. Failed in defining and planning, before code. Healthcare-dot-gov: Waterfall under a fixed deadline with frozen flawed requirements, late integration, and no load testing — Waterfall's weaknesses amplified, and a testing failure. Failed across defining, planning, and testing. Knight Capital: a bad deployment with dead code and no rollback — a version-control, back-end-engineering, and deployment-hygiene failure that lost four hundred and forty million dollars in forty-five minutes. All three say the same thing: the failures happened in the process — requirements, feasibility, methodology, testing, deployment — not in clever code. That's the through-line of the module and the marker for the whole course: justify, manage, document.

NARRATOR: Final exam-style set, mirroring the real exam: a couple of rapid distinguish questions, then one extended integrated response. Pause and attempt each before the model answer. This is your last self-test of the course.

QUESTION: Question one, rapid distinguish. Distinguish quality assurance from testing, and testing from evaluation. Pause, then play.

NARRATOR: Model answer: quality assurance is the overall process — criteria, reviews, continual checking, compliance — that prevents defects across the whole solution. Testing is one activity within quality assurance: executing test cases to detect defects, for example comparing actual with expected output. Evaluation is the broader judgement of overall effectiveness against criteria, using testing results and synthesised feedback as evidence, plus reflection. So QA prevents, testing detects, and evaluation judges with the evidence. Naming all three relationships is the mark.

QUESTION: Question two, rapid recall across modules. A scenario involves storing user passwords and flagging fraud with machine learning. Name the correct technique for the passwords and one bias risk for the model. Pause, then play.

NARRATOR: Model answer: passwords should be salted and hashed, not encrypted — hashing is a one-way function, so even if the database is stolen the passwords can't be reversed, whereas encryption is two-way and reversible. "Hash is one-way, encrypt is two-way." For the fraud model, a key bias risk is that the training data reflects historical or sampling bias — S-H-L-M — so the model could disproportionately flag a particular group, since machine learning mirrors and amplifies whatever is in its data; this is mitigated with representative data, per-group fairness metrics, and human oversight. Getting hashing-not-encryption and naming a real bias source is the mark.

QUESTION: Question three, the extended integrated response — the capstone. A hospital wants a new patient-management system with an online booking portal and an A-I tool that predicts no-show risk for appointments. In one structured answer, recommend a development approach and implementation method with justification, explain how you would secure it, how you would build and test it, how you would responsibly include the A-I feature, and how you would evaluate the finished solution. Pause the player, plan across the whole course, then play. Take your time — this is the one that matters.

NARRATOR: Model answer, walking the whole course through one brief. Approach: WAgile — the patient-records core and regulatory compliance need Waterfall stage-gates for governance, documentation, and predictable milestones, while the booking portal needs Agile sprints with user feedback; explain the when and how — Waterfall where governance dominates, Agile where adaptability dominates. Implementation method, DiP-PP: safety-critical health software, zero tolerance for failure, so recommend a pilot in one department then a parallel rollout — running the new system alongside the old as a fallback to verify outputs — rejecting direct cut-over, the Knight Capital and Denver lesson, and justifying the higher cost by the catastrophic consequences of error. Define requirements as FUN versus PERF with measurable acceptance criteria, assess feasibility with TECO, set boundaries, and plan it on a Gantt around Scope-Time-Cost with I-F-N communication and the three Cs of quality assurance, including compliance with health-privacy and accessibility law.

NARRATOR: Continuing the model answer. Securing it — CIA-AAA, mandatory for sensitive health data: Confidentiality through encryption and strict role-based access so a clinician sees only their patients; Integrity of records; Availability twenty-four-seven; Authentication and Authorisation; Accountability through full audit logging. Hash and salt passwords — hash is one-way, encrypt is two-way; validate and sanitise all input and parameterise queries to stop SQL injection — V-S-E; T-L-S in transit; security written in as a requirement from the start — shift-left — and verified with S-A-S-T and D-A-S-T. Building and testing it: build incrementally — REFF — with quality — S-R-I — version-controlled with R-C-B-M-T, backed up with three-two-one, and a rollback plan. Test with a plan — O-S-A-E-S — using path-and-boundary data and, critically, a real load test before launch, the Healthcare-dot-gov lesson. The responsible A-I feature: the no-show predictor is machine learning — A-I is the goal, M-L is one way — high-risk because it affects patient care, so guard against bias, S-H-L-M, with representative data and per-group fairness metrics, keep a human in the loop so staff decide based on the prediction rather than the model acting automatically, and be transparent — PABT — assessing impact with S-P-E-E. Evaluating it — C-E-R: criteria from the requirements and the C-U-P-S quality criteria; evidence from the test plan, path-and-boundary results, actual-versus-expected comparison, and synthesised user feedback from C-S-A; reflection on maintainability, deployment readiness, and lessons learned. A full-mark answer reaches across all four modules — process, web, security, responsible A-I — justifies the approach and method against the high-stakes risk, names concrete security and A-I safeguards, and evaluates with criteria, evidence, and reflection. That breadth, tied together by the project process, is the synthesis the exam is built to reward.
