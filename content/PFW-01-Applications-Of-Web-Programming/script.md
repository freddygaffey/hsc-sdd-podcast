NARRATOR: Today's topic. The first dot-point of the whole course says: explore the applications of web programming, including interactive websites and webpages, e-commerce, and progressive web apps. That's the home topic — the different kinds of things web programming builds, and the trade-offs that separate them. By the end you should be able to do three things in an exam. One: name the three application types using the exact syllabus wording. Two: describe the defining characteristics of each. Three — where the real marks live — compare them, especially on security and on how they behave when the network drops. Hear those objectives again, because at five times speed a single mention slips past: name the three types, describe each, compare them on security and on connectivity. Name, describe, compare — exactly the verbs the exam will use.

NARRATOR: Start with the big picture, because there's a trap in the first sentence of this topic. "Web programming" used to mean static websites — a page sits on a server, someone asks for it, the server hands back the same file every time, like handing out a printed flyer. That still exists, but it is not what this dot-point is about. The syllabus is pointing at applications — software that runs in the browser, responds to you, and changes based on who you are and what you do. The shift from static documents to dynamic applications is the entire reason this module exists. So whenever you read "web application", hear "software that runs over the web", not "a web page".

NARRATOR: The syllabus gives exactly three categories, and you must use its words. Type one: the interactive website or webpage. Type two: e-commerce. Type three: the progressive web app, abbreviated P-W-A. Because the exam loves to ask you to "list" or "identify", we need a memory hook now, so you never lose them. Take the first letters: Interactive, E-commerce, P-W-A. That's I, E, P. I-E-P. Remember it as "I Eat Pizza". When the first question is "identify three applications of web programming", the first thing you dump on the page is I-E-P — I Eat Pizza — Interactive, E-commerce, progressive web app. Write it in the margin before anything else. That's three guaranteed marks offloaded onto paper in the first ten seconds. I'll use I-E-P again later, and so will future episodes, so lock it in now.

QUESTION: Hang on — aren't all three of these just websites? What actually makes them different categories rather than the same thing with different content?

NARRATOR: Good question, and it's the heart of the topic. They're different because each one solves a different problem, and that problem dictates the architecture and the trade-offs. An interactive site solves "how do I make the page respond to the user without reloading". E-commerce solves "how do I take someone's money safely and manage stock and orders". A P-W-A solves "how do I make a website behave like an installed app, even with no internet". Different problem, different defining feature, different things that can go wrong. So when the exam asks you to compare them, it's really asking whether you understand why each exists. Take them one at a time.

NARRATOR: Type one, the interactive website. The defining characteristic is dynamic content with client-side interactivity — the page reacts to what you do immediately, without fetching a whole new page from the server. Think of a comment section where your comment appears instantly, or a search box that shows results as you type. You already know how this works: JavaScript in the browser manipulates the page and talks to the server in the background to fetch or send small bits of data. The syllabus calls the front part client-side, or front-end, and the server part back-end — we model that split properly in a later episode on web development systems. For now the exam-relevant point is the characteristic: dynamic, responsive, real-time updates, form processing, no full page reload. If a question says "outline two characteristics of an interactive website", you write: it provides dynamic content that changes in response to user actions, and it processes that interactivity on the client side for immediate feedback without reloading the page. Two clean marks.

NARRATOR: Now point sideways, because interleaving — connecting today's topic to the rest of the course — is the best thing you can do for memory. That client-side interactivity takes input from the user and displays it back, inside other users' browsers. That is exactly the setup for an attack called cross-site scripting, where a malicious user sneaks code into a comment and it runs in everyone else's browser. We don't teach cross-site scripting here — it lives in Secure Software Architecture — but file the connection now: the more interactive a site is, the more places untrusted input can sneak in. Interactivity and security pull against each other. Hold that; it comes back hard in the security module.

NARRATOR: Type two, e-commerce. The syllabus word for any application whose job is buying and selling — product catalogues, shopping carts, checkout, payment, order management. The defining characteristic is not the shopping cart; weak answers say "it has a shopping cart" and stop. The defining characteristic that earns marks is that e-commerce handles money and sensitive personal data, so it lives or dies on security and trust. Think what an online store touches: your name, address, card number, purchase history. The moment money and personal data are involved, you inherit a pile of requirements an interactive blog never had — secure transactions, encrypted connections, safe storage of credentials, validation of every input, and legal compliance with the rules around handling payment cards.

QUESTION: You said security is the defining feature of e-commerce. Pause the player for a second on this one — it's worth it. Why is security the *distinguishing* characteristic of e-commerce, rather than just a nice-to-have like it is for any site? Work out an answer, then play.

NARRATOR: Here's the answer. For an interactive news site, a security failure is embarrassing — someone defaces a comment section. For e-commerce, a security failure is existential — you leak thousands of customers' credit card numbers, face legal penalties, and nobody trusts you with their money again. The stakes are categorically higher because the asset being protected is money and identity. That's why, when the exam asks you to "compare an interactive website with an e-commerce platform in terms of security requirements", the mark-earning move is to talk about consequences and assets, not just features. A strong answer says: an interactive website requires basic input validation to prevent misuse, but an e-commerce platform requires comprehensive security — encryption of transactions in transit, secure handling and storage of payment data, and authentication of users — because it processes financial transactions and sensitive personal information, where a breach causes direct financial and legal harm. Hear how that answer keeps pointing at why. That's the difference between a band four and a band six.

NARRATOR: Interleave again. I just said "encryption of transactions in transit". That's secure transport — HTTPS, TLS, certificates — its own episode coming soon in this module, with the full treatment in Secure Software Architecture. E-commerce is the reason that machinery exists. When we reach the protocols episode and I explain how a connection gets encrypted, remember it was the checkout page that needed it. Concepts stick when they're attached to a reason.

NARRATOR: Type three, the progressive web app — P-W-A. This is the one students get wrong most often, so listen closely. A P-W-A is a web application that behaves like a native installed app: you can install it to your home screen, it works offline, and it can do app-like things like send push notifications. The two characteristics that define it, the ones that earn marks, are that it is installable and offline-capable — those two together. The technology behind offline is a service worker: a script the browser keeps running in the background that intercepts network requests and serves cached content when there's no connection. There's also a small file called the web app manifest, which tells the browser the app's name and icon so it can be installed. You don't need to write a service worker for this dot-point, but you must be able to name it as the mechanism behind offline capability. There's a minimal example in Listing 2 if you want to see its shape later.

NARRATOR: Now the trap, and it's a big one, so I'll say it twice in different words. The trap: students think "progressive web app" means "mobile-friendly" or "responsive website". It does not. A responsive site just resizes nicely on a phone — that's layout. A P-W-A is defined by being installable and working offline through a service worker. Said the exam way: responsive is about appearance across screen sizes; progressive is about capability — install and offline. If a question describes a site that "looks good on mobile" and asks "is this a P-W-A", the answer is no, not on that basis alone. To be a P-W-A it needs the manifest and the service worker giving installability and offline support. Lose this distinction and you lose the marks.

NARRATOR: Here's a memory hook for what's inside a P-W-A, because the exam can ask you to "describe the features that distinguish a P-W-A from a standard website" — and that's a list. The essentials spell MISO, like the soup. M for Manifest, I for Installable, S for Service worker, O for Offline. MISO. Manifest, Installable, Service worker, Offline. We reuse MISO in the dedicated P-W-A episode later in this module, so when you hear it again, it's this. If you can say "a P-W-A has a manifest, is installable, uses a service worker, and works offline" — that's MISO, and that's the distinguishing-features answer.

QUESTION: So how do these three actually show up together in a real exam question, rather than one at a time?

NARRATOR: Usually as a scenario where you have to pick the right type and justify it — the most common shape for this dot-point. Let me walk through three scenarios, with a weak answer and a strong answer for each so you can hear the boundary.

NARRATOR: Scenario one: a local community news website with articles, a comments section, and a calendar of local events. Which type, and why? A weak answer says "interactive, because it's a website people use". That's barely a mark — it justifies nothing. The strong answer: this is best built as an interactive website, because its core need is dynamic content and user engagement — readers post comments and the page updates in response — without any requirement for payment processing or offline use. The strong answer names the defining characteristic, dynamic content and interaction, and explicitly rules out the features of the other two types. Ruling out the alternatives is a marker's favourite move; it proves you understand the categories, not just one of them.

NARRATOR: Scenario two: an online banking platform handling transfers and account management. Weak answer: "a P-W-A, because banks have apps". Wrong reasoning — it grabs at a surface feature. The strong answer treats this as e-commerce, or an e-commerce-style platform, because the defining requirement is secure financial transactions and protection of sensitive personal and financial data, demanding encryption, strong authentication, and comprehensive input validation. The asset is money and identity; the architecture must serve security first. That's the e-commerce reasoning, applied.

NARRATOR: Scenario three: a field data-collection tool for scientists in remote areas with patchy or no internet, who need it to keep working and sync up later. Weak answer: "a website, so they can access it anywhere". That misses the point — anywhere is useless with no signal. The strong answer: this needs a progressive web app, because the defining requirement is offline functionality — the service worker lets it run and store data with no connection and synchronise when connectivity returns — together with installability for an app-like experience on mobile devices in the field. That earns full marks because it names the type and ties it to the exact distinguishing feature the scenario demands: offline capability via a service worker. Connectivity is the deciding factor, and you said so.

NARRATOR: See the pattern across all three? The exam hands you a scenario, and the trick is to find the driving requirement — interaction, money and trust, or working offline — and match it to the type whose defining characteristic answers it. Interaction points to interactive. Money and sensitive data point to e-commerce. Offline and installable point to P-W-A. Drive your justification from the requirement, not from "it's a website". A compact version to memorise: interactive is defined by dynamic content, e-commerce by secure transactions, P-W-A by offline installability. Three types, three defining characteristics. That sentence, plus I-E-P and MISO, is most of what this dot-point can ask you.

Second, a link to hold across the whole year: every one of these three types runs on the same plumbing we cover in the next two episodes — how data moves across the internet, and the protocols and ports that carry it. The application type sits on top; the network carries it. An e-commerce checkout is meaningless without the secure transport underneath, and that's the next stop.

NARRATOR: Name the trade-offs explicitly too, because the syllabus says "trade-offs and constraints" and the exam will ask you to weigh things up, not just list features. Interactive sites are quick to build and flexible, but they depend on the network and on JavaScript in the browser, which raises accessibility and security concerns. E-commerce platforms are proven and scalable, but carry heavy security requirements, high development cost, and regulatory compliance. Progressive web apps give a native-like, cross-platform, offline experience from a single codebase, but the offline caching and data synchronisation are genuinely hard to get right, and support on some platforms — Apple's iOS in particular — is more limited than others. Each type buys you something and charges you something. Naming both sides of that trade is what turns a "describe" answer into an "evaluate" answer.

NARRATOR: Lock in today. Web programming builds applications, not just static pages, and the syllabus names three kinds: interactive websites, e-commerce, and progressive web apps — I-E-P, "I Eat Pizza", the first thing you write in the exam. Each has a defining characteristic: interactive means dynamic, client-side content that updates without reloading; e-commerce means secure financial transactions and protection of sensitive data, where security is the distinguishing requirement because the asset is money and identity; and a progressive web app means installable and offline-capable through a service worker, captured by MISO — Manifest, Installable, Service worker, Offline. The number-one trap: a P-W-A is about install-and-offline capability, not looking good on mobile — responsive is appearance, progressive is capability. And the exam skill is matching a scenario's driving requirement — interaction, money, or offline — to the right type, and justifying it by ruling the others out.

NARRATOR: Now the part that earns marks — exam-style questions. I'll pose each the way the HSC would, you pause and attempt it, then I'll give the model answer with the phrasing that scores. Five questions, easy to hard.

QUESTION: Question one, a quick recall one. Identify the three applications of web programming named in the syllabus. Pause, write them down, then play.

NARRATOR: The three are interactive websites and webpages, e-commerce, and progressive web apps. If you wrote I-E-P and expanded it, that's the mark. Note the exact words — "progressive web app", not "phone app", not "mobile site".

QUESTION: Question two. Outline two characteristics of a progressive web app. Pause and answer.

NARRATOR: Model answer: a progressive web app is installable to the device, like a native app, and it is capable of working offline, using a service worker that caches content and serves it when there is no network connection. Two characteristics, each named precisely — installable and offline-capable — with the service worker named as the mechanism. That's full marks. A weaker answer that says "it works on mobile and is fast" earns little, because those aren't the distinguishing features.

QUESTION: Question three. Distinguish between an interactive website and a progressive web app. Pause, then play.

NARRATOR: Model answer: an interactive website provides dynamic content that responds to user actions in the browser but requires an active network connection to function, whereas a progressive web app additionally is installable and can continue to operate offline through a service worker, behaving like a native application. The key word is "distinguish" — so you must state what each one is *and* draw the contrast: the P-W-A adds installability and offline capability on top of interactivity. Naming the difference, not just describing both, is what the verb demands.

QUESTION: Question four, harder. Compare an interactive website with an e-commerce platform in terms of their security requirements. Pause, give it a real attempt, then play.

NARRATOR: Model answer: an interactive website requires basic security such as input validation and protection against malicious content in user-submitted data, but its assets are relatively low-value. An e-commerce platform requires far more comprehensive security — encryption of transactions in transit using HTTPS, secure storage and handling of payment and personal data, strong user authentication, and regulatory compliance for handling card data — because it processes financial transactions and sensitive personal information, where a breach causes direct financial loss, legal liability, and loss of customer trust. The comparison is driven by the *value of the asset being protected* and the *consequences of failure*, which is exactly what a "compare in terms of security" question rewards.

QUESTION: Question five, the extended one. A research organisation wants a tool for scientists to record observations in remote locations with little or no internet connectivity, syncing data when they return to coverage. Recommend the most appropriate web application type and justify your recommendation. Pause, plan a few-sentence response, then play.

NARRATOR: Model answer: the most appropriate type is a progressive web app. Its defining feature, offline capability through a service worker, directly meets the core requirement that the tool keep functioning and storing data with no network connection, and synchronise that data once connectivity is restored. Its installability also gives scientists a reliable, app-like experience on mobile devices in the field without going through an app store. An ordinary interactive website is unsuitable because it depends on a live connection, and an e-commerce platform's strengths — payment and transaction handling — are irrelevant to this task. The marks here come from three moves: naming the type, tying its specific defining feature to the scenario's driving requirement, and ruling out the other two types with reasons. Make all three moves and you've written a band-six response.
