# Supplementary Materials

Read-along reference for this episode. Nothing here is spoken. The narration teaches client
communication (I-F-N), the ethical issues (PABT), collaboration benefits (V-D-Q), and the
scope-negotiation process. These listings are reference tables.

### Listing 1 — Client communication: I-F-N (text)

```text
I-F-N — communication with the client (the syllabus' three communication issues)

I  INVOLVE + empower    ONGOING partnership, not one-off requirements gathering.
                        Practices: regular DEMONSTRATIONS of working software (= Agile sprint
                        review), CO-DESIGN activities (e.g. user-journey mapping), transparent
                        comms incl. bad news. EMPOWER = give their input REAL influence.

F  enable FEEDBACK      Structured, two-way. Practices: multiple CHANNELS (surveys, interviews,
                        usability sessions, informal); regular CYCLES aligned to iterations;
                        CATEGORISE by type/priority/feasibility; TRACK how it changed decisions.
                        (-> the analyse-and-respond loop C-S-A, SEE 26-02.)

N  NEGOTIATE            Scope, around the Scope–Time–Cost triangle. Process:
                        1. clarify the UNDERLYING NEED (the literal request isn't always the fix)
                        2. assess impact + feasibility (effort, dependencies, risk)
                        3. explore OPTIONS (full / simplified / alternative / defer)
                        4. DOCUMENT what's agreed and why
                        e.g. native app (120 hrs, unaffordable) -> responsive web (40 hrs) meets
                        the core need within budget.

TRAP: communication is NOT soft/optional — projects fail MORE from poor communication than from
technical limits. Answer "how to involve a client" with the I-F-N structure, not "keep talking".
```

### Listing 2 — Ethical issues (PABT) and collaboration benefits (V-D-Q) (text)

```text
PABT — ethical issues in project work (each = a callback to another module)

P  Privacy        What personal data is collected? consent? storage/sharing/retention?
                  = privacy-by-design PER (SSA 16-02); legal weight Ed Picid (SSA 19-03).
A  Accessibility  Usable with assistive tech? = WCAG / POUR (PFW 12-04) as an ETHICAL duty.
B  Bias           Could it discriminate? trained on representative data?
                  = S-H-L-M (SA 22-03); "the algorithm isn't neutral".
T  Transparency   Can users understand decisions + control their data / automated decisions?

ANCHOR SCENARIO (worth recalling): a school "student performance analytics" system flagging
"at-risk" students — collides on Privacy (sensitive data on minors), Bias (biased history ->
unfair flags), Transparency (opaque labelling, self-fulfilling prophecy), and stakeholder
conflict (admin wants data / teachers fear bias / parents want help + privacy / students fear
being labelled).

V-D-Q — benefits of collaboration (SSA 19-01), but with fairness duties:
  V  Various points of view  catch blind spots a solo dev shares with their own code
  D  Delegate by expertise   assign work to whoever has depth
  Q  Quality of the solution  diverse review + expertise -> fewer defects, knowledge-sharing
  Ethical duties added: fair attribution of work; equitable contribution; clear team comms.
```
