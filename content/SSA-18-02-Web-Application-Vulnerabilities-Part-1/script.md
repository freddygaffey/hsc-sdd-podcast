NARRATOR: First — cross-site scripting; C-S-S would be confusing so everyone says X-S-S. We met it in sixteen-oh-one; now it gets its full treatment. Cross-site scripting is when an attacker injects malicious script into a web page that then executes in another user's browser. The root cause is the one you know: untrusted input echoed into a page is interpreted by the browser as script instead of text. Remember the unsanitised search route from Programming for the Web thirteen-oh-one, the one I kept promising to fix? It wrote the user's search term straight into the results page. If the term is a snippet of script, the browser runs it. That route is reflected cross-site scripting — and now we close the loop: the bug I planted three episodes into the web module is exactly this vulnerability.

NARRATOR: Three types to name — hook them as Reflect, Store, D-O-M; R-S-D. Reflected X-S-S: the script comes in on a request, say a U-R-L parameter, and is reflected straight back in the response, so the attacker must trick a victim into clicking a crafted link. Stored, or persistent, X-S-S: the script is saved on the server — in a comment, a profile, a forum post — then served to everyone who views that page. This is the worst: it hits every viewer automatically, no clicking required. And D-O-M-based X-S-S: the injection happens entirely in client-side JavaScript that writes untrusted input into the page without round-tripping through the server. What can an attacker do? Run any JavaScript as the victim — steal their session cookie and hijack their account, log keystrokes, deface the page, redirect them. It's an integrity and confidentiality break: the page and the user's data are no longer trustworthy or private.

NARRATOR: The mitigations — these are the marks. One, the primary fix: output encoding, also called escaping. Before placing user input into a page, convert dangerous characters to their harmless H-T-M-L entities, so a less-than sign displays as text instead of starting a tag — the browser renders the characters instead of executing them. Encode on output, in the right context. Two: a Content Security Policy, a C-S-P — a response header telling the browser which sources of script may run, so even injected inline script is blocked, as a second layer. Three: input validation with an allow-list as a supporting layer, and marking session cookies HttpOnly — from S-A-F-E sessions last chapter — so even if X-S-S runs, it can't read the session cookie. The exam line for "how does X-S-S occur and one mitigation": cross-site scripting occurs when an application includes untrusted input in a web page without encoding it, so the browser executes it as script; it is mitigated primarily by output encoding the data so it is treated as text, reinforced by a Content Security Policy.

QUESTION: Quick check. What is the single root cause of cross-site scripting, and what is the primary fix? Pause, answer, then play.

NARRATOR: The root cause is untrusted input placed into a web page without encoding, so the browser interprets it as executable script rather than text. The primary fix is output encoding — escaping the input on the way into the page so dangerous characters become harmless text. Same root cause as S-Q-L injection — input becoming code — just in the browser instead of the database. That's the mark.

NARRATOR: Second — cross-site request forgery, C-S-R-F, said "see-surf." This is the one students confuse with X-S-S, and they're genuinely different, so listen for the contrast. Cross-site request forgery tricks a logged-in user's browser into sending a request the user never intended, to a site where they're already authenticated. The image: forging someone's signature on a cheque while they're not looking — using their identity to do something they never agreed to. How it works: you're logged into your bank in one tab, session cookie sitting in your browser. You visit a malicious page in another tab. That page contains a hidden form or image that quietly fires a request to your bank's transfer endpoint — "transfer one thousand dollars to the attacker." Because the request goes to your bank, the browser attaches your bank session cookie automatically. The bank sees a valid, authenticated request and processes the transfer. You did nothing but visit a webpage. The attacker never saw your password or cookie — they just got your browser to ride your existing session.

NARRATOR: Here's the distinction the marker tests — the key insight. Cross-site scripting injects malicious script that runs in the victim's browser — the attacker runs code as you. Cross-site request forgery injects no script; it forges a request that rides your already-authenticated session — the attacker acts as you, using your logged-in state, without running code or stealing your credentials. The hook: X-S-S injects a Script; C-S-R-F rides a Session. X-S-S executes code in your browser; C-S-R-F abuses the trust the server already has in your browser. Different attack, different fix.

NARRATOR: The mitigations for cross-site request forgery, the marks. One, the primary fix: an anti-C-S-R-F token, also called a synchroniser token. The server generates a secret, unpredictable token, embeds it in the legitimate form, and requires it back on every state-changing request. The attacker's forged request can't include the token because they can't read it — the same-origin policy stops their page from seeing your bank's page — so the forged request is rejected. Two: the SameSite cookie attribute from S-A-F-E sessions — set SameSite to Strict or Lax and the browser won't send your session cookie on cross-site requests, killing the attack at the browser level. Three: for sensitive actions, re-authenticate or require confirmation. The exam line: cross-site request forgery is prevented primarily with an unpredictable anti-C-S-R-F token the attacker cannot forge, reinforced by SameSite cookies that stop the session cookie being sent on cross-site requests.

QUESTION: This is the one to lock in — pause the player and answer it properly, because it's a guaranteed four-marker. Distinguish cross-site scripting from cross-site request forgery. Pause now, give me how each works and one fix for each, then play.

NARRATOR: Here's the full-mark answer. Cross-site scripting, X-S-S, occurs when an attacker injects malicious script into a web page that then executes in another user's browser, because untrusted input was placed into the page without encoding; it lets the attacker run code as the victim, for example stealing their session. Mitigated by output encoding and a Content Security Policy. Cross-site request forgery, C-S-R-F, injects no script — it tricks a logged-in user's browser into sending a forged request to a site where they're authenticated, exploiting the fact that the browser automatically attaches the session cookie; it lets the attacker perform actions as the victim without running code. Mitigated by anti-C-S-R-F tokens and SameSite cookies. The core distinction: X-S-S injects a script that runs in the victim's browser; C-S-R-F forges a request that rides the victim's session. Naming the inject-script-versus-forge-request distinction and a fix for each is the mark.

NARRATOR: Third — invalid forwarding and redirecting, the open-redirect vulnerability, the I in B-X-I-R. Web apps often redirect users — after login, "go back to where you were," using a destination passed in the U-R-L. The vulnerability: if the app blindly redirects to whatever U-R-L the request supplies, an attacker crafts a link that points to your trusted site but carries a redirect parameter sending the victim onward to a malicious site. The victim sees your trusted domain at the start of the link, trusts it, clicks — and lands on a convincing fake login page that harvests their credentials. That's an open redirect: the app is abused as a trusted springboard to an attacker's site. It's gold for phishing because the link genuinely starts at your real domain, and it undermines trust in your brand. Some treat it as harmless — that's the trap; it's a real phishing enabler. The mitigation, and you'll recognise the principle: never redirect to a raw user-supplied U-R-L. Use an allow-list of permitted destinations, allow only relative paths within your own site, or map a short code to a known destination server-side. Allow-list beats block-list, one more time. The exam line: an open redirect occurs when an application redirects to an unvalidated user-supplied U-R-L, enabling phishing by abusing the trusted domain; it's prevented by validating the destination against an allow-list of permitted internal locations.

NARRATOR: Worked example — the weak-versus-strong contrast on "how would you protect a money-transfer feature from cross-site request forgery." A weak answer says: "make sure the user is logged in before allowing the transfer." That earns nothing — being logged in is exactly the condition C-S-R-F exploits; the forged request comes from an authenticated session. A strong answer says: "require an anti-C-S-R-F token on the transfer request — a secret, unpredictable value the server places in the legitimate form and validates on submission; because the attacker's forged request cannot read or include this token, it is rejected. Reinforce with the session cookie's SameSite attribute set to Strict, so the browser won't send it on cross-site requests, and require re-authentication for high-value transfers." Hear the difference: the weak answer relies on the very thing C-S-R-F abuses; the strong answer names the token, explains why the attacker can't forge it, and layers SameSite. The token plus the why — that's the mark.

NARRATOR: Exam-style finish — five questions, recall up to extended. Pause and attempt each properly before the model answer.

QUESTION: Question one, recall. Describe how cross-site scripting occurs and one mitigation. Pause, then play.

NARRATOR: Model answer: cross-site scripting occurs when an application includes untrusted user input in a web page without encoding it, so the victim's browser interprets and executes the input as script rather than displaying it as text — allowing the attacker to run code in the victim's browser, for example to steal their session. A mitigation is output encoding, which escapes special characters so the input is treated as text and cannot execute; this is reinforced by a Content Security Policy restricting which scripts may run. Naming the unencoded-input-becomes-script mechanism and output encoding is the mark.

QUESTION: Question two, the classic. Distinguish between cross-site scripting and cross-site request forgery. Pause, then play.

NARRATOR: Model answer: cross-site scripting injects malicious script into a page that executes in a victim's browser, exploiting unencoded input, and lets the attacker run code as the victim. Cross-site request forgery injects no script; it tricks a logged-in user's browser into sending a forged, unintended request to a site where they are authenticated, exploiting the automatic sending of session cookies, and lets the attacker perform actions as the victim. X-S-S injects a script that runs in the browser; C-S-R-F forges a request that rides the victim's session. Naming the inject-script-versus-forge-request distinction is the mark.

QUESTION: Question three. Explain how an anti-C-S-R-F token prevents cross-site request forgery. Pause, then play.

NARRATOR: Model answer: the server generates a secret, unpredictable token, embeds it in each legitimate form, and requires it back with every state-changing request, validating it server-side. A cross-site request forgery attack is launched from a different origin and, due to the same-origin policy, the attacker's page cannot read the token, so the forged request cannot include the correct value and is rejected. This distinguishes genuine requests, which carry the token, from forged ones, which cannot. Naming the unpredictable token, that the attacker can't read it, and server-side validation is the mark.

QUESTION: Question four. Describe the open-redirect vulnerability and how to prevent it. Pause, then play.

NARRATOR: Model answer: an open-redirect, or invalid forwarding and redirecting, vulnerability occurs when an application redirects users to a destination taken from unvalidated user input, such as a U-R-L parameter. An attacker crafts a link that begins at the trusted site but redirects the victim onward to a malicious site, enabling phishing because the victim trusts the visible domain. It is prevented by never redirecting to a raw user-supplied U-R-L — instead validating the destination against an allow-list of permitted internal locations, or allowing only relative paths within the application. Naming the unvalidated-redirect-enables-phishing mechanism and the allow-list fix is the mark.

QUESTION: Question five, the extended one. A web application has a comment feature, a money-transfer feature, and a post-login redirect. Identify a vulnerability in each and describe how to secure it. Pause, plan a structured response, then play.

NARRATOR: Model answer: the comment feature is vulnerable to stored cross-site scripting — a malicious script saved in a comment would execute in every viewer's browser; secure it by output-encoding comments on display so they're treated as text, reinforced with a Content Security Policy and HttpOnly session cookies. The money-transfer feature is vulnerable to cross-site request forgery — a forged request from a logged-in user's browser could trigger an unintended transfer; secure it with an anti-C-S-R-F token that the attacker cannot forge, plus SameSite cookies and re-authentication for high-value transfers. The post-login redirect is vulnerable to an open redirect — an attacker could redirect users to a phishing site via the trusted domain; secure it by validating the redirect destination against an allow-list of internal locations rather than trusting the supplied U-R-L. A strong response names the specific vulnerability for each feature, explains the mechanism, and gives the targeted fix. That's full marks.
