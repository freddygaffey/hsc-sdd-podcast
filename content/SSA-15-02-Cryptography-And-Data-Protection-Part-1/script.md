NARRATOR: Start with the core distinction, because everything hangs off it. Two cryptographic operations, and they are not the same thing. Encryption is two-way. It transforms readable data — plaintext — into scrambled data — ciphertext — using a key, and crucially it is reversible: with the right key you decrypt the ciphertext back into the exact original plaintext. Use encryption when you need to protect data but also get it back later — files on disk, data crossing the network. Encryption serves confidentiality, the first property from last episode. Hashing is one-way. It runs data through a function that produces a fixed-size fingerprint — a hash or digest — and it is not reversible. There is no "un-hash"; you cannot take the fingerprint and reconstruct the original, by design. Use hashing when you need to verify or fingerprint data, not read it back. Hashing serves integrity. Say the line with me, and say it exactly: hashing is one-way, encryption is two-way. Encrypt when you need it back; hash when you only need to check it.

NARRATOR: The properties of a hash, because they come up as "describe" marks and explain everything that follows. A good cryptographic hash is one-way — irreversible. It's deterministic — the same input always produces the same hash, which is what makes verification possible. It's fixed-size — feed it one word or an entire novel, the fingerprint is the same length. And it has the avalanche effect — change the input by a single character and the hash changes completely, not slightly. Think of a fingerprint at a crime scene: you can match a print to a person, but you can't rebuild the person from the print — that's one-way; the same finger always leaves the same print — that's deterministic. Hold those four: one-way, deterministic, fixed-size, avalanche.

NARRATOR: If you need the original back, you encrypt — it's the only one of the two that reverses. If you only ever need to check a match, never recover the original, you hash — and you should, because one-way is safer when you don't need reversibility. That second case — "check a match, never recover" — describes passwords exactly. Which is where we're going now.

NARRATOR: Password storage. This is the marquee application and a near-certain exam question, so we'll build it the way the marker wants it, mistake by mistake. Start with the worst design: storing passwords in plaintext, the literal characters, in the database. The moment that database is breached — and chapter eighteen is full of ways it gets breached — every password is exposed, and because people reuse passwords, you've handed the attacker their email and banking logins too. Plaintext passwords are an automatic fail, in the exam and in real life.

NARRATOR: Now the tempting wrong answer, the one that costs students the mark: "so encrypt the passwords." It sounds secure — it's cryptography, after all. But encryption is two-way. There's a key that turns the ciphertext back into the real passwords, and that key has to live somewhere on your servers. So the attacker who breaches your database just needs the key too, and in a real breach they very often get both — at which point every password decrypts back to plaintext. You've made the passwords recoverable, and recoverable is exactly what you don't want. Here's the band-six insight: you never actually need to read a password back. When a user logs in, you don't need to know what their password is — only whether the password they just typed matches the one they set. That's a verification problem, not a storage-and-retrieval problem. And verification without recovery is precisely what a one-way function gives you. So you hash.

QUESTION: This is the one to lock in — pause the player and actually answer it, because it's a guaranteed four-marker. Explain why passwords are hashed rather than encrypted. Give me the reason about reversibility and the reason about what you actually need. Pause now, work it out, then play.

NARRATOR: Here's the full-mark answer. Passwords are hashed, not encrypted, because hashing is one-way and encryption is two-way. You never need to recover the original password — at login you only verify that the entered password matches, so you hash it and compare to the stored hash. Because hashing is irreversible, a breach of the database does not reveal the passwords themselves. Encryption is reversible: it requires a key that can turn the stored values back into the real passwords, so if that key is also compromised — which commonly happens in a breach — every password is exposed. Hashing removes that risk by never making the original recoverable in the first place. Notice the shape: name the one-way-versus-two-way distinction, then the "verify, not recover" insight, then the key-as-a-liability point. That's the answer.

NARRATOR: But plain hashing on its own is not enough, and the rest of the episode is the two fixes that turn a naive hash into safe password storage. Fix number one. Because hashing is deterministic, the same password always produces the same hash. So if a thousand users all chose "password one-two-three", their stored hashes are all identical — the attacker instantly sees who shares a password. Worse, attackers precompute giant lookup tables: take every common password, hash it in advance, store the password-to-hash mapping. These are rainbow tables. With one, an attacker doesn't crack your hashes, they just look them up — instant. The fix is a salt. A salt is a random, unique value generated for each user and combined with their password before hashing. Because every user gets a different salt, the same password now produces a different hash for every user, so identical passwords no longer look identical — and the precomputed rainbow table is useless, because it was built without your salts. The salt isn't secret; you store it right next to the hash. Its whole job is to be unique, forcing the attacker to crack every password individually instead of all at once. The memory hook: salt stops the rainbow. And the exam-precise role of a salt: a salt is a unique random value added to each password before hashing, to ensure identical passwords produce different hashes and to defeat precomputed rainbow-table attacks.

NARRATOR: Fix number two. Even with salts, there's a problem: ordinary hash functions like S-H-A two-fifty-six are built to be fast — designed to fingerprint huge files quickly. But fast is bad for passwords, because an attacker with your salted hashes can still guess: take a guess, salt it, hash it, compare — and a fast hash lets them try billions of guesses per second. The fix is to make hashing deliberately slow. This is key stretching, done with a password-hashing algorithm purpose-built to be slow and expensive — bcrypt, said "bee-crypt"; scrypt, "ess-crypt"; Argon-two; or P-B-K-D-F-two run with a high iteration count, repeating the hashing tens of thousands of times. Slowness you barely notice on one legitimate login — a fraction of a second — becomes ruinous for an attacker making billions of guesses. So the rule: never store passwords with a plain fast hash; use a slow, purpose-built password hash.

NARRATOR: Put the whole recipe together with a memory hook, because "how do you store a password securely" is itself an exam question, and it's a three-part list. Hash it, salt it, slow it. Hash it — store a one-way hash, never the password and never an encrypted, recoverable form. Salt it — add a unique random salt per user so identical passwords differ and rainbow tables fail. Slow it — use a deliberately slow algorithm like bcrypt or Argon-two so brute-forcing is too expensive. Hash it, salt it, slow it. And note the order, because the exam asks it: password, then salt, then hash — you combine the password with the salt first, and hash the result. Password, salt, hash.

NARRATOR: Two refinements so your answer sounds expert. First, when a user logs in, you verify by taking the entered password, applying the same salt, hashing with the same slow algorithm, and comparing to the stored hash — using a constant-time comparison, one that always takes the same time regardless of how much matches, so an attacker can't learn the hash character-by-character from timing differences. That's a side-channel concern we'll return to in episode seventeen-oh-two. Second, the golden rule of all cryptography: never roll your own. Don't invent your own hashing or encryption — use established, audited libraries. Your language's standard crypto library already does salting and slow hashing correctly. This is "secure default, easy right path" in action — the library makes the secure way the easy way, so there's no excuse to do it by hand and get it wrong.

QUESTION: Question one, recall. What is the fundamental difference between hashing and encryption? Pause, then play.

NARRATOR: Model answer: hashing is a one-way function — it produces a fixed-size fingerprint of data that cannot be reversed to recover the original. Encryption is a two-way function — it transforms data into ciphertext that can be reversed back to the original plaintext using the correct key. You hash when you only need to verify or fingerprint data; you encrypt when you need to protect data but recover it later. Stating one-way versus two-way, and when each is used, is the mark.

QUESTION: Question two, the guaranteed one. Explain why hashing, rather than encryption, is used to store passwords. Pause, then play.

NARRATOR: Model answer: passwords are hashed because hashing is irreversible, so a breach of the password store does not reveal the actual passwords. The system never needs to recover a password — at login it only verifies a match, by hashing the entered password and comparing to the stored hash. Encryption would be reversible and would require a key that can turn the stored values back into real passwords; if that key were also compromised, every password would be exposed. Hashing avoids this by never storing anything recoverable. Naming irreversibility, the verify-not-recover point, and the key-as-a-liability risk is the mark.

QUESTION: Question three. Describe the role of a salt in password storage. Pause, then play.

NARRATOR: Model answer: a salt is a unique, random value generated per user and combined with the password before it is hashed. Because each user has a different salt, identical passwords produce different hashes, so shared passwords are not visible and attackers cannot use precomputed rainbow tables — built without the salts — forcing them to crack each password individually. The salt is stored alongside the hash and need not be secret. Naming "unique random value, defeats rainbow tables, makes identical passwords differ" is the mark.

QUESTION: Question four, application. A system hashes passwords with a single fast hash and no salt. Identify two weaknesses and how to fix each. Pause, then play.

NARRATOR: Model answer: first weakness — no salt means identical passwords produce identical hashes, leaving the system vulnerable to precomputed rainbow-table attacks; fix it by adding a unique random salt per user before hashing. Second weakness — a fast hash lets an attacker try billions of guesses per second against the stolen hashes; fix it with key stretching, a deliberately slow password-hashing algorithm such as bcrypt or Argon-two, or P-B-K-D-F-two with a high iteration count. Identifying the two distinct weaknesses — no salt and too-fast — and giving the matching fix for each is the mark.

QUESTION: Question five, the extended one. Explain how cryptography contributes to a security-by-design approach when storing user credentials. Pause, plan a structured response, then play.

NARRATOR: Model answer: cryptography lets credentials be protected by design rather than as an afterthought. Passwords are never stored in plaintext or in a recoverable encrypted form; instead each is stored as a one-way hash, so the system can verify a login without ever revealing the password, and a database breach does not expose credentials. Each password is combined with a unique random salt before hashing, ensuring identical passwords differ and defeating precomputed rainbow tables, and the hash uses a deliberately slow, purpose-built algorithm such as bcrypt or Argon-two to make brute-force guessing infeasible. These protections are built in using established, audited libraries rather than custom code, making the secure approach the default and the easy path — the essence of security-by-design. A strong response names hashing over encryption, salting, slow key stretching, and trusted libraries, and explicitly ties them to building security in from the start. That's full marks.
