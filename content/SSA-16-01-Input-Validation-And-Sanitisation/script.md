Three named practices — the home topic. The outcome is S-E-twelve-oh-seven, secure code. High-value, in the exam and in the real world.

NARRATOR: The governing idea, before any technique: never trust input. Treat every piece of data crossing into your program from outside — form fields, U-R-L parameters, uploaded files, A-P-I requests, even values from another system — as potentially hostile until you've checked it. The HSC calls the discipline "defensive data input handling," and that word defensive is the mindset: assume the input is an attack and make the code survive it anyway. Everything else today is three ways of being defensive.

NARRATOR: Objectives up front, so your attention knows what to catch. By the end you'll be able to define validation, sanitisation and error handling and distinguish them — that's a guaranteed "explain the difference" question; explain how a parameterised query stops S-Q-L injection; and explain why allow-list beats block-list. And here's the hook for the set, because "name the defensive input handling practices" is itself an exam question: the three are Validate, Sanitise, handle Errors — V-S-E. Say it: V-S-E. That's the spine of the episode.

NARRATOR: First practice — validation, the V in V-S-E. Validation means checking that input matches the rules for what it's supposed to be, and rejecting it outright if it doesn't. Is the email actually shaped like an email? Is the age between zero and a hundred-and-fifty? Is the username only letters, digits and underscores, and between three and fifty characters? Validation is a gate: input either conforms or it's turned away. The key decision is which philosophy you use, and this is examinable. Two approaches. A block-list — also called a blacklist or deny-list — defines what's forbidden and allows everything else: "reject anything containing the word DROP, or angle brackets, or a semicolon." An allow-list — whitelist or accept-list — defines what's permitted and rejects everything else: "a username may contain only letters, digits, and underscores; reject anything else."

NARRATOR: Here's the rule the marker wants, with the reasoning: allow-list beats block-list. Why? A block-list is a losing game — you have to think of every bad input in advance, and attackers are endlessly creative. They'll evade it with a comment in the middle of a keyword, with mixed case, with U-R-L encoding, with a tag you didn't think of. You will always miss one. An allow-list flips the burden: you only define what good looks like, a small knowable set, and everything else — including attacks you've never heard of — is rejected by default. The hook: allow-list beats block-list — define what's good, not what's bad. Block-listing is trying to name every villain; allow-listing is a guest list. Use a block-list only as a secondary filter, never as your primary defence.

QUESTION: Quick check before sanitisation. You're validating a field for an Australian postcode. Describe the allow-list rule, and say why that's safer than a block-list. Pause, answer, then play.

NARRATOR: The allow-list rule: a postcode must be exactly four digits, zero to nine, and nothing else — accept that, reject everything else. It's safer than a block-list because you don't have to anticipate every malicious string someone might submit; by specifying the exact, narrow form a postcode can take, anything weird — injection payloads, scripts, oversized input — fails automatically, including attacks you didn't predict. Defining what's good is finite; defining what's bad is infinite. That's the whole argument, and it earns the mark.

NARRATOR: Second practice — sanitisation, the S in V-S-E. Here's the distinction students lose marks on, so we'll nail it. Validation rejects bad input. Sanitisation transforms input so it's safe to use, instead of rejecting it. Sometimes you can't just reject — a blog comment can legitimately contain an angle bracket or an apostrophe, and you don't want to refuse the comment, you want to neutralise the dangerous parts and keep the rest. Sanitisation cleans, escapes, or encodes the input so that even if it contains something that would otherwise be read as code, it's treated as harmless data. The classic example is output encoding for the web: turning a less-than sign into its harmless H-T-M-L entity so a browser displays it as text instead of running it as a tag. Validation is at the door — does this belong? Sanitisation is at the counter — make this safe to handle. The exam line: validation checks input against rules and rejects what doesn't conform; sanitisation modifies or escapes input to render it safe for use. One rejects, one cleans.

NARRATOR: Now apply validation and sanitisation to the two vulnerabilities I promised to fix, because this is where the marks live. First, S-Q-L injection — the login query from thirteen-oh-two. The vulnerable code glued the username straight into the query string: select star from users where username equals, then the raw input, in quotes. If an attacker types a username of "apostrophe, or one equals one, dash dash" — their input closes your quote, adds "or one equals one" which is always true, and comments out the rest. The query now returns every row, and they're logged in as the first user, often the admin. The input stopped being data and became S-Q-L. That's injection in one sentence: untrusted input is interpreted as code instead of data.

NARRATOR: The fix — the single most important technique in the episode — is a parameterised query, also called a prepared statement. Instead of building the query by string concatenation, you write it with a placeholder — a question mark or a named parameter — where the value goes, and pass the actual value separately, alongside the query. The crucial part: the database receives the query structure and the data through two different channels. It compiles the query plan first, with the placeholder, then slots your value in as pure data that can never change the query's meaning. So even if the value is "apostrophe, or one equals one," it's treated as a literal string to search for — a username with weird characters, which simply matches nothing — not as S-Q-L to execute. The hook: parameterise — pass the value separately so it can never become code. And the exam-precise answer to "how do parameterised queries prevent S-Q-L injection": they separate the query from the data, so user input is always treated as a value and never as part of the S-Q-L command, removing the attacker's ability to alter the query's structure. The hardened login is in Listing one, and the parameterised query in Listing two — but you don't need to see them; separate the data from the command is the whole idea.

NARRATOR: Second, cross-site scripting — the unsanitised search echo from thirteen-oh-one. The vulnerable route wrote whatever the user searched for straight into the H-T-M-L of the results page. If an attacker searches for a snippet of script — angle bracket, script, and so on — the browser doesn't display it, it runs it, in the victim's browser, with the victim's session. That's cross-site scripting, X-S-S: untrusted input echoed into a page is interpreted as script instead of text. It's the exact same disease as S-Q-L injection — input becoming code — just in a different interpreter: the browser instead of the database. The fix is sanitisation, specifically output encoding: before you put user input into the page, escape it, so the less-than sign becomes its harmless entity and the browser shows the characters instead of executing them. A second layer is a Content Security Policy, a C-S-P, a browser instruction that restricts which scripts are allowed to run at all. We go deep on X-S-S in episode eighteen-oh-two — for now the link is the lesson: injection and X-S-S are the same root cause, untrusted input crossing a boundary and being run as code, and the defence is the same family, V-S-E.

NARRATOR: Third practice — error handling, the E in V-S-E, and the one students forget is even a security control. When input fails your checks, or something goes wrong, how you respond is itself a security decision. The danger is information disclosure: a raw error message or a stack trace dumped to the user can leak your database structure, file paths, library versions, even fragments of a query — a free map for an attacker. So the rule is two-faced. Log the detailed error internally, where your developers and monitoring can see it — that's accountability from fifteen-oh-one, the audit trail. But return only a generic, safe message to the user: "invalid username or password," "a temporary error occurred, please try again." Never show the real exception outside. And notice a subtlety in that login message: you say "invalid username or password," not "no such user" versus "wrong password" — because telling an attacker which usernames exist helps them. The hook: log it private, show it generic. Detailed inside, vague outside. And "fail securely" — when in doubt, reject and deny, don't fall through to allowing the action.

QUESTION: This is the one to lock in — pause the player and actually answer it, because it's a guaranteed "explain the difference" marker. Distinguish between input validation and sanitisation, and give an example of each. Pause now, work it out, then play.

NARRATOR: Here's the full-mark answer. Input validation is checking incoming data against a set of rules for what is acceptable and rejecting it if it does not conform — for example, requiring an email field to match the format of an email address, and refusing the submission if it doesn't. Sanitisation is transforming input so it is safe to use, rather than rejecting it — for example, H-T-M-L-escaping a comment before it's displayed, so any angle brackets are shown as text and cannot run as script. The key distinction: validation decides whether to accept or reject input by testing it against rules; sanitisation modifies or neutralises input to make it safe. They're complementary — you validate first to throw out clearly bad input, and sanitise what you keep before you use it. Naming "validate rejects, sanitise cleans," with an example of each, is the mark.

NARRATOR: Worked example — the weak-versus-strong contrast on "describe how to prevent S-Q-L injection," because hearing the boundary is worth more than another definition. A weak answer says: "you stop S-Q-L injection by checking the input for dangerous words like DROP and DELETE and blocking them." That earns very little — it's a block-list, it's evadable, and it shows no grasp of the mechanism. A strong answer says: "S-Q-L injection is prevented primarily by using parameterised queries, which send the query structure and the user-supplied data to the database separately, so the input is always treated as a literal value and can never alter the query's commands; this is reinforced by validating input against an allow-list of the expected format and applying least privilege to the database account." Hear the difference: the weak answer block-lists keywords; the strong answer names the parameterised query, explains why it works — data separated from command — and layers validation and least privilege on top. Mechanism, why it works, and defence in depth — that's the band-six answer.

NARRATOR: One more trap, the big one, because it's the most common real failure: client-side-only validation. You might add checks in JavaScript so the user gets instant feedback — good for usability. But those checks run in the browser, which the user fully controls and can disable, bypass, or skip entirely by sending a request directly. So validation in the browser is never a security control; you must re-validate on the server, every time, no exceptions. Client-side for friendliness, server-side for safety. QUESTION: Question one, recall. Name the three defensive data input handling practices, and state the purpose of each. Pause, then play.

NARRATOR: Model answer: the three are validation, sanitisation, and error handling — V-S-E. Validation checks incoming data against rules for what is acceptable and rejects non-conforming input. Sanitisation transforms or escapes input so it is safe to use, neutralising anything dangerous rather than rejecting it. Error handling responds safely to bad input or failures by logging details internally while returning only generic messages externally, so no sensitive information is disclosed. Naming all three and giving each one's purpose is the mark.

QUESTION: Question two. Explain how parameterised queries prevent S-Q-L injection. Pause, then play.

NARRATOR: Model answer: S-Q-L injection occurs when user input is concatenated directly into a S-Q-L command, allowing the input to alter the query's structure — for example, adding an always-true condition to bypass a login. A parameterised query, or prepared statement, defines the query with placeholders and passes the user-supplied values to the database separately from the query text. Because the database compiles the query structure first and then treats the supplied values purely as data, the input can never be interpreted as part of the S-Q-L command, so injection is prevented. Naming the separation of query from data, and that input is always treated as a value, is the mark.

QUESTION: Question three. Distinguish between an allow-list and a block-list approach to validation, and state which is preferred and why. Pause, then play.

NARRATOR: Model answer: an allow-list defines exactly what input is permitted and rejects everything else; a block-list defines what input is forbidden and allows everything else. The allow-list is preferred because it is secure by default — you only have to specify the finite set of acceptable inputs, so any unexpected or malicious input, including attacks not anticipated by the developer, is rejected automatically. A block-list is inherently incomplete, because it is impossible to anticipate every malicious variation, and attackers can evade it through encoding or obfuscation. Naming the define-good-versus-define-bad distinction and the "secure by default, can't anticipate every attack" reasoning is the mark.

QUESTION: Question four, application. A web form's input is validated only with JavaScript in the browser. Explain why this is insufficient and how to fix it. Pause, then play.

NARRATOR: Model answer: client-side JavaScript validation is insufficient because it runs in the browser, which the user fully controls — it can be disabled, modified, or bypassed entirely by sending a request directly to the server, so an attacker can submit any input regardless of the client-side checks. The fix is to re-validate all input on the server, treating server-side validation as the authoritative security control; keep client-side validation only as a usability aid for immediate feedback. Naming that the client is untrusted and can be bypassed, and that validation must occur server-side, is the mark.

QUESTION: Question five, the extended one. A registration handler takes a username, email and comment from a form and stores them, then later displays the comment to other users. Explain how you would apply defensive input handling to make this secure. Pause, plan a structured response, then play.

NARRATOR: Model answer: apply V-S-E. First, validate every field against an allow-list on the server: the username must match an allowed pattern and length, the email must match the expected email format, and over-length input is rejected — non-conforming input is refused outright, and the client cannot be trusted to do this. Second, when storing to the database, use parameterised queries so the values are passed separately from the command and cannot become S-Q-L injection. Third, sanitise on the way out: when the comment is later displayed to other users, H-T-M-L-escape it so any markup is shown as text and cannot execute as a cross-site scripting payload, optionally reinforced with a Content Security Policy. Finally, handle errors safely — log full details internally for accountability, but return only a generic message to the user, and fail securely by rejecting on any doubt. A strong response names validation with an allow-list, parameterised queries for storage, output encoding for display, and safe error handling, all enforced server-side, and ties them to preventing injection and cross-site scripting. That's full marks.

NARRATOR: That's defensive input handling. Never trust input; the three practices are V-S-E — validate with an allow-list, sanitise by cleaning and escaping, handle errors by logging private and showing generic. Parameterised queries and output encoding kill injection and cross-site scripting because both are the same root cause: input becoming code. And it all happens server-side.
