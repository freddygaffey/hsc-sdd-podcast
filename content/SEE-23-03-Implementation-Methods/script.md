NARRATOR: Recap first. Last episode, ideation and modelling tools. Ideation was B-M-S — Brainstorm to generate ideas without judging, Mind-map to organise them around a central concept, Storyboard to walk through the user's journey and catch usability problems early. Then the modelling tools: the data dictionary, which documents every data element and its type before you build; algorithm design, expressing logic in NESA pseudocode first because logic errors are cheap on paper; then code generation, testing and debugging, installation, maintenance. The verb was justify, and the big trap was jumping to code before modelling — the same shift-left "fix it early, it's cheap" idea from Secure Software Architecture.

NARRATOR: And before that, requirements and feasibility. FUN versus PERF — functional is what it does, non-functional is how well, and every requirement must be measurable and testable. Feasibility is TECO — Technical, Economic, Cost-and-schedule, Operational. And boundaries stop scope creep. I'm re-firing these because today they come together: we've defined what to build, modelled it, and now we decide how to switch it on for real users. That's implementation methods. Today's outcome is S-E-twelve-oh-six again — justify the selection and use of tools and methods — and the verb, as always in this module, is justify.

NARRATOR: The framing. You've built the system. It works in testing. There's a live, current way of doing things — an old system, or a manual process — and you have to switch over to the new one. How you make that switch is the implementation method, sometimes called the changeover or cut-over strategy. It's a high-stakes decision, because changeover is the riskiest moment in a project's life. Get the method wrong and you can lose data, lose users, or lose the organisation's confidence in one bad day. The syllabus names four methods, with a clean hook — DiP-PP. Direct, Phased, Parallel, Pilot. Say it as "dip, pee-pee." Write DiP-PP in the margin and you've got all four. I'll teach each with its risk profile, because the risk profile is what the exam actually tests.

NARRATOR: Objectives up front. By the end you'll define all four methods, state the risk and cost trade-off of each, compare any two, and — the extended-response skill — recommend and justify a method for a given scenario. The classic questions are "compare direct and parallel implementation" for around four marks, and "recommend and justify an implementation method for this scenario," an extended response. This episode is built to nail those.

NARRATOR: First, the D in DiP-PP: Direct. Direct implementation, also called the "big bang," switches off the old system and switches on the new one in a single changeover — all at once, everyone at the same time. One cutover date: old system stops, new system starts. The advantage is speed and cost — it's the fastest way to get the benefits, and you never pay to run two systems at once. The disadvantage: there is no fallback. If the new system fails on day one, there's no old system to fall back to, and everyone is disrupted at once. So direct is cheap and fast but high-risk. The one-liner to memorise: Direct is cheap and fast, but if it breaks there's no safety net. It suits simple systems where speed matters and failure is tolerable — a school swapping email providers over a weekend.

NARRATOR: Second, the first P: Phased. Phased implementation introduces the new system gradually, in stages — typically by functionality. You roll out part one, learn from it, then part two, and so on until the whole system is live. For a student information system: phase one, core student records; phase two, enrolment; phase three, the gradebook; phase four, reporting. The advantage is manageable complexity — each piece is a smaller change, and you learn from each phase before the next. The disadvantage is a longer timeline, the coordination cost of the staged rollout, plus handling a system that's half-old, half-new during the transition. Phased is medium-risk: safer than direct, because a problem in one phase only affects that phase, but slower. The one-liner: Phased rolls out a piece at a time, learning as it goes.

NARRATOR: Third, the second P: Parallel. Parallel implementation runs the old system and the new system at the same time, both processing the same data, for a period — until you're confident the new one is correct, then you decommission the old. The big advantage is safety: the old system is a live safety net throughout, and you can compare the new system's output against the known-good old output to verify it before committing. The disadvantage is cost — you run and staff two complete systems at once, double the work, for the whole overlap. So parallel is the safest method but the most expensive. The one-liner, and pair it with Direct as the exam loves to: Direct is cheap and risky; Parallel is safe and costly. The two ends of the trade-off.

NARRATOR: Fourth, the last P: Pilot. Pilot implementation rolls the full new system out to a small, representative group of users first — one department, one class — while everyone else stays on the old system. You test it in the real world with real users and real data, gather feedback, fix issues, and only then roll out to everyone. The advantage: problems surface in a controlled environment and only affect the small pilot group, not the whole organisation, and you get genuine real-world feedback before full commitment. The disadvantage: full deployment is delayed, and you briefly run a two-tier experience — pilot group on the new, everyone else on the old. Pilot is low-risk. The one-liner: Pilot tries the whole thing on a small group first. The distinction students miss: phased rolls out part of the system to everyone, gradually; pilot rolls out the whole system to part of the users. Phased splits by feature; pilot splits by user group.

QUESTION: Pause-the-player, the highest-value retrieval here. Without looking, write down all four methods from DiP-PP, and for each one give its risk level and its one-line trade-off. Pause now, dump all four, then play to check.

NARRATOR: Here they are. Direct — high risk — cheap and fast, but no fallback if it fails. Phased — medium risk — roll out a piece at a time, learning as you go, but slower. Parallel — low risk — run both systems at once as a safety net and compare outputs, but expensive. Pilot — low risk — try the whole system on a small representative group first, but delays full rollout. Produce DiP-PP with a risk level and a trade-off for each, and you can answer any comparison or recommendation question on this dot-point. The order from riskiest to safest is Direct, Phased, Parallel, Pilot — direct is the gamble, pilot is the careful one.

NARRATOR: Now the distinction the exam exploits — phased versus parallel, and phased versus pilot, because these three get muddled. Parallel runs two systems at once; phased runs the new system in stages but doesn't keep the old one fully running alongside. Phased splits by feature and gives each feature to everyone; pilot keeps the system whole and gives it to a subset of users. If you remember one thing here: parallel runs both at once for safety; phased stages the build-out; pilot uses a small test group. Don't write "phased" when you mean "parallel" — that's a guaranteed lost mark.

NARRATOR: Now the stories, because these methods are best remembered through real disasters — and this module cashes both in properly later. First, Knight Capital, two thousand twelve. Knight was a major American trading firm. They deployed new trading software as a direct, big-bang cutover, and it went wrong in the most instructive way. Old, dormant code that should have been removed was accidentally reactivated on some servers but not others, so the system started firing off millions of unintended trades. Because it was a direct changeover with no staged rollout and no working kill-switch or rollback, there was no safety net — nothing to fall back to. In forty-five minutes, Knight Capital lost about four hundred and forty million dollars and was effectively destroyed as a company. The lesson in this episode's exact terms: a direct, big-bang deployment of a high-stakes system with no rollback path is the maximum-risk choice — and Knight took it. A pilot or parallel approach, or even a tested rollback, would likely have caught it first. We'll tell the full story at version control and deployment, twenty-five-oh-four, because the deeper failure was deployment hygiene.

NARRATOR: Second story, the one from episode one, now complete: Denver airport's automated baggage system. Last time I told you the requirements were unbounded and the project wasn't feasible. The implementation made it worse: a direct, big-bang cut-over of an enormous, complex, never-fully-tested automated system across the entire airport at once. No pilot on one concourse, no parallel run alongside human baggage handlers, no phased rollout terminal by terminal — just switch the whole thing on. Bags were destroyed and lost; the system never worked reliably; the airport opened sixteen months late at a cost of hundreds of millions. The lesson: for a system that large, complex, and critical, direct cut-over was exactly the wrong method. A pilot on a single concourse, or a parallel run with the manual system as backup, was what the risk profile demanded. Denver is the textbook case of choosing the riskiest method for a system that could least afford it.

NARRATOR: Now the worked example, because "recommend and justify" is the money question. Two scenarios. One: a hospital replacing its patient records system. Two: a small startup launching a new version of its consumer app. Same question for both — which method, and why? The hospital is a critical, life-safety system where data loss or downtime could literally kill someone, and it must be available twenty-four-seven. Risk tolerance is essentially zero. So you recommend parallel — run the old and new records systems together so there's always a working fallback and you can verify the new system's output against the old — possibly preceded by a pilot in one low-risk department like outpatients. It's expensive, but for a hospital safety overwhelmingly outweighs cost. The justification names the risk profile — critical, zero tolerance — and matches it to the safest methods. The startup app: speed to market matters, the system is relatively simple, the cost of a brief failure is low, and they can't afford to run two systems. So you recommend direct — fast, cheap, and the risk is tolerable. Again, match method to context: low stakes, tight budget, speed matters, so the cheap fast method is right.

NARRATOR: Now the weak-versus-strong contrast, because this is where extended-response marks are won and lost. Weak answer to "recommend a method for the hospital": "they should use parallel because it's the safest." That states the method but doesn't justify it against the scenario — band four. Strong answer: "parallel implementation is recommended because a patient records system is safety-critical and must remain continuously available; running the old and new systems simultaneously provides a guaranteed fallback if the new system fails and allows staff to verify the new system's output against the known-correct old system before committing. The higher cost of operating two systems is justified by the catastrophic consequences of data loss or downtime in a healthcare setting." Hear the difference — the strong answer names the method, links its specific advantage to the specific risk in the scenario, and weighs the cost against the stakes. Method, plus advantage, plus the scenario's risk, plus the trade-off weighed. That's the full-mark shape.

NARRATOR: Consolidate the interleaving. Backward: this topic depends on feasibility from episode one — the risk you manage at changeover is the risk you assessed under TECO. Backward to Secure Software Architecture: availability is a CIA property, so choosing parallel for a hospital is an availability decision; and the disaster-recovery and rollback thinking from eighteen-oh-one — an untested backup is a guess — is exactly what Knight Capital lacked. Forward: implementation method links straight to data backup and version control in twenty-five-oh-four, where rollback is the safety net direct cut-over throws away; and to testing in twenty-six-oh-one, because a parallel or pilot run is itself live testing before full commitment. The thread: how you change over is a risk-management decision, connected to feasibility behind it and to backup, rollback, and testing ahead of it.

NARRATOR: Lock in today. Four implementation methods — DiP-PP: Direct, Phased, Parallel, Pilot. Direct is cheap, fast, high-risk, no fallback — the big bang. Phased rolls out feature by feature to everyone, medium-risk. Parallel runs both systems at once for safety and verification, low-risk but expensive. Pilot tries the whole system on a small representative group first, low-risk but delays full rollout. The exam pairing: Direct is cheap and risky, Parallel is safe and costly. The recommendation skill: match the method's risk profile to the scenario's stakes, and justify by weighing the trade-off. Traps: choosing direct for a high-risk system — Knight Capital, Denver — and confusing phased with parallel or pilot. Phased splits by feature; pilot splits by user group; parallel runs both at once.

NARRATOR: Exam-style finish — four questions, recall up to extended. Pause and attempt each before the model answer.

QUESTION: Question one, recall. Define direct and pilot implementation. Pause, then play.

NARRATOR: Model answer: direct implementation, the big bang, replaces the old system entirely with the new one in a single changeover, all users switching at once with no overlap. Pilot implementation deploys the complete new system to a small, representative group of users first, while the rest continue on the old system, so issues are found and fixed in a controlled setting before full rollout. Defining direct as a single all-at-once switch and pilot as a small-group trial of the whole system is the mark.

QUESTION: Question two, compare. Compare direct and parallel implementation methods. Pause, then play.

NARRATOR: Model answer: direct implementation switches entirely to the new system in one changeover — the fastest and cheapest method because no duplicate systems run, but high-risk, since there's no fallback if it fails and all users are disrupted at once. Parallel implementation runs the old and new systems simultaneously for a period — the safest method, because the old system remains a working fallback and outputs can be compared to verify correctness, but the most expensive and resource-intensive, since two full systems operate at once. The comparison is a direct trade-off: direct optimises cost and speed at the expense of safety; parallel optimises safety at the expense of cost. Stating both methods, their risk levels, and framing it as the cost-versus-safety trade-off is the mark.

QUESTION: Question three, application. Distinguish phased implementation from pilot implementation. Pause, then play.

NARRATOR: Model answer: phased implementation rolls out the system in stages by functionality, delivering one part to all users, then the next, until the whole system is deployed — it splits the rollout by feature. Pilot implementation deploys the entire system to a small, representative subset of users while everyone else stays on the old system — it splits the rollout by user group. The key distinction: phased gives part of the system to everyone gradually; pilot gives the whole system to part of the users first. Naming that feature-split versus user-split distinction is the mark.

QUESTION: Question four, the extended one. A national bank is replacing its core transaction system. Recommend an implementation method and justify your choice. Pause, plan a structured response, then play.

NARRATOR: Model answer: parallel implementation is recommended, very likely preceded by a pilot in a limited region or branch. A core banking transaction system is safety-critical and must be continuously available, and any error in transaction processing — lost, duplicated, or incorrect transactions — would cause severe financial and reputational harm and likely breach regulatory and compliance requirements. Running the old and new systems in parallel provides a guaranteed fallback if the new system fails, and lets the bank compare the new system's transaction outputs against the known-correct old system to verify accuracy before committing fully — an integrity safeguard. A pilot in one region first limits the blast radius of any undiscovered fault to a small population before national rollout. The substantial cost of operating two systems is justified by the catastrophic consequences of failure in core banking, and a direct big-bang cut-over would be reckless here — exactly the high-risk changeover that destroyed Knight Capital in forty-five minutes. A response that names the method, links its advantages to the scenario's critical, regulated, zero-tolerance risk profile, weighs the cost against the stakes, and explicitly rejects the risky alternative earns full marks.
