Hear that now, because the verb "evaluate" is the whole game.

NARRATOR: Core idea, and the single biggest trap-killer in the module: evaluation is not "it works." Students write "the program runs and does what I wanted" and get a band four. Evaluation is a justified judgement of effectiveness against defined criteria, supported by evidence, with reflection. The NESA verb "evaluate" means weigh it up and reach a conclusion — strengths and weaknesses, against a yardstick, with proof. No criteria, no evidence, no judgement, no marks.

NARRATOR: The hook, the marquee evaluation mnemonic for the whole module — C-E-R, said "see-ee-arr." C, Criteria — the measurable standards you judge against. E, Evidence — the proof you gathered, from tests and feedback. R, Reflection — maintainability, deployment, and lessons learned. C-E-R: Criteria, Evidence, Reflection. Every strong evaluation has all three. Take each in turn, because each pulls in a whole earlier episode.

NARRATOR: C, Criteria — the through-line of the entire module. The criteria you evaluate against are the requirements you wrote in twenty-three oh-one and the quality criteria you set in QA, twenty-four oh-six — C-U-P-S: Correctness, Usability, Performance, Security. This is why sloppy requirements make evaluation impossible: with no measurable target, you have nothing to evaluate against. So a good evaluation opens by restating the criteria — "the requirement was ninety-fifth-percentile page load under three hundred milliseconds at two hundred users" — because that's the yardstick. Criteria must be measurable; "it should be good" can't be evaluated, but "p-ninety-five under three hundred milliseconds" can.

NARRATOR: E, Evidence — the proof, in two streams. First, the test results: the test plan from twenty-six oh-one, run with path-and-boundary test data, and the central technique the dot-point names — comparing actual output with expected output. For each test you state what you expected and what actually happened, and a discrepancy is a finding, not a failure to hide. Second, the synthesised feedback: the C-S-A output from twenty-six oh-two, the dot-point's "report to synthesise feedback." So your evidence is objective — tests — plus subjective — feedback — triangulated, exactly the QA idea of measuring against criteria. Listing one is an actual-versus-expected results table; listing two is an evaluation rubric mapping criteria to evidence.

NARRATOR: Dwell on actual-versus-expected, because it's the most examinable single technique. For a given input, you state the expected output — derived from the requirement — and the actual output the system produced, and compare. Match means evidence the criterion is met. Differ means you analyse the discrepancy: the gap, how severe, and the likely cause. A worked instance: expected output for a G-P-A calculation is "three-point-seven-five"; actual is "three-point-seven-five" — match, correctness evidence. But expected for a scheduling conflict is "conflict detected: room one-oh-one double-booked"; actual is "no conflicts detected" — a discrepancy, high severity, likely a logic error in the conflict-detection branch, which a path test should have caught. Naming the expected, the actual, the gap, and the probable cause is the technique markers reward.

QUESTION: Quick check. Your evaluation criterion is "the search returns results in under two seconds." You run a test: expected, results in under two seconds; actual, results in three-point-one seconds. Is this a pass or a fail, and what do you write? Pause, decide, then play.

NARRATOR: It's a fail against that criterion — and evaluation means you say so honestly and analyse it. You write: "Expected under two seconds; actual three-point-one seconds, so the performance criterion is not met. The discrepancy is one-point-one seconds over target; the likely cause is an un-indexed database query, identified by profiling. Recommendation: add an index, which a measure-optimise-measure pass — twenty-six oh-one — should bring it under target." That's a band-six move: compared actual versus expected, judged against the criterion, found a discrepancy, analysed the cause, recommended a fix. The weak answer hides the slow result or just says "search works." Evaluation rewards honest judgement with evidence, not pretending everything passed.

NARRATOR: R, Reflection — the part students skip, and it's a third of the hook. Reflection looks beyond "does it pass the tests" to the bigger picture: maintainability — is the code documented and structured so it can be maintained, the S-E-twelve-oh-nine concern; deployment readiness — is it production-ready, with backups and rollback in place, the twenty-five oh-four concern; and lessons learned — what you'd do differently next time. This is where you show maturity. A maintainability reflection — "the modular structure and current documentation make this easy to extend, though test coverage of error handling is a gap" — is exactly the judgement the marker wants, and it's the manage-and-document marker one last time.

NARRATOR: Anchor it in the case study one last time, because Healthcare-dot-gov has a famous evaluation chapter — the post-mortem. After the twenty-thirteen launch collapse, the U-S government ran a rescue and a structured evaluation: they set clear criteria — the site must handle the real concurrent load and complete the core enrolment workflow; they gathered evidence — load tests, error rates, actual-versus-expected on the broken workflows; and they reflected — the lessons were frozen and flawed requirements, integration too late, and load testing skipped, twenty-six oh-one's exact failure. The rescue worked because it was a real evaluation — criteria, evidence, reflection — not "is it up." C-E-R is how you actually fix a failed project, and the failures were precisely what this module teaches you to avoid.

NARRATOR: Worked example, weak versus strong — the band-four-versus-band-six boundary on the highest-mark question in the topic. The brief: evaluate a checkout feature against the requirement "complete a purchase in under three steps and under five seconds." The weak answer: "The checkout works well and users liked it." Band four — no criteria, no evidence, no judgement, no reflection. The strong answer, using C-E-R: "Against the criterion — purchase in under three steps and under five seconds — the evidence is as follows. Testing: I compared actual against expected output across boundary and path test cases; the checkout completes in two steps, meeting the step criterion, but actual completion time was five-point-four seconds against an expected under five, a discrepancy of point-four seconds, with the likely cause being an un-cached payment lookup. Feedback: synthesised user testing showed eighty-five percent found it intuitive, but a recurring theme was confusion at the confirmation step. Judgement: the feature is largely effective — it meets the step and usability criteria — but partially fails the performance criterion. Reflection: the code is well-documented and maintainable, deployment-ready with rollback in place; the lesson is to load-test the payment integration earlier next time." Measurable criteria, evidence from tests and feedback with actual-versus-expected, an honest judgement of partial effectiveness, and a maintainability reflection — that's the band-six shape.

NARRATOR: Exam-style finish — four questions, including the big extended one. Pause and attempt each before the model answer.

QUESTION: Question one, recall. What does the NESA verb "evaluate" require you to do? Pause, then play.

NARRATOR: Model answer: to evaluate is to make a justified judgement about the effectiveness or worth of something, weighing strengths and weaknesses against defined criteria, supporting it with evidence, and reaching a conclusion. For a software solution, that means judging it against its requirements and quality criteria, using evidence from testing and feedback, rather than merely describing that it works. Stating "justified judgement against criteria with evidence" is the mark.

QUESTION: Question two, application. Explain how comparing actual output with expected output supports evaluation. Pause, then play.

NARRATOR: Model answer: comparing actual output with expected output provides objective evidence of whether the solution meets its functional criteria. For each test case, the expected output is derived from the requirement and the actual output is what the system produces; a match is evidence the criterion is met, while a discrepancy is a finding to be analysed for size, severity, and likely cause. This turns evaluation from subjective opinion into an evidence-based judgement, and it directly tests correctness — the C of C-U-P-S. Naming the expected-versus-actual comparison as objective evidence and as discrepancy analysis is the mark.

QUESTION: Question three, recall. Distinguish testing from evaluation. Pause, then play.

NARRATOR: Model answer: testing is executing test cases against the software to detect defects and gather data, such as comparing actual and expected outputs. Evaluation is the broader judgement of the solution's overall effectiveness against its criteria, using testing results as one source of evidence alongside synthesised user feedback, and adding reflection on maintainability and lessons learned. In short, testing produces evidence; evaluation interprets that evidence into a justified judgement. Stating testing-produces-evidence and evaluation-judges-with-it is the mark.

QUESTION: Question four, the extended one. Evaluate the effectiveness of a software solution against its requirements. Pause, plan a full structured answer, then play.

NARRATOR: Model answer: an effective evaluation is structured as criteria, evidence, and reflection — C-E-R. First, state the criteria: the solution's measurable requirements and quality criteria — Correctness, Usability, Performance, Security — for example, "complete a booking in under three steps and load pages in under two seconds." Second, present the evidence against each criterion: from testing, run the test plan with path and boundary test data and compare actual output with expected output, reporting matches as met criteria and discrepancies with their severity and likely cause; from feedback, present synthesised user feedback showing satisfaction levels and recurring themes. Third, make the judgement: weigh strengths against weaknesses to conclude how effective the solution is overall — for example, "effective on correctness and usability, partially effective on performance, where one criterion was missed by a measurable margin." Fourth, reflect: assess maintainability — is the code documented and modular; deployment readiness — are backups and rollback in place; and lessons learned for next time. The full-mark answer names measurable criteria, gives evidence from both testing, including actual-versus-expected, and feedback, reaches a balanced justified judgement rather than "it works," and includes reflection — and noting it mirrors the Measure-Audit-Improve loop from security shows synthesis across the course.
