NARRATOR: The dot-point, quoted exactly: "design, develop and implement a safe application programming interface, A-P-I, to minimise software vulnerabilities." The outcome is S-E-twelve-oh-seven, secure code. You already know what an A-P-I is from Programming for the Web — the defined contract a system exposes so other code can talk to it, and in thirteen-oh-one the back-end boundary your front end called. So I'm not teaching what an A-P-I is. I'm teaching how to make one safe, in syllabus terms, with the controls a marker wants named.

NARRATOR: The core idea, in one sentence: an A-P-I is a door into your system, opened on purpose to the whole internet, so every request arriving at it is untrusted and must be checked before it's allowed to do anything. The same "never trust input" mindset from last chapter, but now the input is an entire request from an unknown caller. Everything today is a layer of checking on that door.

NARRATOR: Objectives up front. By the end you'll be able to name and explain the measures that secure an A-P-I; explain the purpose of rate limiting; and explain why being authenticated is not the same as being authorised — the classic A-P-I trap. Here's your hook for the set of measures, because "describe measures to secure an A-P-I" is a six-marker and a list needs an anchor. The four core measures: Authenticate, Authorise, Rate-limit, and secure the Transport. First letters — A, A, R, T — give you A-A-R-T. Secure an A-P-I with A-A-R-T: Authenticate, Authorise, Rate-limit, Transport-secure. Let's build each.

NARRATOR: First A — Authenticate. Every request must prove who is calling. The A-P-I has no login screen, so it uses a credential carried in the request itself, and there are three to name, in increasing sophistication. One: an A-P-I key — a long secret token issued to a developer or application, sent with each request to identify the caller. Simple, good for public A-P-Is and basic rate limiting, but it identifies the application, not an end user, and offers no fine-grained permissions. Two: a J-W-T, a JSON Web Token — a signed token the server issues after login, carrying the user's identity and permissions inside it, which the client sends back on every request. Because it's signed — the digital signature from fifteen-oh-two — the server can trust its contents without storing session state: stateless authentication. The catch to name: a J-W-T is signed, so it can't be tampered with, but it's only encoded, not encrypted, so never put secrets in the payload. Three: OAuth two-point-oh — the delegated-authorisation protocol behind every "log in with Google" button, letting a user grant a third-party app limited access without handing over their password. Name those three: A-P-I key, J-W-T, OAuth.

NARRATOR: Second A — Authorise, the one students drop, so we'll hit it hard. Authentication told you who is calling. Authorisation decides whether this caller is allowed to do this specific thing. Here's the trap, and the most common real A-P-I breach: checking authentication, then forgetting authorisation. A user logs in successfully — authenticated — then requests "slash A-P-I slash orders slash one-thousand-and-seven," an order belonging to someone else. If your A-P-I returns it just because the caller is logged in, you've got broken access control — authenticated but not authorised, "logged in is not allowed," exactly the trap I flagged in fifteen-oh-one. The fix is to check, on every request, not just "are you logged in?" but "do you own or have permission for this specific resource?" — role-based access control plus an ownership check, applying least privilege. The exam line: authentication verifies identity; authorisation verifies permission for the specific action and resource — and an A-P-I must enforce both on every request, because being authenticated does not mean being authorised.

QUESTION: Quick check before rate limiting. An A-P-I endpoint lets any logged-in user fetch an order by its number in the U-R-L. What's the vulnerability, and what's the fix? Pause, answer, then play.

NARRATOR: The vulnerability is broken access control — also called insecure direct object reference. The endpoint authenticates the user but never authorises them for that specific order, so any logged-in user can change the number and read anyone's order. The fix is an authorisation check on every request: confirm the requested order belongs to the authenticated user, or that their role permits it, and deny otherwise. Authenticated is not authorised. That's the mark, and a near-certain exam scenario.

NARRATOR: Third — R — Rate-limit. Rate limiting caps how many requests a single caller can make in a time window — say, a hundred per minute per A-P-I key or per I-P address. Two reasons it matters, both worth naming. One: availability — it stops one abusive or malfunctioning client from flooding your A-P-I and starving everyone else, the denial-of-service defence from the Availability property in C-I-A-A-A. Two: it blunts brute-force and scraping attacks — an attacker trying thousands of passwords or harvesting your whole database is throttled to a crawl. The mechanism to name is the token bucket or a sliding window, and the response when a caller exceeds the limit is the H-T-T-P status "four-two-nine, too many requests." The exam line for "explain the purpose of rate limiting": rate limiting restricts the number of requests a client may make in a given period, protecting availability against denial-of-service and abuse, and slowing brute-force and scraping attacks.

NARRATOR: Fourth — T — Transport-secure. Every A-P-I request and response must travel over H-T-T-P-S, which is T-L-S — the hybrid encryption from fifteen-oh-two. Without it, A-P-I keys, tokens, passwords and data all cross the network in clear text, readable by anyone in between. An A-P-I over plain H-T-T-P is broken before you add a single other control, because the credential you worked so hard to check is being shouted across the wire. This is confidentiality in transit, assumed by everything else. So A-A-R-T: Authenticate the caller, Authorise the action, Rate-limit the volume, secure the Transport. And underneath all four, two more rules you already know: validate every input parameter at the boundary — V-S-E applied to the A-P-I — and fail securely, returning a generic error, never a stack trace, never leaking whether a record exists.

NARRATOR: Now session management, because the dot-point's neighbours include it and it bridges to the vulnerabilities chapter. A session is how a server remembers a user is logged in across multiple requests, since H-T-T-P itself is stateless — it forgets you between requests. After you authenticate, the server issues a session identifier, usually stored in a cookie, and you send it back on each request to prove you're still the same logged-in user. The security rules for sessions are a "name these" target, so make sessions a strong S-A-F-E box. S: Strong, cryptographically random session I-Ds, so they can't be guessed. A: Attributes on the cookie — mark it HttpOnly so client-side JavaScript can't read it, blocking a cross-site scripting attack from stealing it; mark it Secure so it only travels over H-T-T-P-S; and set SameSite to limit it being sent on cross-site requests, which is anti-cross-site-request-forgery. F: Fixed lifetime — sessions time out and expire, so a stolen one doesn't live forever. E: re-issue on Escalation — regenerate the session I-D when the user logs in or changes privilege, preventing session fixation. Strong, Attributes, Fixed lifetime, re-issue on Escalation — S-A-F-E sessions. The two attacks those defend against — session hijacking, stealing a live session, and session fixation, planting a known one — plus broken authentication, are detailed in episode eighteen-oh-two, so this primes that.

NARRATOR: Last piece, a favourite exam target because it's so often misunderstood: CORS, cross-origin resource sharing. Start with the rule it relaxes — the same-origin policy, which I previewed in Programming for the Web twelve-oh-three. The same-origin policy is a browser security rule: by default, a web page's JavaScript can only make requests to its own origin — its own domain — and the browser blocks it from reading responses from other domains. That stops a malicious page from quietly calling your bank's A-P-I using your logged-in session. CORS is the controlled exception: a set of response headers by which a server says "I permit pages from these specific other origins to call me." So CORS doesn't tighten security — it loosens the same-origin policy, in a controlled way, so your legitimate front end on one domain can call your A-P-I on another.

NARRATOR: Here's the trap the marker sets, so lock it in: CORS is not an access-control mechanism. Students write "we use CORS to stop unauthorised users accessing the A-P-I" — wrong. CORS only tells the browser which web origins may read responses; it does nothing about a request from a script, a command-line tool, or an attacker's server, none of which obey CORS at all. Access control is authentication and authorisation — A-A. CORS just manages which browser origins may talk to you. The exam line: the same-origin policy restricts a page to its own origin by default; CORS is a controlled relaxation of it that lets a server permit specific other origins — it is not a substitute for authentication and authorisation. And the related trap: a permissive CORS policy allowing all origins — the wildcard — combined with credentials is a real misconfiguration, so allow-list your origins, never wildcard them. Allow-list beats block-list, even here.

QUESTION: This is the one to lock in — pause the player and answer it properly, because it's a guaranteed six-marker. Describe three measures you would use to secure a public A-P-I, and explain what each protects against. Pause now, work it out, then play.

NARRATOR: Here's a full-mark answer using A-A-R-T. First, authentication: require every request to carry a valid credential — an A-P-I key or a signed J-W-T — so the A-P-I only serves identified callers and rejects anonymous ones. Second, authorisation: on every request, check that the authenticated caller has permission for the specific resource and action, using role-based access control and ownership checks with least privilege, preventing broken access control where a logged-in user reaches another user's data. Third, rate limiting: cap requests per client per time window, protecting availability against denial-of-service flooding and slowing brute-force and scraping attacks. And underpinning all three, transport security: serve the A-P-I only over H-T-T-P-S so credentials and data are encrypted in transit, plus validate all input to prevent injection. Notice the shape: name the measure, then say precisely what it protects against. Measure plus threat-countered, three or four times — that's the band-six answer, and naming A-A-R-T as your structure shows the marker you have a framework.

NARRATOR: Worked example — the weak-versus-strong contrast on "secure this endpoint." Imagine the endpoint "slash A-P-I slash orders." A weak answer says: "make it secure by requiring users to log in, and use CORS to control access." That earns little — authentication but no authorisation, and it misuses CORS as access control. A strong answer says: "require a valid signed token for authentication; on every request, authorise the caller by verifying they own the requested order or hold a permitted role, applying least privilege; rate-limit per token to prevent abuse; serve only over H-T-T-P-S; and validate the order I-D as an input. CORS is configured to allow only our own front-end origin, but it is not relied on for access control, which is enforced by authentication and authorisation." Hear the difference: the weak answer stops at "logged in"; the strong answer adds the per-request authorisation check, names the real controls, and corrects the CORS misconception. Authorisation on every request, and CORS is not access control — that's where the marks are.

NARRATOR: Exam-style finish — five questions, recall up to extended. Pause and attempt each before the model answer.

QUESTION: Question one, recall. Explain the purpose of rate limiting on an A-P-I. Pause, then play.

NARRATOR: Model answer: rate limiting restricts the number of requests a client may make to an A-P-I within a given time period — for example, a hundred per minute per key or I-P. Its purpose is to protect availability by preventing one client flooding the A-P-I and denying service to others, and to slow abuse such as brute-force credential guessing and large-scale data scraping. When the limit is exceeded the A-P-I responds with "too many requests." Naming the request cap, the availability protection, and the brute-force or scraping mitigation is the mark.

QUESTION: Question two. Distinguish between authentication and authorisation in the context of an A-P-I, and explain why both are needed. Pause, then play.

NARRATOR: Model answer: authentication verifies the identity of the caller — who is making the request, typically through an A-P-I key or a signed token. Authorisation verifies that the authenticated caller has permission for the specific resource and action requested. Both are needed because authentication alone only proves identity; without an authorisation check on each request, any logged-in user could access data or actions they should not — broken access control. The A-P-I must enforce both: who you are, then what you may do. Naming the identity-versus-permission distinction and "authenticated is not authorised" is the mark.

QUESTION: Question three. What is the same-origin policy, and how does CORS relate to it? Pause, then play.

NARRATOR: Model answer: the same-origin policy is a browser security rule that, by default, only allows a web page's scripts to read responses from its own origin — its own domain — blocking responses from other origins. CORS, cross-origin resource sharing, is a controlled relaxation of this policy: through response headers, a server explicitly permits scripts from specified other origins to access it. CORS is not an access-control mechanism for the A-P-I itself — it only governs which browser origins may read responses, and does not affect non-browser clients; access control must still be enforced by authentication and authorisation. Naming "same-origin restricts by default, CORS relaxes it controllably, and it is not access control" is the mark.

QUESTION: Question four, application. A developer secures an A-P-I by requiring login, then returns any record whose I-D is in the U-R-L. Identify the vulnerability and describe the fix. Pause, then play.

NARRATOR: Model answer: the vulnerability is broken access control, also called an insecure direct object reference — the A-P-I authenticates the user but never authorises them for the specific record, so any logged-in user can change the I-D in the U-R-L and read others' records. The fix is an authorisation check on every request: verify the authenticated user owns the requested record or holds a role permitted to access it, applying least privilege, and return a generic denial otherwise. Naming broken access control and the per-request ownership or role authorisation fix is the mark.

QUESTION: Question five, the extended one. Design the security for a public A-P-I that lets registered users read and update their own profile. Explain the measures and what each protects against. Pause, plan a structured response, then play.

NARRATOR: Model answer: structure it with A-A-R-T plus input handling and sessions. Authentication: require a valid signed token, such as a J-W-T issued after login, on every request, so only identified users are served. Authorisation: on each request, verify the user is acting on their own profile, not someone else's, using ownership and role checks with least privilege, preventing broken access control. Rate limiting: cap requests per user to protect availability against flooding and to slow brute-force or scraping. Transport security: serve only over H-T-T-P-S so tokens and personal data are encrypted in transit. Input handling: validate and sanitise every field on update — V-S-E — use parameterised queries to prevent injection, and handle errors generically so nothing leaks. Manage sessions securely with strong random identifiers, HttpOnly, Secure and SameSite cookies, timeouts, and identifier regeneration on login. Configure CORS to allow only the application's own front-end origin, while relying on authentication and authorisation, not CORS, for access control. A strong response names A-A-R-T, ties each control to the threat it counters, adds input validation and secure sessions, and corrects the CORS misconception. That's full marks.
