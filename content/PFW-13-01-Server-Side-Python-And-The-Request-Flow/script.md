NARRATOR: The lifecycle first, with a hook to lock the stages: R-H-D-R. Route, Handle, Data, Respond. Four stages. When a request arrives at your application, the framework first routes it — matches the U-R-L to the right piece of code; then the handler function runs — your business logic; the handler fetches or stores data — usually a database query; and finally it builds a response — H-T-M-L or J-S-O-N — and sends it back. Route, Handle, Data, Respond — R-H-D-R. That's the spine of "describe the steps a web server takes to process a dynamic request", a four-to-six mark question.

NARRATOR: But before R-H-D-R, get one distinction right, because the dot-point names two different things — web server software and a web framework — and students constantly conflate them. The web server software — nginx or Apache — accepts the incoming network connection, parses the raw H-T-T-P, and serves static files like images and C-S-S directly. The web framework — Flask, in this course — is the Python code that contains your application's logic: the routes and handlers. Not the same layer. In production, the web server sits in front and forwards dynamic requests to the framework, often through an application server in between. The mark-earning distinction: the web server software handles the network connection and static content, while the web framework runs the application's routing and business logic. The named trap: nginx is a web server; Flask is a web framework — don't call Flask a web server or nginx a framework.

QUESTION: Pause the player. In your own words, why have a separate web server like nginx in front of the framework at all — why not let Flask handle everything? Work it out, then play.

NARRATOR: Because they're good at different things. The web server is highly optimised for accepting many connections and serving static files fast, and it handles T-L-S termination — remember T-L-S, the Tasty in Dogs Take Tasty Hambones — decrypting H-T-T-P-S so the framework doesn't have to. The framework is for your dynamic logic, not for efficiently shovelling images at thousands of clients. So the web server takes the load and the static content, and forwards only the dynamic requests to the framework. Separation of responsibilities — the same separation-of-concerns instinct you met with C-S-S, applied to infrastructure. A strong answer names: the web server efficiently handles connections, static files, and T-L-S, while forwarding dynamic requests to the framework where the application logic lives.

NARRATOR: Now walk a real route end to end with R-H-D-R — narrating one is the exam skill. Take a search feature: a user submits a search box, hitting the U-R-L slash search with their query. Stage one, Route: the framework matches slash search to the search handler function. Stage two, Handle: the handler runs — it reads the user's query out of the request object, the framework's object holding everything about the incoming request: the U-R-L, the form data, the headers. Stage three, Data: the handler queries the database for matching records. Stage four, Respond: it passes those records to a template, which renders H-T-M-L, and the framework returns that as the response. Route, Handle, Data, Respond. There's a Flask search route in Listing 1 and the same flow as N-E-S-A pseudocode in Listing 2 — "begin handle request, get query, query database, render response, end" — so you can rehearse the exam form.

NARRATOR: Now the most important interleave in the episode — a forward reference worth its weight. In stage two, the handler read the user's query out of the request object. That value came from the user — and the user is on the client side, which we established is visible, tamperable, and cannot be trusted. So what if the user's "search query" isn't a word, but a piece of malicious code? Drop that raw input straight into a database query by gluing strings together, and an attacker can rewrite your query — that's S-Q-L injection. Drop that raw input straight into the H-T-M-L you send back, and the attacker's code runs in other users' browsers — that's cross-site scripting, X-S-S, which we first flagged on interactive sites in episode one. The rule, and it's the spine of the security module: never trust input from the request. The handler must validate and sanitise everything that comes in, and use parameterised queries for the database — taught properly next episode and in full in Secure Software Architecture. For now, when you narrate a back-end flow, flag it: "the handler reads the user's input — which must be validated, never trusted." That sentence earns marks here and is the entire foundation of the next module.

QUESTION: Quick one to make it stick — pause if you like. Why is it not enough to check that search input in the browser with JavaScript before sending it? Then play.

NARRATOR: Because — as we hammered in the system-modelling episode — client-side checks can be bypassed entirely; the attacker sends a request straight to your slash-search endpoint without ever loading your page or running your JavaScript. So the back-end handler must re-validate. Client-side validation is convenience; server-side validation is security. Third time you've heard that — the repetition is deliberate, because it's the single most important idea bridging this module and the next.

NARRATOR: The rest of the dot-point's vocabulary, because the exam can ask you to list what's involved. Objects: the framework gives you a request object for the incoming request, a response object for what you send back, and a session object that remembers a logged-in user across requests. Libraries: you pull in reusable libraries — for hashing passwords, talking to the database, or calling external services like a payment gateway. Databases: the handler interfaces with the Store, the database, to read and write persistent data. And a neat mapping that ties three topics together with its own hook: CRUD. Create, Read, Update, Delete — the four things you do to data. They line up across the stack: Create-Read-Update-Delete maps to the S-Q-L commands INSERT, SELECT, UPDATE, DELETE, and to the H-T-T-P methods POST, GET, PUT, DELETE. CRUD — Create, Read, Update, Delete. That's a forward link into next episode on databases and S-Q-L, and a tidy way to remember how H-T-T-P verbs, S-Q-L, and data operations all correspond.

NARRATOR: Exam-style finish — five questions, recall to extended. Pause and attempt each, then play for the model answer.

QUESTION: Question one, recall. Distinguish between web server software and a web framework. Pause, then play.

NARRATOR: Model answer: web server software, such as nginx or Apache, accepts incoming network connections, parses H-T-T-P requests, and serves static files directly, often handling T-L-S. A web framework, such as Flask, is the application code that defines routing and business logic for dynamic requests, which the web server forwards to it. The web server handles the connection and static content; the framework runs the application logic. Naming both layers and their different jobs is the mark.

QUESTION: Question two. List, in order, the main stages a back end goes through to process a dynamic request. Pause, then play.

NARRATOR: Model answer: first, routing — matching the request's U-R-L to the correct handler function; second, handling — executing the handler's business logic; third, data access — querying or updating the database; and fourth, responding — building and returning the response, such as rendered H-T-M-L or J-S-O-N. Route, Handle, Data, Respond — R-H-D-R. The ordered four stages are the marks.

QUESTION: Question three. Explain why a back-end handler must validate user input even if the front end already checks it. Pause, then play.

NARRATOR: Model answer: client-side validation runs in the user's browser and can be bypassed — an attacker can send a request directly to the endpoint without the page or its JavaScript — so unvalidated input could reach the server. The back-end handler must therefore validate and sanitise all input server-side to prevent attacks such as S-Q-L injection and cross-site scripting. Client-side validation is for convenience; server-side validation is for security. Naming the bypass and the server-side requirement is the mark.

QUESTION: Question four. A user submits a search term that is processed by a Flask route which queries a database and returns results. Describe the request flow and identify one security risk. Pause, give a real attempt, then play.

NARRATOR: Model answer: the framework routes the slash-search request to the search handler; the handler reads the search term from the request object, queries the database for matching records, then renders those records into an H-T-M-L response that is returned to the browser — Route, Handle, Data, Respond. A key security risk is that the search term is untrusted user input: if it is concatenated directly into the S-Q-L query it enables S-Q-L injection, and if it is inserted unescaped into the H-T-M-L it enables cross-site scripting. The mitigation is to validate and sanitise the input and use parameterised queries. Naming the flow and a specific injection risk with its mitigation is the full mark.

QUESTION: Question five, the extended one. Describe the complete process by which a web application's back end handles a dynamic request, from the web server receiving it to the response being returned, referring to the components involved. Pause, plan a structured response, then play.

NARRATOR: Model answer: the web server software, such as nginx, accepts the T-C-P connection and parses the H-T-T-P request; if it is for a static file it serves it directly, otherwise it forwards the dynamic request to the web framework, often via an application server. The framework then routes the request, matching its U-R-L to the correct handler. The handler executes the business logic, reading data from the request object and validating it server-side, because user input cannot be trusted. It interfaces with the database — using parameterised queries to safely read or write data — and may call libraries or external services such as a payment A-P-I. The handler then builds the response, typically rendering H-T-M-L through a template engine or returning J-S-O-N, and the response travels back through the framework and web server to the browser. Throughout, sessions track the authenticated user and input is validated to prevent injection attacks. That response names the web server, the framework, routing, the handler, validation, the database with parameterised queries, the template engine, and the response path — a full-mark integrated answer that also previews the security module.
